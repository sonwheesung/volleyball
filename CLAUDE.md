# 백년배구 (가칭) — 게임 설계 문서

> 이 문서는 Claude Code가 개발을 진행하기 위한 단일 기준 문서다.
> 모호하면 이 문서의 "설계 원칙"을 우선한다. 새 결정은 이 문서에 반영한 뒤 진행한다.

---

## 1. 한 줄 요약

모바일 배구 단장/감독 시뮬레이션. 가상 V리그(여자부)를 무대로, 플레이어는 구단을
장기간(수십 시즌) 운영한다. 평소 경기는 AI가 자동으로 굴리고, 중요한 경기만
직접 개입한다. 모티브는 "백년야구".

---

## 2. 핵심 기둥 (우선순위 = 설계 충돌 시 판단 기준)

1. **관전형 장기 시뮬 (2026-06-11 정정, 구 "방치형")** — 조작하는 게임이 아니라 **보는 게임**이다.
   자동진행이 기본이고, 핵심 경험은 내 팀 경기와 시즌 서사를 **관전**하는 것.
   따라서 경기 보드의 연출 품질(실제 배구처럼 보이기)은 부차 요소가 아니라 1순위 투자처다.
   자동진행·무푸시·"매 순간 손이 가지 않게" 원칙은 그대로 유지(6장 정의).
2. **데이터 누적 서사** — 가상 선수의 기록이 세월 쌓이며 의미(스토리)가 된다.
3. **현실적 밸런스 (안티 과금)** — 선수는 노쇠하고 망한다. 과금이 현실성을 깨면 안 된다.
4. **단장 의사결정의 무게** — 드래프트/FA/외국인 영입이 전력을 좌우한다. (트레이드는 제외, 7장)

> 충돌 예시: "감독으로 매 랠리 개입" vs "관전형". → 1번이 우선이므로
> **선택적 개입**으로 해결한다 (아래 4장). 보는 건 권장, 조작 강요는 금지.

---

## 3. 플레이어 역할 — 구단주의 건의 (2026-06-11 정정)

플레이어는 **구단주**다. 현장 권한(선발·교체·타임아웃)은 항상 **감독**에게 있고,
구단주는 **건의**한다 — 감독은 성향(카리스마·합리)에 따라 수락하거나 거절한다(OWNER_SYSTEM
벤치 건의와 동일 패턴). "감독으로 핸들을 잡는" 모드는 없다.

### 구단주 권한표 (어디까지 내 마음대로인가)

| 영역 | 권한 | 비고 |
|---|---|---|
| **직접 결정 (프런트 권한)** | | 현실에서도 단장·구단 소관 |
| 선수 계약 — 재계약·방출·FA 영입·드래프트 지명 | ✅ 직접 | 단, 방출 하한(10명)·캡·운영 자금이 막을 수 있음 |
| 외국인 — 트라이아웃 지명·재계약·시즌 중 교체 | ✅ 직접 | 교체는 시즌 1회 |
| 스태프 — 감독 선임(교체)·코치·스카우터 계약 | ✅ 직접 | 감독이 말 안 들으면 바꾸는 게 구단주의 칼 |
| 재정 — 운영 자금 사용 | ✅ 직접 | 모기업·관중·굿즈 수입 안에서 |
| 훈련 방향 설정 | ✅ 직접(구단 방침) | 감독 기본값 오버라이드 |
| **상대 의지에 막히는 것** | | 결정해도 거부당할 수 있다 |
| 선수 면담(설득) | ⚠ 선수가 거절·결렬 가능 | 들볶으면 안 만나줌 |
| 재계약 제안 | ⚠ 불만 선수는 뿌리침 | 면담이 좌우 |
| **건의만 (현장 권한 = 감독)** | | 감독 성향따라 수락/거절 |
| 선발 기용·벤치 지시 | 🗣 건의 | 에이스 빼라면 강하게 저항 |
| 세트 중 작전 방침(핀치서버 등 토글) | 🗣 방침 | 집행은 감독·엔진 |
| **불가 (월권 금지)** | | |
| 타임아웃·교체 등 경기 중 현장 운영 | 🚫 | 감독 고유 권한 — 건의도 안 한다(2026-06-16 정정). 흐름 보고 부르는 건 감독 |
| 랠리 단위 지시(토스 타겟 등)·감독 무시 직접 라인업 | 🚫 | 넣지 않는다 |

### 경기 관련 건의 (랠리 위 레이어만)
- 경기 전: 선발 기용 건의("이 선수를 선발로/빼고") — 감독 판정
- 세트 중: 작전 방침 토글(핀치서버/블로킹/수비 — 기존 SubPolicy, 감독이 집행)
- 빅매치: 관전만. **타임아웃은 감독 고유 권한**이라 건의하지 않는다(2026-06-16 정정 —
  구단주가 코트 옆에서 타임아웃을 부르는 건 현실에 없고 권한 분리를 흐림). 흐름이 밀리면
  감독이 알아서 부른다(MATCH_SYSTEM 7.4). 마음에 안 들면 → 감독 교체.

> **금지**: 매 토스마다 "누구에게 줄지" 수동 선택 + 감독을 건너뛰는 직접 지휘.
> 개입은 항상 "건의/방침" 레벨이고, 판단은 감독이, 실행은 엔진이 한다.
> 감독이 마음에 안 들면 — 그게 바로 감독 교체(STAFF)의 이유가 된다.

## 4. 경기 시뮬레이션 엔진 (게임의 심장)

### 4.1 랠리 체인 (1점이 나기까지의 판정 흐름)

```
서브 ──> 리시브 ──> 세트(토스) ──> 공격 ──┬─> 득점 (공격 성공)
                                          ├─> 블로킹 차단 (상대 득점)
                                          └─> 디그 성공 ──> [상대 공격으로 전환, 루프]
```

- 야구와 달리 **랠리는 루프다.** 디그가 성공하면 공수가 바뀌어 체인이 다시 돈다.
  한 점에 체인이 2~3바퀴 돌 수 있다. 랠리 길이 자체가 변수.
- 각 단계는 능력치 + 확률 판정. 시드 기반 결정론적 시뮬을 쓴다(같은 시드 = 같은 결과,
  리플레이·검증 가능).

### 4.2 단계별 판정 개요 (의사 공식, 튜닝 대상)

```
서브 에이스 확률   = f(server.serve, receiver.recv)
서브 범실 확률     = 고정 기저 + 서브 공격성에 비례
공격 성공 점수     = hitter.atk × (0.6 + 0.4 × setter.set/100)   // 세터가 승수
블로킹 차단 확률   = f(blocker.blk, hitter.atk)
디그 성공 확률     = f(digger.dig, 공격 강도)
```

> 위 계수는 전부 placeholder다. 밸런싱 단계에서 시즌 통계가 현실 V리그
> 분포(세트 스코어, 공격 성공률 ~40~55% 등)에 수렴하도록 튜닝한다.

### 4.3 로테이션 (야구에 없는 핵심 변수)

- 서브권을 따낼 때(사이드아웃) 6명이 시계방향 1칸 회전.
- 전위(2·3·4번)만 네트 앞 공격/블로킹 가능. 후위는 후위 공격만.
- **세터가 전위냐 후위냐**에 따라 공격 옵션이 바뀐다.
- 후위로 간 미들블로커는 리베로와 교체(수비 전문). 규칙 반영 필수.
- 로테이션을 빼면 "배구"가 아니라 "숫자 야구"가 된다. 반드시 구현.

### 4.4 세트/경기 규칙

- 1~4세트 25점, 5세트 15점. 모두 듀스(2점차) 적용.
- 5세트 3선승. 한 팀이 3세트 먼저 따면 경기 종료.
- 랠리포인트제: 모든 랠리가 점수로 직결.
- 매 세트 서브권 시작 팀 교대.

## 5. 선수 스탯 — 2층 구조

화면에 보이는 종합 스탯은 밑단 세부 스탯의 조합으로 산출된다.
"위치선정·반응속도"는 개별 기술이 아니라 **여러 기술 밑에 깔리는 공통 속성**이다.

### 5.1 밑단 (세부 스탯, 엔진 계산용)

| 분류 | 스탯 | 성장 특성 |
|---|---|---|
| 신체 | 키 | 고정 (나이 무관) |
| 신체 | 점프력 | 노쇠 시 하락 |
| 신체 | 민첩성 | 노쇠 시 하락 |
| 신체 | 체력(stamina max) | 노쇠 시 하락 |
| 신체 | 체젠(stamina regen) | 노쇠 시 하락 |
| 공통 | 반응속도 | 완만한 곡선 |
| 공통 | 위치선정 | 경험으로 성장(느린 하락) |
| 멘탈 | 집중력(clutch) | 경험으로 성장 |
| 멘탈 | 기복(consistency) | 경험으로 성장 |
| 멘탈 | VQ(배구 IQ) | 경험으로 성장 |

> **체력 vs 체젠**: 체력은 보유 상한, 체젠은 회복 속도. 둘을 나눠 "후반에 무너지는
> 선수 / 금방 회복해 체력전에 강한 선수"를 구분(MATCH_SYSTEM 7.1).
> **VQ**: 포지션 폴트(평균 VQ)·공격 코스·블로킹 타이밍 판단에 관여(MATCH_SYSTEM 0장).
> **기세(momentum)는 선수 스탯이 아니라 팀 단위 임시 수치**다(MATCH_SYSTEM 7.2).

### 5.2 윗단 (종합 스탯, 표시 + 엔진 입력)

```
스파이크 = f(키, 점프력, 공격기술) × consistency
블로킹   = f(키, 점프력, 반응속도, 블로킹기술)
디그     = f(민첩성, 반응속도, 위치선정, 디그기술)
리시브   = f(반응속도, 위치선정, 리시브기술)
세팅     = f(세팅기술, 집중력)               // 팀 공격 전체에 곱해지는 승수
서브     = f(서브기술, 집중력)
```

> "공격기술/블로킹기술/디그기술/리시브기술/세팅기술/서브기술"은 훈련·경기로
> 성장하는 순수 기술치. 밑단 신체/공통 스탯과 곱해져 윗단이 나온다.

### 5.3 포지션별 가중치 (스카우팅·드래프트 차별화의 엔진)

색이 진할수록 핵심. (3=핵심, 2=보통, 1=낮음, ·=무관)

| 스탯 | 세터 S | 아웃사이드 OH | 아포짓 OP | 미들 MB | 리베로 L |
|---|---|---|---|---|---|
| 키 | 2 | 2 | 2 | 3 | · |
| 점프력 | 1 | 3 | 3 | 3 | 1 |
| 체력 | 2 | 3 | 2 | 2 | 3 |
| 체젠 | 2 | 3 | 2 | 2 | 3 |
| 서브 | 2 | 3 | 3 | 2 | · |
| 리시브 | 1 | 3 | · | 1 | 3 |
| 세팅 | 3 | 1 | · | 1 | 1 |
| 스파이크 | · | 3 | 3 | 3 | · |
| 블로킹 | 1 | 2 | 2 | 3 | · |
| 디그 | 2 | 2 | 1 | 1 | 3 |
| VQ | 3 | 2 | 2 | 2 | 2 |
| 멘탈 | 3 | 2 | 2 | 2 | 2 |

핵심 함의:
- **리베로**: 키 무관. 키 작아도 디그·리시브만 최상이면 됨. 공격/블로킹/서브 불가.
- **세터**: 본인은 공격 거의 안 함. 세팅이 팀 공격수 전체에 곱해지는 승수. 영입 가치 큼.
- **미들블로커**: 신체 의존도 최고 → 전성기 짧고 노쇠 빠름. "전성기 짧은 선수" 서사.
- **아포짓(OP)**: 여자부에서 외국인 선수 자리. 팀 공격의 핵. 용병 결정이 가장 극적.

## 6. 장기 시스템 (백년야구식 누적 서사)

- **훈련·성장**: 현실처럼 아주 천천히 성장(스탯별 숨은 XP 바 + 스탯별 포텐셜).
  훈련은 12종으로 세분, **감독 선호(핵심2+보조3)** 에 따라 팀마다 다른 스탯이 성장 →
  장기 분화. 상세는 `docs/TRAINING_SYSTEM.md`.
- **나이/성장/노쇠 곡선**: 신체 스탯은 전성기 후 하락. 기술은 더 오래 유지.
- **명예의 전당 / 영구결번 / 누적 기록**: 가상 선수 통산 기록 영구 보존.
- **시즌 아카이브**: 우승/MVP/개인 타이틀 기록을 연표로.
- **환생/세대교체**: 은퇴 후 유망주 유입으로 100년+ 운영 지속.
- **시상식(2026-06 구현)**: MVP·챔프MVP·신인상·기량발전상·기록왕·베스트7·라운드MVP.
  상은 OVR이 아니라 실제 코트 생산에 준다. → `docs/AWARDS_SYSTEM.md`
- **기록 경신 마일스톤(구현)**: 개인 통산·구단·레전드 추월을 "사건"으로. → `docs/MILESTONE_SYSTEM.md`
- **플레이어 업적(2026-06 구현)**: 구단주 본인의 장기 성취(우승·시상·레전드·기록·운영)를 트로피로.
  누적 데이터 파생·저장 없이 재계산(관전형 척추 — 자발적 목표). → `docs/ACHIEVEMENT_SYSTEM.md`
- **선수 특성(구현)**: 클러치·대기만성·유리몸 등 긍정+부정. id 시드 결정론, 엔진은
  `player.traits`만 읽음(소폭 보정). → `docs/TRAIT_SYSTEM.md`
- **부상(구현)**: 엔진 밖 시즌 계층에서 "그날 출전 명단"을 깎음(결정론 보존). 경미 위주,
  동시부상 상한 3. → `docs/INJURY_SYSTEM.md`
- **뉴스 피드(구현)**: 위 산출물을 읽을 수 있는 연대기로(캡스톤). → `docs/NEWS_SYSTEM.md`
- 데이터 정확도보다 "세월이 쌓여 만들어지는 이야기"가 목표 (백년야구 철학).

> **관전형의 정의(2026-06 확정, 06-11 표현 정정)**: 시즌은 앱 사용 중에만 진행되며 **푸시 알림은 없다.**
> 모든 서사 전달은 앱 안에서 볼 때 조용히 surface(피드·위젯·연표). 강제 관람 없음 —
> 단, 핵심 경험은 "보는 것"이므로 보고 싶어지게 만드는 것(경기 연출·빅매치 표시)이 제품의 일이다.

## 8. 기술 스택 & 아키텍처

- **프레임워크**: React Native + Expo (Expo Go 실행)
- **언어**: TypeScript
- **상태관리**: Zustand (가벼운 단일 스토어. 시뮬 상태가 크므로 selector 활용)
- **저장소(중요)**:
  - **MVP는 로컬 우선.** 오프라인 싱글 방치형이라 서버 불필요.
  - **현 구현: zustand persist + AsyncStorage 단독**(세이브는 리플레이 기반이라 작음 —
    base 스냅샷 + currentDay + results). `expo-sqlite`는 보류(필요해지면 도입, `db/schema.ts`는 TODO).
  - Supabase는 **나중에** 클라우드 백업·글로벌 명예의전당·랭킹 붙일 때 옵션으로.
    MVP에서 서버부터 깔면 방치형 본질과 어긋나고 개발만 무거워짐.
- **네비게이션**: expo-router (파일 기반)
- **시뮬 엔진**: 순수 TS 모듈(`/engine`). React 의존성 0. 시드 결정론. 단독 테스트 가능.
  - 엔진은 "상태 + 시드 → 다음 상태"의 순수 함수 집합으로. UI와 완전 분리.

## 11. 작업 원칙 (Claude Code 준수)

- 엔진(`/engine`)은 React/Expo에 의존하지 않는다. 순수 함수 + 시드. 테스트 먼저.
- UI는 엔진 상태를 "그리기만" 한다. 게임 로직을 UI에 두지 않는다.
- 관전형이 1순위. 어떤 기능이 "매 순간 손이 가게" 만들면 설계를 의심한다.
  반대로 "보는 경험"(경기 연출·중계 가독성)의 품질은 핵심 투자처다.
- 과금/부스트가 현실성·밸런스를 깨는 방향이면 넣지 않는다.
- Expo Go에서 항상 실행 가능한 상태를 유지한다(네이티브 모듈 추가 시 사전 확인).
- 새 설계 결정은 코드보다 먼저 이 문서에 반영한다.
- **통계 프로토콜**(`docs/STATS_PROTOCOL.md`): 표본 1만+, 문서 기입 시 `(N · 엔진 커밋 · 날짜)` 필수,
  엔진 로직 변경 시 기존 통계는 재측정하거나 "재시뮬 필요"로 무효 처리.

### SOLID / 시스템 분리 (수정 파급 최소화)
- **단일 책임**: 각 시스템(훈련/노쇠/경기/생산/연봉…)은 `engine/`의 독립 순수 모듈.
  훈련을 바꿔도 경기·생산이 영향받지 않게 한다.
- **의존 방향**: UI(`app/`) → 셀렉터(`data/`) → 엔진(`engine/`). 역방향 금지.
- **계약(타입)에만 의존**: 엔진끼리는 구현이 아니라 출력 타입에만 의존한다.
  예) `production`은 경기 구현을 모른 채 `SimResult` 형태에만 의존 → 경기 엔진 교체 가능.
- **오케스트레이션은 함수 호출 합성**: "진행"은 `advance`(무엇을 할지) + 진화 셀렉터(훈련/노쇠)
  + `simMatch`(경기) + `production`(집계)을 **호출로 조합**한다. 시스템은 갈아끼울 수 있어야 한다.
- **결정론 유지**: 모든 시스템은 시드 기반 순수 함수. 상태는 저장 대신 `currentDay`+`results`에서
  재계산(리플레이)한다.
