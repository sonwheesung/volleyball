### v2 엔진 실제 로직 요약 (`simulateMatch(seed, home[], away[], { edge?, home?, away? })`)
- 입력: 양 팀 **로스터 Player[]** + 시드 + (선택) 감독 정보 `{ style, charisma }`(없으면 밸런스/50).
- 세트마다 기세 50·회전 0 리셋, 1~4세트 서브권 홀수세트 홈·짝수세트 원정 교대, **5세트 첫 서브는 코인토스**(아래 v2.1), 타임아웃 2회. 랠리포인트제 25/15·듀스·3선승.
- 한 랠리: **서브 타입 선택**(에이스/범실/난이도) → 포지션 폴트 → 리시브 품질(서브 난이도만큼 하락)로 시작
  → 공격 종류 선택(찬스볼이면 속공 불가) → 세터 승수×케미×리시브품질×기세×blockAvoid×체력효율 →
  **블로킹 3축** vs 디그 → 블록아웃/스터프(실점)/소프트(수비 전환)/디그(공수 전환 hop 최대 8).
- **타임아웃(7.4)**: 상대 연속득점이 임계(수비형 3·밸런스 4·공격형 5) 도달 시 지는 팀이 호출 →
  양 팀 기세를 50으로 수렴(폭 = 호출 감독 카리스마). 좋은 흐름일 때 부르면 자기도 깎임.
- **감독 성향(8장)**: 서브 공격성·속공 비중·스터프/소프트·디그 이점·타임아웃 타이밍에 반영(트레이드오프).
- 체력: 코트 인원이 공격·블록마다 소모(체력 높을수록 덜 지침), 랠리·세트 사이 체젠으로 회복 →
  긴 랠리·5세트에서 효율 저하(0.70까지). 체력/체젠 스탯이 후반 경기력을 가른다(아래 7.1 검증).
- 사이드아웃 시 득점팀 회전 + 서브권 이전. 출력은 `SimResult`(간이판과 동일 계약).

#### v2.1 5세트 첫 서브 = 코인토스 (2026-06-25 — 배구 도메인 검수)
> **왜**: 독립 세션 도메인 검수(배구 규칙만 인지)에서 "5세트 첫 서브가 코인토스가 아니라 결정론적으로 홈"임을
> 유일한 실제-규칙 불일치로 지적. FIVB/KOVO는 **5세트(결승 세트) 시작 전 새 코인토스**로 서브/코트를 정한다
> (1~4세트의 1-2-1-2 교대를 그대로 잇지 않는다). 구 코드 `serving = setNo%2===1?home:away`는 5세트(홀수)에
> 항상 홈이 먼저 서브 → 결승 세트에 체계적 편향.
- **측정(`_dv_firstserve`, 거울매치 A vs A, N=20,000 · 엔진 afa82ba)**: 첫 서브팀 세트 승률 **48.98%**(z=−5.8σ)
  — 받는 팀이 사이드아웃 이점으로 ~1% 유리(실제 배구와 동일 방향, **엔진이 규칙을 올바로 재현**). 그 탓에
  5세트 홈 승률 **48.76%**(z=−2.0σ)로 홈이 결승 세트에서 ~1% 불리. 단 **매치 승률 49.91%**(z=−0.2σ)로
  매치·시즌(홈/원정 교대)에서는 상쇄 — 밸런스 사안이 아니라 **규칙 정합성** 사안.
- **수정**: 5세트만 첫 서브를 전용 사이드스트림 코인(`(seed ^ 0x517cc1b7)` 기반 RNG 1드로우)으로 50/50 결정.
  메인 랠리 RNG 스트림 불간섭 → 1~4세트·코인=home인 5세트는 바이트 동일, 코인=away인 5세트만 변경(의도된 변화).
  결정론 보존(시드 고정 = 동일 결과). 검증: 수정 후 5세트 홈 승률 ~50% 수렴 + 코인 ~50/50(`_dv_firstserve` A/B).

  `setNo%2`로 재도출하면 어긋난다**(코인=away인 5세트에 home이 서브하는 것으로 잘못 그림). 그래서
  `SimResult.setFirstServers: Side[]`(세트별 첫 서브 팀, 인덱스=세트−1)를 엔진이 채워 보내고, 소비자는 그걸 읽는다:

  - 검증 `_dv_firstserve` (C): **엔진 `setFirstServers[4]` == 독립 오라클(첫 set5 point recvId) == 보드 `reconstructRallies` 시작 서브**

- **스위칭(1.5)**: 서브 직후 전 선수가 전문 포지션으로 슬라이드. 세터는 **자기 팀 공격 때만 네트 침투**, **수비 땐 리시브 준비**. OH 좌·MB 중앙·OP 우.
- **블로커 형성(교차 방지)**: 공격수에 가까운 인원이 **자연 좌우 순서 유지**하며 모인다(왼쪽 선수가 오른쪽서 블록하는 교차 없음).
- **공격 커버 + 미끼**: 토스 받는 동안부터 커버 형성하되, **전위 일부는 속공/페이크(미끼)로 네트에 남아 커버 못 옴** → 커버 인원 가변(무조건 3명 X). 공격수·세터·미끼 제외한 선수가 공격수 뒤로.
- **루스볼 쫓기**: 디그 실패·리시브 실패 시 가까운 선수들이 공 쪽으로 달려간다(`chasersTo`). 리시브 실패 공은 **라인 밖/사이드로** 튕겨 나간다.
- 토스는 항상 세터에서 **다른 공격수**에게 보낸다 — **본인한테 토스 금지.**
- **스파이크 코스**: 수비수가 아니라 **상대 코트의 다양한 위치**(좌우·깊이)로 때린다(`spikeTarget`).
- **리시브/디그 패스**: 세터 자리 정확히가 아니라 **주변 일정 범위로 랜덤** 낙하. **세터가 닿으면 세터가, 못 닿으면(난조) 가장 가까운 다른 선수가 대신 토스**(아웃오브시스템).
- **블로킹 인원**: **세터 토스(인시스템)=2장**, **다른 선수 토스(아웃오브시스템)=3장**(갱블록). 블록 안 가는 전위는 뒤로 빠져 수비.
- **점프 중 이동 금지**: 마커가 점프(서브·토스·스파이크·블로킹) 중이면 제자리 고정, **착지 후에 이동**(공중에선 못 움직임).
- **공 궤적**: **경기 중(인플레이)에만** 흰색 점선(`trailDot`)으로 표시(서브하러 가는/되돌아오는 구간·득점 멈춤 땐 숨김). 점선 끝점은 **의도(aim)** — 터치아웃은 점선=공격수가 원한 코트 코스, 실제 공은 네트서 블록에 스쳐 아웃.
- **수비 움직임**: 토스 때 블로커가 공격수 쪽 네트로 형성, 스파이크 때 **가장 가까운 후위 수비가 낙구점으로 이동**해 디그.
- **수비 후위 부채꼴**: 공격 빌드업·스파이크 때 후위 디거 3명이 **코트 전체 폭(좌·중·우)을 커버하되 공격 방향으로만 살짝 치우침**(가운데 얕게=팁, 양쪽 깊게=라인/크로스) — 한곳에 안 뭉침.
- **스파이크 경로 vs 블록(여자배구: 원터치 빈번)**: 의도 코스가 **블록 폭 안이면** → 블록아웃(att 득점)/스터프(블로킹 당함)/원터치 디그, **각으로 빠지면** → 클린 킬/디그. 블록 폭은 2장 좁게·3장 넓게.
- **재생 속도**: 전역 `SPEED`(현재 2 = 2배 느리게). `DUR`/마커 이동에 곱해짐. 2x 버튼은 그 위에 ×0.4.
- **리시브 미스(투터치 등, `RECV_FAULT`)**: 더블 콘택트는 토스가 아니라 **리시브 단계**에서. 지는 쪽이
  가끔 서브 리시브를 흘려 자기 코트에 떨어뜨려 실점(엔진 득점 결과와 일치하는 랠리에만).

> 튜닝 상수(모두 `MatchCourt.tsx` 상단): `DUR`·`ARC`·`BALL_SCALE`·`JUMP`·`SERVE_OUT`·`RECV_FAULT`,
> 마커 이동 300ms. 보드는 간이 연출이며, 실제 타자/블로커 등 정밀 이벤트는 엔진이 직접 출력하도록 강화 가능.

## 1. 로테이션 시스템

### 1.1 사이드아웃 룰 (회전 조건)

랠리포인트제에서 득점은 항상 발생하지만, **로테이션은 서브권이 넘어올 때만** 돈다.

```
랠리 종료 → 득점한 팀 판정
  ├─ 직전 서브권 팀이 득점  → 회전 없음, 같은 선수 계속 서브
  └─ 서브권 없던 팀이 득점   → 서브권 획득(사이드아웃) + 시계방향 1칸 회전
```

- 회전 = 6명이 포지션 1→6→5→4→3→2 순서로 한 칸 이동. 새 1번이 서브.
- 구현: `rotation` 인덱스(0~5)를 사이드아웃 시에만 +1.

### 1.2 전·후위 규칙

- 전위(2·3·4번)만 네트 앞 점프 공격·블로킹 가능.
- 후위(1·5·6번)는 공격선(3m) 뒤에서만 점프 공격 가능(백어택).
- 세터가 전위냐 후위냐로 공격 옵션이 바뀜(전위 세터는 공격 가담 가능, 후위 세터는 토스 전념).

### 1.3 리베로 교체 룰

- 리베로는 **후위에서만** 출전. 서브·네트 위 공격·블로킹·전위 세트(점프 토스) 불가.
- 미들블로커(MB)가 전위→후위로 회전하는 순간 리베로와 교체(MB 벤치 OUT, 리베로 IN).
- 리베로가 후위→전위로 가야 하는 순간 다시 MB 복귀(리베로 OUT, MB IN).
- 이 교체는 심판 허가 없이 랠리 사이에 자유롭게(정규 교체 횟수 미소모).
- 구현: 후위 3슬롯 중 MB 자리에 리베로를 스왑하는 로직. 회전 시마다 검사.

```
회전 후 검사:
  for each MB in 후위 슬롯:
     해당 슬롯 → 리베로로 치환 (벤치의 MB는 대기)
  for each MB가 전위로 복귀:
     리베로 → MB로 원복
```

### 1.3b 작전 교체 — 서브 전문 / 블로킹 강화 / 수비 강화 ★ (설계 2026-06)

> CLAUDE.md 3장 "세트 중 선수 교체"의 구현 설계. 엔진이 개별 선수를 로테이션으로 추적하므로
> (서브=`six[serverIndex]`, 블록=전위 정렬, 디그=후위 수비수) **교체가 실제로 결과를 바꾼다.**
> 방치형 원칙: **프리셋/방침 레벨**(매 토스 수동 금지). AI가 평소 자동 적용, 플레이어는 중요 경기에서 프리셋 토글.

**교체 예산**: 실제 배구처럼 **세트당 6회**. 한 선수의 IN+OUT 왕복 = 2회 소모. 리베로 교체는 별도(미소모, 1.3).

**팀 프리셋 `SubPolicy`** (감독이 자동 집행 — 엔진 `DEFAULT_POLICY`. 구단주 수동 토글은 2026-06-18
제거: 관전형 강화, 타임아웃 제거와 같은 결. 엔진 교체 매니저 자체는 그대로 — AI가 늘 적용):

| 정책 | 트리거 | 동작 | 엔진 효과 |
|---|---|---|---|
| **핀치 서버** | 약한 서버(세터 등 `skServe` 하위)가 서브 차례(zone1) + 벤치에 월등한 서버 + 예산 | 그 로테이션 서브를 벤치 서버가 수행, 다음 회전에 원복 | `server` 의 `skServe`↑ → 에이스율↑. **벤치 신인 출전 → 서브 XP(육성 채널)** |
| **블로킹 강화** | 접전(듀스 근접·세트 후반) + 전위에 약한 블로커 + 벤치 장신 블로커 + 예산 | 전위 슬롯에 블로커 투입(공격 차례 끝나면 원복) | `blockEval` 전위 block↑ → 스터프율↑ |
| **수비 강화** | 상대 강서버 로테이션 + 리시브 약한 후위 선발(예: OP) + 벤치 수비수 + 예산 | 후위 슬롯에 수비 스페셜리스트 투입(서브-리시브 국면) | `defenders` 리시브/디그↑ → 에이스 허용↓ (리베로 후위 MB는 이미 1.3에서 커버) |

**결정론**: 교체 판정은 시드 RNG + 현재 상태(로테이션·점수·예산)만 사용 → 같은 시드 = 같은 교체 = 같은 결과(리플레이 보존).

**구현 위치**:
- `engine/lineup.ts` — 선발 6 외 **벤치 역할별 후보**(서버/블로커/수비수) 추출.
- `engine/match.ts` set 루프 — 회전·서브 직전에 `SubPolicy` 평가 → 그 랠리용 `effectiveSix` 산출(예산 차감).
- `types` — `SubPolicy`. `MatchOpts` 에 팀별 정책 주입(미지정 시 감독성향 기본값).

**育成 연결**: 핀치 서버/수비 교체로 벤치 신인이 코트에 서면 `production`→`applyMatchXp` 로 성장(1.3b가 1.7 훈련 외 출전 성장원을 늘림). "신인 서브만이라도 투입"이 곧 육성.

**페이즈**: **① 핀치 서버 · ② 블로킹 강화 · ③ 수비 강화 — 전부 ✅ 구현(2026-06)**.

**구현·검증**: `engine/match.ts` 통합 교체 매니저(`activeSubs` slot맵 + 예산), `types.SubPolicy`.
- ① 핀치 서버: 약한 서버(서브레이팅 차 ≥`PINCH_SERVE_GAP`12) 차례에 벤치 서버 IN, 서브권 상실 시 OUT.
- ② 블로킹 강화: 막판 접전(목표−4 & 2점차 이내)에 전위 약한 블로커를 벤치 블로커로(차 ≥`BLOCK_SUB_GAP`12). 후위行 시 OUT.
- ③ 수비 강화: 받는 측 후위 약한 리시버(MB 제외 — 리베로 커버)를 벤치 수비수로(차 ≥`DEF_SUB_GAP`12). 전위行 시 OUT.
- 세트당 6회 예산(왕복 2회), 결정론(상태 기반·RNG 무관), 슬롯 충돌 방지.
검증(3종 ON): 킬 56%·스터프 9.5%·에이스 6.1%(전부 KOVO 범위), 사이드아웃 58.3%, parity std 4.52·r −0.01(무손상), 70/70 테스트.

### 1.3c 교체 출전 → 육성 XP ✅ 구현(2026-06)

"신인을 서브만이라도 투입해 키운다"의 엔진. `simulateMatch`가 작전 교체로 코트에 선 선수의
**출전 랠리 수(`SimResult.subUse`)** 를 기록 → `attributeProduction`이 코트타임 비례로 출전 XP를
적립(`matches += min(1, 랠리/40)`, 풀세트≈1경기). 경기 결과엔 무영향(관측만), 시즌 누적 후 1.7 경로로 성장.

검증(`simPlayers` 60시즌): 신인 주전(72+) 도달 **17.7%→20.0%**, 리그 평균 OVR 63.4(더 안정),
parity std 4.61·r 0.05(무손상), 70/70 테스트. 교체로 출전 기회를 받은 벤치 신인이 더 큰다.

### 1.3d 부상 교체 — 경기 내 부상 → FIVB 예외적 교체 ★ (설계·구현 2026-07-07, 이번 패스)

> **문제**: 지금까지 `maybeInjure`(rally.ts, 공격수 대상)가 발동하면 선수를 `injured` Set에 넣어
> 효율 ×0.5(`INJ_EFF`)만 걸고 **선수는 코트에 계속 남는다**. 실제 배구라면 크게 다치면 교체된다.
> **이번 패스**: 그 "부상당한 채 뛰기"(흔한 경우)는 그대로 두고, **드물게(중상)만 실제 교체**로 승격한다.

**심각도 게이트 (필수)** — 부상 대부분은 "참고 뛴다"(현상 유지, ×0.5). `maybeInjure`가 발동하면
`injured`에 넣어 ×0.5를 거는 건 오늘과 동일하고, **추가로 시드 심각도 판정**
`rng.next() < SEVERE_INJURY_FRAC`(=`0.12` placeholder, 측정으로 튜닝)을 한 번 더 굴려 **중상일 때만**
교체를 발동한다. 교체 전까지(같은 랠리~직후)는 ×0.5. **기존 rng 스트림 사용**(결정론 — 새 스트림 없음).

**부상 교체 경로 (별도 — `applySub` 재사용 금지)**: 작전 교체(`subIn`/`subOut`)를 재사용하면
교체 예산·재진입 회계가 오염되고 세트말 원복이 **부상 선수를 되살린다.** 그래서 별도 경로로 슬롯을 직접 교체:
`slot = at.six.indexOf(injured); at.six[slot] = replacement`. 부상 선수는 **경기 끝까지 복귀 불가**(세트 넘어 지속).
- **FIVB 예외적 교체**: 부상 교체는 6회 예산·재진입 제한 **밖**이다 → `subBudget`·`usedSubIn`·
  `usedStarterOut`·`activeSubs`를 **건드리지 않는다**. 작전 교체 6회는 그대로 다 쓸 수 있다.
- **세트 넘어 지속**: 별도 구조 `injuryReplaced`(사이드별 slot→{out,in}). 작전 교체 복원 루프(match.ts
  "복원: 조건 안 맞으면 OUT")도, 세트말 원복 루프도 **이 슬롯을 절대 되돌리지 않는다.** 부상 교체 선수는
  경기 끝까지 코트에 남고, 부상 선수는 다시 못 들어온다.
- **부상 슬롯이 마침 작전 교체 중이면(엣지)**: 그 슬롯에 `activeSubs` 항목이 있으면 부상 교체를 적용할 때
  그 항목을 **삭제**한다 — 안 그러면 복원 루프/세트말이 부상 교체 선수를 빼거나 작전 원선발(`orig`)을
  되살린다. 부상 교체 선수가 영구히 남는다.
- **동적 벤치 선정**: 경기 시작 시 한 번 고른 `benchSpecialists`를 그대로 쓰지 않는다. 벤치에서
  `injured` id·현재 `six[]` id·이전 부상 교체 선수를 **제외**하고 고른다. **포지션 매치 우선**(특히
  벤치에 'S'가 있으면 세터는 세터로 — `setterOf`가 'S' 없으면 six[0]로 폴백하므로 유지 중요).
  **결정론 픽**(그 슬롯 레이팅 최고, **새 rng 없음** → 바이트 교란 최소).
- **전술 교체도 `injured` 배제 (2026-07-07 잠복버그 수정)**: `subIn`(작전 핀치/블록/수비 교체)도 들어오는
  선수가 `st.injured`에 있으면 **early return** 한다 → 부상 선수는 **전술 교체로도 코트 복귀 불가**(subIn·injuryReplaced
  이중 차단). 배경: 작전 교체가 쓰는 `benchSpecialists`는 **경기 시작 때 한 번 고정**이라, 부상 교체로 코트에 들어온
  벤치 스페셜리스트가 **다시 중상(이중부상)** 나서 `injured`에 들어가도 여전히 벤치 명단(예: 블록 스페셜리스트)에
  남아 있어, 이후 전술 교체가 그를 **재투입**할 수 있었다(FRAC=0.12에서 2571경기 중 0건이나 도달 가능한 잠복버그).
- **벤치 소진/후보 없음(폴백)**: 넣을 사람이 없으면 부상 선수를 **코트에 그대로 둔다**(×0.5, 현상 유지).
  **몰수·크래시 없음**("라인업 파탄 방지").
- **리베로는 도달 불가**: `maybeInjure`는 공격수(=`six[]` 멤버)에게만 호출되고 리베로는 공격하지 않는다.
  호출부 드리프트 방어용 주석/단언을 둔다. 리베로 대체 로직은 없음.

**결정론 격리 (유지)**: 이건 **경기 한정**이다. 시즌 계층 부상(`data/dynamics.ts`)에 **먹이지 않는다** —
부상 선수는 **다음 경기엔 멀쩡**하다(시즌 결장은 별도 시즌 롤; 결합하면 리플레이 격리가 깨진다). INJURY_SYSTEM 0장.

**보드 이벤트**: `SubEvent`에 `kind:'injury'` 추가·부상 SubEvent를 emit. 보드 문구는 **소프트**(들것 아님):
"몸 상태 · 교체" / "이상 · 교체"(🚑 중상 프레임 금지) — **시즌 영향이 0**이라서. 보드 라벨맵
(`MatchCourt.tsx SUB_KIND_KO`)·`courtDirector.applySubsToSix`가 `kind:'injury'`를 **영구(비원복) 스왑**으로 취급.

**교체 감사기 예외(중요)**: `tools/checkSubs.ts` net-zero 검사(모든 교체가 세트말 base로 원복)는
부상 교체를 통과시키면 실패한다(부상 교체는 의도적으로 net-zero 아님·세트 넘어 지속). `applySubsToSix`(보드 재생)와
`checkSubs.ts`(감사) 둘 다 `kind:'injury'`를 **영구 스왑**으로 보고 net-zero/재진입/세트당1회 합법성 검사에서 **면제**한다.

**ENGINE_VERSION 4→5**: 결과를 바꾸는 엔진 변경(부상 시 코트 선수가 바뀜) → 버전 +1 → 저장 sim 캐시 무효화.
**ENGINE_VERSION 5→6 (2026-07-07)**: `subIn`이 `injured` 선수를 배제(위 잠복버그 수정) → 드문 경우 six[]가 바뀌어
결과가 달라짐 → 버전 +1 → 저장 sim 캐시 무효화.

**검증**: `tools/_dv_injurysub.ts`(A/B) — (a) 결정론(같은 시드=동일 결과), (b) 부상 아웃 선수 재등장 0,
(c) 부상 교체가 예산/재진입 미소모(작전 교체 정상), (d) 심각도 게이트(교체율 ≪ 참고뛰기율), (e) N≥2000 경기 부상 교체율 출력.
`tools/checkSubs.ts`는 부상 면제 후에도 PASS 유지.

### 1.3e 피로 교체 — 지친 주전을 잠시 쉬게 (kind:'rest', 보드 "체력 안배") ★ (설계·구현 2026-07-07, 이번 패스)

> **배경(측정)**: 체력 감사(2,500경기·엔진 v7) — OH/OP가 5세트엔 78~81%까지 떨어지고, 5세트 경기의
> **12.2%** 에서 코트에 체력 30% 미만 선수가 있는데도 **감독이 아무 대응을 안 했다**(기존 작전 교체 3종은
> 핀치/블록/수비 = 레이팅-전술용이라 피로에 반응하지 않는다). 실제 감독은 지친 주전을 잠깐 뺐다가 다음
> 세트에 다시 넣는다. 이 패스가 그 "체력 안배" 교체를 추가한다. (핀치·블록·수비보다 **먼저** 평가.)

**발동 조건 (한 슬롯이 전부 만족해야)**:
- `player.position !== 'S'` — **세터는 절대 안 뺀다**(5-1 무결성; MB/L은 실전에서 잘 안 지친다).
- `stamina(st.stam.get(id) ?? 1) < REST_THRESHOLD(0.35)` — 주전이 실제로 퍼졌을 때만.
- **접전 종반이 아님**(`crunch` = `max(h,a) ≥ 목표−4 && |h−a| ≤ 2`) — 23-23에서 지친 에이스를 빼는
  감독은 없다. 보드가 교체 사유를 노출하므로 이건 **관전 신뢰성** 문제(가짜 장면 금지). 기존 `crunch` 재사용.
- `policyOf(side).restSub`(SubPolicy·DEFAULT_POLICY에 추가, 기본 켜짐).
- `subBudget[side] ≥ 4` — **핀치보다 높은 문턱**(일반 교체는 subIn 내부 ≥2). 피로 교체가 핀치 예산을
  굶기면 안 됨: 한 번 발동해도 5회 남아 핀치 왕복 2회분 확보. (핀치가 피로 교체를 굶히는 건 허용 — 우선순위 방향이 맞다.)

**교체 선수 픽 (결정론·rng 없음)**:
- **같은 포지션 벤치 최고 1명(엄격)** — 같은 포지션 벤치가 없으면 **교체 안 함**. load-bearing:
  `receivers()`가 `position==='OH'`로 리시브 W라인을 만들기 때문에 다른 포지션으로 바꾸면 리시브 라인이
  조용히 줄어든다.
- 제외: 코트에 있는 선수·부상(`injured`, subIn이 재검사)·이미 이 세트에 투입된 선수(`usedSubIn`). 벤치
  후보의 체력도 `st.stam.get(id) ?? 1`로 읽는다(핀치 서브를 이미 뛴 벤치 OH는 쌩쌩하지 않다).

**합리 코치 게이트 (가짜 드라마 방지)**:
- `roleScore(bench) × eff(bench) ≥ roleScore(starter) × eff(starter)` 일 때만 발동. `roleScore`는 기존 맵
  (OH=spike+receive 등) 재사용, `eff = 0.70 + 0.30×체력`(부상 ×0.5) — rally.ts와 동일(로컬 복제).
  85 레이팅 스타 vs 60 벤치라면 지쳐도 스타를 계속 굴린다(현실적 — 스타의 탈진은 버그가 아니라 서사다).
- **히스테리시스**: `벤치 체력 − 주전 체력 ≥ 0.3`(살짝 지친 걸로 스와핑 반복 방지).

**경로·수명**:
- 기존 `subIn(side, slot, player, 'rest')`로 라우팅 → FIVB 가드 전부 상속(예산·`usedSubIn`/`usedStarterOut`·
  부상 배제·`activeSubs` 슬롯 락). 슬롯당 세트당 최대 1회 자동 보장.
- **세트 중 원복 없음**: 복원 루프는 kind별인데 'rest'엔 절을 두지 않는다 → 세트말 net-zero 원복까지 유지
  (쉰 선수는 **다음 세트에 복귀**). 작전 교체와 동일한 net-zero(부상 교체와 달리 영구 아님).
- `SubKind` union(match.ts)·`SubEvent.kind`(simMatch.ts)에 `'rest'` 추가.

**보드**: `MatchCourt.tsx SUB_KIND_KO`·`_proofSubBadge.ts` KO맵에 `rest: '체력 안배'`. `checkSubs`·
`_dv_injurysub`는 'rest'를 일반 net-zero 작전 교체로 취급(부상 면제 대상 아님) — 정상 통과.

**공유 풀 주의**: `bench.defender`(벤치 최고 OH)와 피로 교체 픽이 같은 선수인 경우가 잦다 — 먼저 발동한
쪽이 세트당 1회 진입을 소비한다(결정론적 희소성, 허용).

**빈도**: 경기당 **0.05~0.5회**(드묾 — 5세트 접전에서만). 가드가 이 밴드 밖이면 FAIL(`simStamCurve` 2차 단언 — 기능이 조용히 죽거나 폭주하는 것 방지).

**ENGINE_VERSION 7→8 (2026-07-07)**: 피로 교체(A) + TTO 회복 재튜닝(B)이 둘 다 결과를 바꾼다 → 버전 +1 →
저장 sim 캐시 무효화.

**검증**: `tools/simStamCurve.ts`(새 지표·피로 교체율 밴드), `tools/checkSubs.ts`(net-zero·FIVB 합법),
`tools/_dv_injurysub.ts`(작전 교체 무회귀), 인라인 스팟증명(접전 중 미발동·세터 미발동·다음세트 복귀·게이트).

### 1.4 포지션 폴트 (VQ로 구현 — 네 요청 핵심)

서브 임팩트 순간 오버랩 규칙을 어기면 폴트 → 즉시 실점.
프로도 가끔 내는 실수("로테이션 안 바꿔서 실점")를 VQ로 확률화한다.

```
teamVQ = avg(6명 onCourt VQ)              // 평균 VQ로 확정
faultChance = clamp(0.024 × (1 - teamVQ/100), 0, 0.04)   // 받는 팀만 판정·계수 2배 — engine/rally.ts:~415
```

- **받는 팀만 판정 (2026-07-07 정정, FIVB 2025-2028 Rule 7.4 + KOVO 25-26 채택)**: ~~서브 팀·받는 팀 **양쪽 모두** 폴트를 굴렸다~~ → 서브 순간 **서브 팀 전원은 위치 자유**(오버랩 면제)이고 오버랩 폴트는 **받는 팀만** 성립. 그래서 서브 팀 판정을 제거하고 받는 팀만 굴리되, 계수 `0.012→0.024`·상한 `0.02→0.04`로 **총 기대 폴트율을 보존**(KOVO 상대범실 분포 정렬 유지, 현실적으로 오버랩 폴트는 받는 팀 몫). rng 소비 2→1회(스트림 이동) → `ENGINE_VERSION 7`.
- ~~`faultChance_base = 0.015 × situationFactor`~~ → **실제: 기저 0.024(구 0.012의 2배) + clamp [0, 0.04]**(코드와 일치, 2026-06-27·2026-07-07 정정).
- `situationFactor`(복잡한 로테이션·연속 실점 가중)는 **미구현(설계 단순화)** — 현재 폴트는 `teamVQ` 단일 의존.
  (향후 구현 시 이 항 부활.)
- 폴트 발생 시 로그: "포지션 폴트 — 상대 득점". 드물지만 장기 시즌에서 누적되어
  "VQ 낮은 팀은 어이없는 실점이 쌓인다"는 현실감 제공.
- 신인/유망주는 VQ 낮게 시작 → 경험으로 성장(데이터 누적 서사와 연결).

### 1.5 스위칭 (서브 직후 위치 이동)

- 서브 임팩트 순간에만 오버랩 위치 유지, 직후 전문 포지션으로 이동.
- 시뮬에서 코트 좌표를 매 프레임 계산하지 않으므로, **추상화**한다:
  - 서브 순간: 포지션 폴트 판정만 수행(1.4).
  - 폴트 안 나면: 곧바로 "전문 포지션 배치"로 간주하고 능력치 계산.
- UI(경기 보드)에서는 서브 순간 마커를 로테이션 위치에 두었다가,
  서브 직후 전문 위치로 마커를 슬라이드시켜 시각적으로 표현(선택).

## 2. 서브 시스템

서브 타입을 선수 성향/작전으로 선택. 각 타입은 (에이스, 범실, 리시브 난이도) 트레이드오프.

| 서브 타입 | 에이스 확률 | 범실 확률 | 리시브 난이도 | 특징 |
|---|---|---|---|---|
| 언더/안전 서브 | 매우 낮음 | 매우 낮음 | 낮음 | 거의 안 씀(프로). 위기 시 안전책 |
| 플로터 서브 | 낮음 | 낮음 | 중 | 무회전 흔들림. 안정적, 코스 변화 |
| 점프 플로터 | 중 | 중 | 중상 | 플로터 + 점프. 균형형 |
| 스파이크(점프) 서브 | 높음 | 높음 | 높음 | 강력. 하이리스크 하이리턴 |

```
serveOutcome = f(server.skServe, server.focus, serveType)
  ace_p   = base_ace[type]   × (server.skServe/100) × (1 - receiver.recv/100)
  error_p = base_error[type] × (1 - server.focus/100 × 0.5)
  난이도   = base_diff[type]  → 리시브 품질(2.x)에 음수 보정
```

- 작전/감독 성향으로 "공격적 서브(스파이크 서브 비중↑)" vs "안정 서브" 선택.
- 접전(듀스)에서는 멘탈·focus가 범실 확률에 크게 작용.

## 3. 리시브 & 수비 범위

### 3.1 리시브 품질

서브를 받는 즉시 다음 공격의 질을 좌우. 3단계 품질로 이산화.

```
recvQuality = f(receiver.recv, receiver.positioning, receiver.reaction, 서브난이도)
  → A(정확):  세터에게 정확 → 모든 공격 옵션 가능(속공 포함)
  → B(보통):  오픈/일부 옵션만
  → C(흐트러짐): 찬스볼/2단 공격만(7장 연계)
```

**리시브 담당 = 리베로 + OH 전원(W형)** — `receivers(t)`. 현실 KOVO 5-1처럼
세터·아포짓(OP)·MB는 리셉션에서 숨긴다(서브 리시브 안 함). 후위 디그(`defenders`)와
구분: 디그는 후위 3인(MB↔리베로 교체) 전원이 담당하지만, **서브 리시브는 W 대형의
리베로 + 아웃사이드만**. 그룹 강도 = `0.5·max + 0.5·avg`(리베로가 항상 max → 실효 ≈ ⅔).
> 효과: 2번째 OH 리시브 스탯이 실제로 반영되고, 리시브 약한 세터/OP를 보유한 팀이
> 과도한 패널티를 받지 않는다(에이스 5.9%·사이드아웃 ~58%, KOVO 유지).

### 3.2 수비 범위 (네 질문 — 넣기로 결정)

넣는 걸 추천. 누가 공을 받느냐를 "범위"로 결정해 개성을 만든다.

- 각 수비수에 `coverage` = f(agility, reaction, positioning, dig). 리베로는 보너스.
- 애매한 코스의 공이 떨어질 때, 후보 수비수 중 coverage 가중으로 처리자 결정.
- 수비 좋은 선수/리베로가 인접한 약한 수비수 영역까지 커버 → 디그 성공률↑.
- 프로라 다들 기본기 높지만, 이 범위 차이가 "리베로가 다 받아낸다" 같은 장면을 만듦.
- 구현 부담이 크면 MVP에선 단순화(처리자 = 후위 중 dig 최고)하고 후순위로.

### 3.3 공간 텔레메트리 (서브~득점 좌표 기록, 2026-06-09)

랠리의 **모든 순간을 구조적 좌표 이벤트로 기록**해 "공이 엉뚱한 곳에 가지 않는지"를
검증한다. 코트는 9m×18m(네트 y=9), 미터 단위(`engine/court.ts`). `simulateMatch(.., {events})`로
켜며, **좌표용 난수는 랠리별 독립 srng**라 메인 결과 RNG를 건드리지 않는다 → **승패·킬 55.8%·
파리티 전부 불변**(`events.test.ts`로 못박음). 추상 엔진이 정한 결과(q·킬·스터프)에서 좌표를 *파생*.

| 이벤트 | 기록 내용 |
|---|---|
| `serve` | 서버·종류·출발/의도목표/실제낙하·오차(m)·`in/ace/fault` (범실=코트 밖) |
| `receive` | 패서(리베로/OH)·위치·공 위치·도달거리·`good/poor/shank/ace`(난조일수록 거리↑) |
| `set` | 토서(인시스템=세터)·세터위치·타점·토스낙하·오차·인/아웃오브시스템·공격종류 |
| `attack` | 공격수·종류·타점·코스(킬=상대코트 안, 범실=밖, 디그=디거 쪽) |
| `block` | 블로커 명단·포지션·인원(1~3, 리베로 불참) |
| `dig`·`point` | 디거·위치·도달거리 / 득점자·사유 |

검증 도구 `tools/simMoments.ts`: 한 랠리 좌표 추적 + 대량 무결성 단언(범실 100% 코트밖,
토스 100% 자기코트, 킬 100% 상대코트) + 6항목 분포. **미모델(3단계 예정):** 토스 타이밍→속공
불발, 세터 2어택(현재 세터 공격은 전위에 OH/OP 없는 로테이션의 폴백 ~0.01%뿐).

## 4. 토스(세트) & 공격 전술

리시브 품질 + 세터 능력 + 작전에 따라 공격 종류 결정. 종류별로
(블로킹 회피력, 성공 기대치, 난이도) 다름.

| 공격 종류 | 토스 | 블로킹 회피 | 비고 |
|---|---|---|---|
| 오픈(하이볼) | 높게 사이드로 | 낮음 | 블로커 다 따라붙음. 공격수 단독 능력 싸움 |
| A퀵(A속공) | 세터 1m 내 짧고 빠르게 | 높음 | 미들블로커 전유. 블로커가 못 따라옴 |
| B퀵(B속공) | 세터 1~2m | 높음 | 미들. A보다 약간 넓은 거리 |
| 시간차 공격 | 속공 페이크 후 뒤늦게 | 높음 | 블로커 타이밍 무너뜨림. 세팅·호흡 필요 |
| 이동 공격 | 미들이 이동하며 | 중상 | 블로커 따라가기 어려움 |
| 백어택(파이프 등) | 후위에서 3m 뒤 | 중 | 후위 공격수 가담. 공격 루트 다변화 |

```
attackSelect = f(recvQuality, setter.skSet, setter.VQ, 작전방침)
  - recvQuality=A: 속공·시간차 포함 전 옵션
  - recvQuality=B: 오픈·백어택 위주
  - recvQuality=C: 7장 찬스볼 처리
blockAvoid[공격종류] → 블로킹 차단 확률에 음수 보정
```

**구현 현황(2026-06-09):** 엔진 `Atk`는 `quick`·`tempo`(시간차)·`open`·`back` 4종.
`quick`은 다시 **A퀵·B퀵·이동속공(slide)** 으로 세분(`quickKindOf`) — **난수 없이 상황으로 결정론
분류**(패스 q·세터 VQ·미들 신장/슬라이드 성향). 좋은 패스+영리한 세터→A퀵, 좋은 패스+슬라이드형
장신 미들→이동속공, 그 외→B퀵. 토스 타점도 종류별로(A퀵 세터앞 1m·B퀵 레프트 2.4m·이동속공
세터뒤 1.6m, 공간 텔레메트리에 기록). **현재는 분류·좌표만 — 블록회피/성공률은 셋 다 `quick`
공통**(밸런스 불변, kill 55.8% 유지). 종류별 차등 효과는 3단계 과제. 분포 측정: A퀵 ~37%·B퀵
~42%·이동속공 ~21%(`tools/simMoments.ts`·`simTrace.ts`).

- 세터의 `skSet`·`VQ`가 높을수록 빠른 공격(속공·시간차) 선택·성공률↑ → "팀 공격의 승수".
- AI 감독/작전 방침으로 "속공 위주", "백어택 활용" 등 분포를 조절(매 토스 수동 선택 아님).

## 5. 공격 마무리 & 블로킹

### 5.1 스파이크(공격 마무리)

공격수가 어떤 방법으로 때리는지. 블로킹·디그 상황과 교차해 결과 분기.

| 공격 방법 | 효과 |
|---|---|
| 강타(파워) | 정직한 힘 싸움. atk·점프 높으면 강력 |
| 크로스(대각) | 코트 넓게. 일반적 |
| 스트레이트(직선) | 라인 타격. 블록 사이드 비면 효과적 |
| 블록 아웃(터치 아웃) | 블로커 손 맞고 아웃 유도. 블록 강할 때 노림수 |
| 페인트(연타/속임수) | 강타 페이크 후 살짝 넘김. 상대 수비 전진 시 효과 |

```
spikeResult = f(hitter.atk, hitter.VQ, 공격방법, blocker 상태, 후위 digger 배치)
  - 블록이 강하게 섰다 → 블록아웃/페인트가 유리
  - 상대 수비가 앞으로 쏠림 → 페인트 성공률↑
  - 공격수 VQ 높을수록 "상황에 맞는 방법" 선택(코스 선택 정확)
결과: 득점 / 블로킹 차단 / 디그(랠리 지속) / 공격 범실
```

> **스파이크 레이팅 스케일 보정(2026-06-09):** 기존 `spike` 공식이 `×consistency`(−30%)와
> 약한 키 항으로 다른 레이팅보다 ~30% 낮게 나와(평균 40 vs 디그·리시 65~70) OVR이 공격수를
> 과소평가했다. `ratings.ts`에서 키를 풀레인지(`(키−165)/30`)로, consistency 곱을 약보정
> (`0.8+0.2·c`)으로 바꿔 **스파이크 평균 40→63**(타 레이팅과 정합). 표시 OVR에서 OP/OH/MB가
> 제자리(예: OP 59→68). **엔진 화력은 `rally.ts ATK_K=0.64`로 옛 캘리브레이션에 고정** —
> 표시 spike(n≈0.63)×ATK_K ≈ 옛 spike(n≈0.40). 결과 킬 55.8%·스터프 9.5%·에이스 5.9%
> (전부 KOVO 유지), 70/70 테스트·결정론·파리티 무손상.

> **반응속도 가중 재밸런스(2026-06-14):** 반응속도는 리시브(0.35)·디그(0.25)·블록(0.2) 3종 레이팅에
> 깔려 합산 레버리지 0.80 → 통제 실험에서 **80.1%로 과대**(전용 기술들보다 위, 리시브에선 반응=리시브기술
> 동급이라 기술이 묻힘). `ratings.ts`에서 **리시브 반응 0.35→0.25·리시브기술 0.35→0.45**,
> **디그 반응 0.25→0.20·디그기술 0.25→0.30**(블록은 0.2 유지). 결과: 반응 80.1→**74.6%**(점프·위치선정과
> 동급으로 "공통 속성"답게), 디그기술 67.5→**70.6%**·리시브기술 60.3→**62.5%**(전용 기술이 주역).
> KOVO 박스스코어 전 항목 유지·parity 개선(σ 3.45→3.28·왕조 6.3→4.6)·166 테스트·결정론 무손상. → `tools/simStatEffect.ts` 기준표.

- 공격수 VQ가 코스 선택의 영리함을 결정 → 단순 파워형 vs 영리한 공격수 구분.

### 5.2 블로킹 시스템

블로킹은 3개의 독립 축으로 정의된다. 각 랠리에서 블로킹은 (축1, 축2, 축3)의 조합으로 결정.

**축 1 — 목적(블로킹 타입)**

| 타입 | 효과 | 리스크 |
|---|---|---|
| 공격적 블로킹(스터프) | 차단 시 즉시 득점 확률↑. 손목 꺾어 코트로 찍음 | 빗나가면 블록아웃·터치아웃·네트터치 범실 위험↑ |
| 소프트 블로킹 | 직접 득점 적지만 공 감속 → 자기팀 디그 성공률 크게↑ → 역습 셋업 | 안전. 강공을 죽여 받기 좋게 만듦 |

```
blockType 선택 = f(blocker.VQ, 작전방침, 상대 공격 강도)
  - 상대 공격수가 강하고 타점 높음 → 소프트로 살려서 디그 노림이 합리적
  - 차단 각이 좋고 블로커 우위 → 공격적 블로킹으로 득점 노림
```

**축 2 — 인원수(원/투/스리블록)**

- 블로킹 가담 인원 = 1~3명(전위 3명 한도).
- 인원↑ → 차단 확률↑ + 코스 커버 넓어짐. 단, 코트 다른 곳이 비고 **블로킹 커버
  인원이 줄어듦**(5.2의 커버 항목과 트레이드오프).
- 기본형: 센터(MB) 중심 + 해당 사이드 윙이 가담하는 **투블록**. 윙이 직선,
  미들이 크로스를 담당해 양쪽을 동시 제어.
- 오픈(하이볼)처럼 공격 루트가 노출되면 스리블록 성공률↑. 단 약한 공격수에
  기계적 스리블록은 비효율(다른 곳이 빔).
- 속공/시간차엔 1~2명만 따라붙기 벅참.

**축 3 — 타이밍(리드 vs 커밋)**

| 방식 | 설명 | 조건 |
|---|---|---|
| 커밋(게스) 블록 | 토스 전에 공격 루트를 예측해 미리 점프 | 현대배구 기본. 예측 빗나가면 헛손질(블록 0) |
| 리드 블록 | 토스를 본 뒤 따라가 공격수 앞에서 점프 | 반응속도·VQ 높아야 가능. 못 따라가면 유효블록만 |

```
blockTiming = f(blocker.reaction, blocker.VQ)
  - reaction·VQ 높음 → 리드 블록 가능 → 페이크(속공·시간차·싱크로)에 덜 속음
  - 낮음 → 커밋 블록 의존 → 세터의 다지점 공격에 헛손질 잦음
```

> 설계 핵심: **싱크로/속공/시간차 공격이 강한 이유 = 상대가 커밋 블록이라서.**
> 세터가 여러 지점에 올려 블로커를 헷갈리게 → 헛손질 유도 → 1대1/노블록. 상대가
> 리드 블록을 잘 구사할수록(reaction·VQ↑) 이 페이크 효과가 줄어든다.

**블로킹 커버 (블록 맞고 튄 공 살리기)**

- 공격 시, 블록에 맞고 떨어질 공을 대비해 **공격 비가담 선수**가 커버 포지션.
- **공격에 가담해 점프한 선수는 커버 불가**(자기 동작 중). → 싱크로 공격으로
  여러 명 띄울수록 블록은 헷갈리지만 커버 인원이 줄어드는 **트레이드오프**.
- 커버 성공 시 랠리 지속(공 다시 셋업, 단 품질 낮음 → 찬스볼/2단 공격).

```
coverPlayers = onCourt - (이번 공격에 점프한 공격수들)
coverChance  = f(coverPlayers 수, 그들의 dig·reaction·positioning)
공격 가담 인원↑ → 블록 교란↑  &  coverChance↓   (트레이드오프)
```

**공격 vs 블로킹 상호작용 (스파이크 결과 분기에 반영)**

5.1의 `spikeResult`에 블로킹 상태를 교차시킨다. 공격수가 블록을 이기는 경로:

- **코스 회피**: 블록이 대각 막으면 스트레이트로, 정면 막으면 코스 변경(공격수 VQ·기술)
- **블록 위로**: 타점(키+점프력)이 블록보다 충분히 높으면 블록 위 통과(괴물 용병형)
- **블록아웃/터치아웃**: 블록 손 맞고 코트 밖 유도(블록이 강하게 섰을 때 노림수)
- **리바운드(자가 셋업)**: 살짝 때려 블록 맞고 튄 공을 자기팀이 다시 공격 전환
- **블로킹 커버**: 위 커버 항목. 차단당해도 커버가 받아 랠리 지속

```
공격 성공률 보정:
  + (hitter.타점 - block.높이)            // 신체 우위
  + hitter.VQ × (블록 형태 vs 선택 코스 적합도)   // 영리한 코스
  - block.차단력 × (인원수, 리드/커밋 적중)
결과: 득점 / 블로킹 차단(상대 득점) / 블록아웃(공격 득점) /
      커버 성공(랠리 지속) / 디그(랠리 지속) / 공격 범실
```

## 6. 찬스볼 (네 요청 반영)

수비가 흐트러져 겨우 살린 공. 정상 공격 루트가 깨진 상태.

- 발생: 디그는 성공했으나 품질 낮음(recvQuality=C에 해당) → 세터가 정위치에서
  못 잡음.
- **세터가 못 잡아도 "끝"이 아니다**: 코트 위 선수 중 `skSet`(세팅 기술)이
  일정 이상이면 누구든 토스 가능. 단 품질은 세터보다 낮음.

```
chanceBallSetter = 코트 위 선수 중 skSet 최고(세터 우선, 없으면 차선)
tossQuality = setterSkill × penalty(0.5~0.8)   // 정상 세트보다 하향
→ 이 낮은 품질로는 속공/시간차 불가, 오픈·백어택만 가능
→ 다음 공격 성공률 전반 하락
```

- 함의: 세터가 아니어도 세팅 좋은 선수(예: OH, 베테랑 리베로)가 적당히 올려줌.
  "센터가 공 안 잡았다고 무조건 찬스볼"이 아니라, **누가 살리느냐 + 그 선수의
  세팅 능력**으로 다음 공격 질이 결정됨.
- 찬스볼 공방이 반복되면 양 팀 모두 공격 효율 낮아져 랠리가 길어짐(7장 연계).

## 7. 랠리 시스템 (네 요청 — 같이 설계)

여자부는 랠리가 길다. 랠리 길이를 체력·멘탈에 연동해 "긴 랠리의 무게"를 만든다.

### 7.1 체력 (스탯 분리: 체력 + 체젠)

체력 관련 밑단 스탯을 둘로 나눈다(5장 신체 스탯 반영).

- **체력 (Stamina, max)**: 보유 상한. `tempStamina`의 시작값·천장.
- **체젠 (Stamina Regen)**: 회복 속도. 쉬는 구간에 얼마나 빨리 차오르는지.

작동:

- 랠리 1회의 공방(hop)마다 참여 선수의 `tempStamina`를 소량 차감.
- `tempStamina`가 떨어지면 점프력·반응속도·정확도에 일시 페널티 → 공격·디그 효율↓.
- **회복은 자연 회복**: 랠리 사이·타임아웃·세트 간 등 쉬는 구간에 **체젠만큼** 차오름.
  타임아웃은 회복을 따로 주는 게 아니라 "쉬는 시간을 버는" 것(7.4).
- 누적: 긴 랠리가 연속되면 회복이 못 따라가 후반 효율 저하 → 5세트 체력전.

```
매 hop:   tempStamina -= cost
쉬는구간:  tempStamina = min(체력max, tempStamina + 체젠 × 휴식길이)
효율보정:  effFactor = 0.7 + 0.3 × (tempStamina / 체력max)
```

> 선수 유형 분화: 체력 max 높고 체젠 느린 선수는 후반에 무너지고, 체력 보통이지만
> 체젠 빠른 선수는 랠리 사이 금방 회복해 5세트 체력전에 강함.

**검증·재튜닝 (2026-06-12)** — `tools/simStatEffect.ts`가 "체력·체젠 무효"를 검출
(회복 +0.05/랠리가 소모 −0.02/공격을 상쇄 → 잔량이 안 깎임). 경제 재설계:

- 소모: 서브 1.0 / **공격 hop당 1.2(점프가 가장 힘들다)** / 블로커 0.4 / 리시브 라인 0.2 /
  디그 시도 0.4 / **토스 0.3(세터도 사람 — 빈도 최다·강도 낮음)** × `HOP_COST 0.024 / (0.6+0.8×체력)`
- 회복: 랠리 사이 ~~`0.009`~~ **`0.005`**×(0.4+체젠) · 세트 사이 ~~`0.12`~~ **`0.035`**×(0.4+체젠) — 완전 리셋 없음(누적 피로) ·
  타임아웃 `0.04` 양 팀 공통(7.4). ⟨ENGINE_VERSION 2, 2026-06-28 체력 재튜닝 — 후반 체력 하락 강화⟩
- 효율: `eff = 0.70 + 0.30×잔량` (서브·공격·블록·세트·리시브·디그 강도에 곱)

검증치 (STATS_PROTOCOL 형식 — N · 엔진 커밋 · 날짜):

- 단독 효과(동일 전력 80vs45, 진영 교대): 체력만 **54.3%** · 체젠만 **53.8%** ·

### 7.2 기세 (모멘텀) — 팀 단위 스탯

7.1의 임시 멘탈을 **기세(모멘텀)** 로 정식화한다. 선수 개인 스탯이 아니라
**팀 단위 임시 수치**다(경기 중에만 존재).

- 범위 0~100, 경기·세트 시작 시 50.
- **연속 득점(런)**: 득점한 팀 기세 ↑, 실점한 팀 기세 ↓. 런이 길수록 변동 가속.
- **기세 높음** → 그 팀 **전체 스탯에 버프**(공격·서브·디그·리시브·블로킹 전반에
  곱해지는 승수).
- **기세 낮음** → 전체 스탯 디버프 + 서브 범실·공격 범실·포지션 폴트 확률 동반
  상승(악순환).
- 베테랑(consistency 높은 선수)이 코트에 있으면 하락폭 완충.

```
득점 시:  momentum[winner] += base × runFactor
          momentum[loser]  -= base × runFactor
런 가속:  runFactor = 1 + 0.15 × (연속 득점 수)
효율보정:  momFactor = 0.96 + 0.0008 × momentum   // 0.96~1.04 (±4%) — engine/rally.ts:23
```
> ~~`teamFactor = 0.85 + 0.30 × (momentum/100)` (±15%)~~ → **±4%로 재캘리브레이션**(2026-06, KOVO 정렬 — 구현현황
> 표·코드와 일치). momFactor는 svPow·recvSkill·attackPower·blockEval·digStr에 곱해진다. **직접 범실보정 항(`+= (1-momentum/100)×k`)은
> 미구현** — momentum은 momFactor 승수로만 간접 영향(범실은 teamVQ 기반, §1.4).

> `tempMental`이라는 선수별 임시치 대신 **팀 momentum 하나**로 통합한다.
> (개인 focus·consistency·VQ는 5장 밑단 스탯으로 유지되며, 기세 변동폭·완충에 관여.)

- 랠리 hop 상한(**8** — `engine/rally.ts CAP`, §7.3)으로 무한 루프 방지. 상한 도달 시 양 팀 효율 급감으로 마무리.

### 7.4 타임아웃

타임아웃은 **양 팀이 함께 쉬는 공통 시간**이다. 부른 쪽만 이득 보지 않는다.

- **기세 보정 (양 팀 공통)**: 양 팀 기세를 각각 50 쪽으로 수렴시킨다.
  - 기세 낮은 팀(보통 부른 쪽) → 끌어올림
  - 기세 높은 팀(보통 상대) → 끌어내림
  - 완전 리셋(50 고정)이 아니라 50 방향으로 당기는 것.
  - **함의**: 자기 팀 기세가 좋을 때 부르면 자기 기세도 깎인다 → 함부로 못 부름.
    지고 있을 때(기세 낮을 때) 부르는 게 이득 → 실제 배구 타임아웃 심리와 일치.
- **수렴 폭 = 부른 팀 감독의 카리스마**에 비례(8장). 카리스마 높은 감독이 부르면
  강하게 수렴, 낮으면 미미.
- **체력**: 타임아웃이 체력을 따로 주지 않는다. 멈춰 있는 동안 **자연 회복**(체젠만큼
  차오름). 즉 타임아웃의 체력 효과 = "쉬는 시간 확보".
  - 구현(2026-06-12): `recover(±, TIMEOUT_REST=0.04)` 양 팀 공통 — 랠리 사이(~~0.009~~ **0.005**, v2)의 ~8배 휴식.
- **지친 조기 호출(2026-06-12)**: 코트에 체력 50% 미만으로 퍼진 선수가 있으면 감독이 임계를
  1 낮춰 한 박자 일찍 부른다(최저 2). 평균이 아니라 **최저** 기준 — 주포가 퍼진 걸 세터·리베로의
  쌩쌩함이 가리면 안 된다. 발화 빈도: 일반 리그 타임아웃의 0.5%(드묾), 전원 저체력 리그 36%.

```
onTimeout(caller):
  pull = chARISMA(caller.coach) / 100 × maxPull          // 수렴 강도
  momentum[home] += (50 - momentum[home]) × pull
  momentum[away] += (50 - momentum[away]) × pull
  // 체력은 별도 처리 없음 — 휴식 구간 자연 회복 로직이 적용됨
```

- 권한(8장, 2026-06-16 정정): 타임아웃은 **감독 고유 권한** — 평소·빅매치 모두 AI 감독이 성향대로
  자동 호출한다. 구단주는 호출도 건의도 하지 않는다(CLAUDE.md 3장: 코트 옆에서 타임아웃을 부르는 건
  권한 분리를 흐림). 마음에 안 들면 → 감독 교체.
- 세트당 보유 횟수 제한(`timeoutsLeft`).
- **관전 모달(2026-06-19 구현)**: 감독이 타임아웃을 부르면 보드가 멈추고 **코트 선수 체력**을 보여준다
  (관전형 — 구단주가 벤치 분위기를 "보는" 장면). 엔진이 호출 순간을 `timeouts`(순수 가산 로그, 승패
  무영향 — `subEvents`와 동일 패턴)로 기록: `point`(랠리 인덱스)·`side`(부른 팀)·스코어·연속실점·
  **양 팀 코트 체력 스냅샷(회복 전)**·기세. 보드는 그 랠리 종료 후 한 번 멈추고 모달을 띄운다("경기
  진행하기"로 재개, 본 타임아웃은 재진입 시 재팝업 안 함). **미래 확장 진입점**: 여기서 작전 교체·기세
  확인. 현재는 체력만 표시. 검증: 경기당 ~7.8회 발생, simKovo 분포 불변(가산 = 승패 불변).

#### 7.4b 테크니컬 타임아웃 (KOVO, 2026-07-07 구현)

KOVO는 FIVB가 2021 폐지한 **테크니컬 타임아웃(TTO)** 을 여전히 운영한다(1~4세트, 5세트는 8점 코트체인지만).
과거 문서의 "KOVO도 폐지 추세"는 FIVB 기사에서 추론한 오류였다 → KOVO_RULES_COMPARISON §1·§4 정정.

- **트리거(고정 점수, rng 미소비)**: 1~4세트(`setNo ≤ 4`)에서 **리드팀 점수 = `max(h,a)`** 가 **8**에 처음 도달, 다시 **16**에 처음 도달할 때 각 1회(세트당 최대 2회). 세트가 이미 끝났으면 미발생. 점수는 1점씩 오르므로 "처음 도달" = 정확히 그 임계 점수 순간(fired 플래그로 세트당 1회 보장).
- **효과 = 코치 타임아웃과 유사(회복폭만 다름)**: 양 팀 `recover(TTO_REST)` + 기세를 50 쪽으로 수렴(`TTO_PULL=0.3` 고정 — 코치 타임아웃의 `charisma/100×0.6`에서 카리스마 50 상당의 중립값. TTO는 감독이 부른 게 아니라 공식 자동 휴식이라 카리스마 무관) + `streak`/`lastScorer` 리셋.
  - **회복폭 재튜닝 (2026-07-07, ENGINE_VERSION 8) — `TTO_REST = 0.03`**: TTO는 세트당 2회(8·16점) 자동 발화라 코치 타임아웃 `TIMEOUT_REST=0.04`를 그대로 쓰면 세트 내 회복이 과다해진다(피로 곡선 붕괴 — TTO 도입 직후 `simStamCurve` 5세트 코트 평균 83.1%로 밴드 상한 82% 초과). **의도적으로 60초/30초 현실 회복비를 깨고** TTO만 낮춘다: 세트 내 총 휴식을 낮춰 **정밀 튜닝한 피로 곡선(설계 기둥)** 을 지킨다. 코치 `TIMEOUT_REST`는 0.04 유지(감독 타임아웃은 세트당 최대 2회·조건부라 영향 작음).
    - **튜닝 스윕 (N=3000, 엔진 v8, 2026-07-07 · `simStamCurve`)**: TTO_REST를 낮추면 5세트 체력↓·피로 교체(1.3e)↑, 올리면 반대 — **두 밴드(체력 60~82·피로율 0.05~0.5)를 동시에 만족하는 유일한 값이 0.03**. `0.02`→ 5세트 79.9% PASS·피로율 0.634 FAIL(과다). `0.03`→ 5세트 **81.5%** PASS·피로율 **0.453** PASS. `0.035`→ 5세트 82.1% FAIL(초과). `0.04`→ 82.5% FAIL. 채택: **0.03**. (TTO_REST만으로 못 들었으면 2차 노브 `STAM_REGEN_BASE`였으나 불필요.)
- **팀 타임아웃 잔여 미차감**: `timeouts[side]` 건드리지 않음 — TTO는 감독 타임아웃과 **별개의 자동 휴식**(KOVO).
- **결정론**: 점수만 보는 트리거라 rng 0소비. 리드팀을 판별자 `side`로 실어 보드가 표시(라벨 "⏱ 테크니컬 타임아웃").
- **보드**(`components/MatchCourt.tsx`): `TimeoutEvent.technical` 판별자로 제목/부제 분기. 기존 코치 타임아웃 모달(코트 체력)과 같은 UI, 라벨만 다름.
- **5세트**: TTO 없음(코트체인지는 코트 추상화라 시뮬 무영향 — 의도적 생략).
- 검증: 경기당 세트별 ≤2회·정확히 8/16 도달점에서 발생·5세트 0회·팀 타임아웃 예산 불변·판별자 존재(`tools/_dv_*` + `match.test.ts`).

## 8. 감독 시스템

감독은 **플레이어가 선임**하는 별개 존재다(플레이어 ≠ 감독). 평소 경기는 감독이
자동으로 운영하고, 플레이어는 중요 경기에서만 직접 개입한다(선택적 개입).

> 범위 한정: **경기 운영에 직접 쓰이는 능력만** 정의한다. 선수 훈련·성장 보정 등은
> 이 게임 단계에서 제외(나중에 다룸).

### 8.1 감독 스탯

| 스탯 | 용도 |
|---|---|
| **카리스마** | 타임아웃 시 기세를 50으로 당기는 수렴 폭(7.4). 높을수록 흐름 장악 |
| **성향** | 자동 운영 스타일. `공격형 / 수비형 / 밸런스` |

### 8.2 성향에 따른 자동 운영

- **공격형**: 공격적 서브·속공/시간차 비중↑, 공격적 블로킹 선호. 타임아웃을 아끼고
  밀어붙임(위기에도 늦게 호출).
- **수비형**: 안정 서브·소프트 블로킹·디그 중심. 상대 런이 짧아도 타임아웃 빨리 호출.
- **밸런스**: 중간. 무난한 분포.
- 성향은 라인업·작전 방침·타임아웃 타이밍의 자동 결정에 반영된다.

### 8.3 권한 분리 (타임아웃·작전)

| 상황 | 타임아웃/작전 주체 |
|---|---|
| 평소 경기 (방치형) | **AI 감독** (성향대로 자동) |
| 중요 경기 + 플레이어 개입 | **플레이어** (직접 호출·작전) |

- 플레이어가 개입해도 카리스마는 **선임된 감독의 값**을 사용(타임아웃 수렴 폭).
  즉 좋은 감독을 뽑아두면 내가 직접 부르는 타임아웃도 더 강하게 작동.

> 경고: 위를 다 넣으면 방치형(1순위)과 충돌. **1~2번은 엔진 핵심으로, 3~6번은
> Phase 단위로 점진 도입**, 7~8번은 보류. 매 항목이 "손이 더 가게" 만들지 않는지
> 항상 검사.

## 11. 밸런싱 체크리스트 (Phase 0에서 검증)

- [ ] 한 세트 평균 랠리 길이가 여자부답게 약간 길게 분포하는가
- [ ] 공격 성공률, 서브 에이스/범실 비율이 현실 범위인가
- [ ] 포지션 폴트가 "드물지만 시즌에 몇 번"으로 나오는가(평균 VQ 낮은 팀일수록 ↑)
- [ ] 긴 랠리 후 효율 저하가 체감되는가(체력전, 체력 max vs 체젠 차이가 드러나는가)
- [ ] 기세(momentum)가 연속 득점에 자연스럽게 오르내리고, 버프/디버프가 과하지 않은가
- [ ] 타임아웃이 흐름을 끊되, 좋은 기세일 때 부르면 손해가 되는가(카리스마 반영)
- [ ] 커밋/리드 블록 차이로 싱크로·속공의 효과가 드러나는가
- [ ] 같은 시드로 항상 동일 결과가 재현되는가(결정론 검증)

## 12. 패스 품질(q) 측정 기록 & 추후 수정사항

> 측정 도구: `tools/simQuality.ts` (`npx tsx tools/simQuality.ts 10000`).
> q 계측은 `RallyStats` 싱크(텔레메트리 전용 — 결과 불변). 통계 프로토콜 준수.

### 12.1 측정값 (N=10,000경기 · 랠리 1,741,782 · 엔진 dd5559b · 2026-06-16)

**서브 리시브 품질** (인플레이 리시브만 — 에이스·서브범실·리시브범실 제외, 표본 1,435,089):
- 평균 q **0.505** · 분포: 좋음(≥0.60) 21.0% / 보통(0.45~) 45.4% / 난조(0.32~) 29.2% / 찬스볼(<0.32) **4.3%**
- 리시브 범실(셰이크) 3.5%(전체 리시브 시도 대비)

**랠리 중 전환 품질** (종류별 분리, 표본 1,317,144):
- 일반 디그 **0.527**(83.1%) / 팁 디그 0.634(10.3%) / 소프트블록 0.700(6.6%) / 가중평균 0.550

**해석:**
- 평균 리시브 q가 기저(0.58)보다 낮은 0.505 — 서버들의 공격적 서브(`SERVE_DIFF`)가 리시브를 의미 있게 깎음(서브가 밥값을 함).
- "찬스볼"(완전 속공 잠금) 기록은 4.3%로 드물지만, 빠른 공격 가용은 q 0.44부터 서서히 풀려 **기능상 속공 불가는 ≈33%**(q<0.44). 둘은 다른 수치 — 혼동 금지.

### 12.2 추후 수정사항 (낮은 우선순위)

- **[관찰됨] 일반 디그 q(0.527) > 서브 리시브 q(0.505)** — 현실은 계획된 리시브가 스크램블 디그보다
  품질이 살짝 높아야 자연스러운데 **미세하게 뒤집힘(격차 0.022 ≈ 4%)**. 원인: 리시브 기저(0.58)가
  디그 기저(0.40)보다 높지만, 서브 난이도 페널티가 그 우위를 상쇄하고도 더 끌어내림.
  - **현재 미수정.** 격차가 노이즈 수준(4%)이고, 디그 기저·서브 페널티를 손대면 KOVO 박스스코어
    밸런싱(20지표) 전체를 재검증해야 하므로 비용 > 이득.
  - 손댈 때: 일반 디그 기저(0.40, `rally.ts`) 소폭 하향 또는 `SERVE_DIFF` 완화 후 `simQuality`+`simKovo`
    동시 재측정. 단독 변경 금지(킬·사이드아웃 동반 이동).
