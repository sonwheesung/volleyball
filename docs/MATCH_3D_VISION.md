## 0. 왜 3D인가 — 기둥 정합

- CLAUDE.md 기둥 1 **관전형 장기 시뮬**: "조작하는 게임이 아니라 **보는 게임**. 경기 보드의 연출 품질(실제 배구처럼 보이기)은
  부차 요소가 아니라 **1순위 투자처**." → 3D 경기화면은 이 기둥의 정면 강화다(손이 더 가게 만드는 게 아니라 **더 보고 싶게**).
- 기둥 2 **데이터 누적 서사**: 3D는 서사를 **바꾸지 않는다** — 같은 결정론 결과를 더 실감나게 **그리기만** 한다.
- 원칙 정합(11장): "UI는 엔진 상태를 그리기만 한다. 게임 로직을 UI에 두지 않는다." → 3D는 **렌더 계층 교체**이지 엔진 변경이 아니다.
  결정론·무저장 리플레이 척추는 렌더러와 **완전 분리**되어 안전(§4).

### 스코프 경계 (지금 하지 않는 이유)

- **Expo Go 이탈**: 3D 렌더(`expo-gl`+three)는 커스텀 dev client(prebuild)를 요구 → CLAUDE "Expo Go 항상 실행" 원칙과 충돌.
  별도 빌드 트랙(dev client/AAB)에서만 가능 → **출시 크리티컬패스(#43 결제·스토어 등록 등) 위에 올리지 않는다.**
- **대형 과제**: 애셋 제작(모델·모션)·모바일 성능 튜닝·이벤트 배선까지 수 개월 규모(§6). 출시 후 착수 대상.
- 이 문서의 즉시 실행 부분은 **오직 하나** — [모캡 테스트](./MOCAP_TEST_GUIDE.md)로 "우리가 실제 배구 모션을 뽑을 수 있는가"의 **실현성만 반나절 검증**.
  코드 무접촉·비상업·소스 교체 전제(§5).

## 1. 조사 요약 (2025~2026, 변동 가능)

> 전체 표·출처는 스크래치패드 `3d-pipeline-research.md`. 여기엔 결론만.

### 1.1 3D 모델 생성 AI — 3강

| 도구 | 강점 | 리깅/모션 | 상업 라이선스 | 우리 용도 |
|---|---|---|---|---|
| **Meshy** (v4/v5) | 캐릭터+오토리그+프리셋 모션 **원스톱**, 공식 API+MCP | 오토리그 <30초, 600+ 프리셋(크레딧 미소모) | 유료 플랜=에셋 소유(상업 OK) | 선수 베이스 모델 대량 생성 1순위 |
| **Tripo** (P1/Studio) | 속도·게임레디 토폴로지, Auto Rig, 공식 API+MCP | Auto Rig(스켈레톤 포함) | Pro+ 풀 상업권 | 대량 배치 생성 양강 |
| **Rodin / Hyper3D** (Gen-2.5) | 지오메트리 품질 1위(쿼드 메시) | 리깅 별도(Mixamo/AccuRIG) | 전 플랜 풀 상업권 | 대표 모델 고품질 필요 시 |

### 1.2 MCP (에셋 제작 반자동화 — 런타임 아님)

- **존재 확인**: Blender MCP(ahujasid, ~22k★), 공식 **Meshy MCP**(리깅/애니 포함), 공식 **Tripo MCP**(Blender 임포트), mcp-blender(RFingAdam, 218툴 멀티백엔드).
- 성격: AI가 Blender·생성API를 **제어**하는 개발 편의 계층. **런타임 게임엔진 아님** — 앱엔 산출물 GLB만 들어감. 필수 아님(웹 UI로도 가능).

### 1.3 RN/Expo 3D 렌더링 — 조건부 실현 가능

- **표준 경로 존재**: `expo-gl` + `three` + `@react-three/fiber` + `@react-three/drei/native`(`useGLTF`·`useAnimations`)로 GLB 스켈레탈 애니 재생.
- **제약**: **Expo Go 사실상 불가** → dev client(prebuild) 필요. iOS 시뮬 OpenGL ES 불안정 → 실기기 테스트 필수.
- **성능 리스크**: 6~12인 동시 스켈레탈 애니 + 코트는 모바일 GPU 부담 → 로우폴리(선수당 ≤1.5만 tri)·본 ≤40·인스턴싱·LOD·카메라 앵글 제한 필수.

### 1.4 2.5D 완충안

- 풀 3D 대신 **제한 카메라 3D** 또는 **2.5D(Spine/Rive 스켈레탈 2D)**로 시작 → 성능·용량 리스크 대폭 감소.
- 참고: 상용 스포츠 게임도 상당수 **정해진 카메라 앵글 + 소수 캐릭터**로 임팩트를 냄. 풀 3D가 목표 감성의 유일 경로가 아니다.

## 2. 엔진 이벤트 로그 → 애니 대본 매핑 (3D 준비도 진단)

> 실코드 대조(2026-07-23): `engine/events.ts`·`engine/court.ts`·`engine/spatial.ts`·`engine/simMatch.ts`·`data/matchBox.ts`.
> 핵심 결론: **3D는 시뮬 로직 재작성이 아니라, 이미 존재하는 opt-in 텔레메트리 채널을 소비하는 렌더러 교체 + 결정론 합성이다.**

### 2.1 이미 있음 — "애니 대본"이 사실상 갖춰져 있음

`engine/events.ts` `RallyEvent[]`는 랠리별 공간 텔레메트리로 **이미 산출된다**(별도 srng → 승패 불변, 결정론 보존):

- **액션 분류 7종**: `serve` · `receive` · `set`(토스) · `attack` · `block` · `dig` · `point`.
- **2D 코트 좌표**: 각 이벤트에 `from`/`target`/`landing`/`course`/`at`/`ball`(`Pt = {x,y}`, **미터 단위**). 코트계 `engine/court.ts` = 9m(폭)×18m(길이), 네트 y=9, 존→XY 매핑(`zoneXY`/`playerXY`).
- **동작 세부**: `serveType`(safe/float/jumpfloat/spike), `atk`(quick/tempo/open/back) + `quickKind`(A/B퀵/슬라이드), 결과(`kill`/`blocked`/`dug`/`error`/`blockout`/`softblock`, 리시브 good/poor/shank/ace).
- **스탯 근거**: 이벤트별 `rating`(그날 진화·폼 반영 종합 스탯) · `eff`(체력·부상 실효 배수) → "어떤 선수가 어떤 상태로 그 동작을 했는지"까지 실림.
- **좌표 파생 수학**: `engine/spatial.ts`(서브 낙하 `serveLanding` · 토스 낙하 `tossLanding` · 공격 코스 `attackCourse`) — 추상 결과 + 좌표 → 구체 낙하점을 이미 계산.
- **부가 이벤트**: `SubEvent`(교체 시점·in/out·사유) · `TimeoutEvent`(타임아웃 시점·체력/기세 스냅샷). 3D에서 교체 워크·타임아웃 컷의 트리거로 재사용 가능.

→ 즉 **액션 종류 + 시전자/대상 + 코트 좌표 + 결과 + 순서**가 결정론적으로 이미 나온다. 이것이 3D 애니 타임라인의 "대본"이며,
보고서 §5.7의 "결정론 로그가 그대로 애니 대본" 주장은 **실코드로 실증됨**.

### 2.2 배선 필요 (로직 변경 아님 — opt-in 채널 연결)

- `events`는 **opt-in**이다: `simulateMatch(..., { events })`로 배열을 넘길 때만 채워진다(`engine/match.ts` L69).
- **현재 소비자는 검증 도구/테스트뿐**(`tools/simActionTrace.ts`·`tools/simMoments.ts`·`engine/events.test.ts`). 실제 보드 경로
  `data/matchBox.ts`(`buildMatchBox`→`simulateMatch`)는 `events`를 **요청하지 않고** `box`/`boxTimeline`(누적 스탯)·`touches`(디그 마커)만 그린다.
- → 3D 렌더러는 이 배열을 **opt-in으로 받기만 하면 된다**(새 엔진 로직 0). 결정론 프리픽스·바이트 동일성도 안전 — `events`는 별도 srng라 넘겨도 승패 불변(테스트 `events.test.ts`가 seed 42 이중 실행 동일성 검증).

### 2.3 합성 필요 (결정론 파생 — 엔진 아닌 렌더러 작업)

로그가 대본이긴 하나, 3D로 그리려면 **아래 4가지는 렌더러가 결정론적으로 합성**해야 한다(로그엔 없음):

1. **Z축(높이)** — 좌표가 전부 **지면 2D**(x,y). 스파이크 정점·서브 궤적·점프 높이·공의 포물선은 없음 → `landing`+액션종류에서 **파라볼라 합성**(결정론).
2. **타이밍/지속** — 이벤트는 **순서만** 있고 타임스탬프·duration 없음 → 클립 길이·페이싱을 렌더러가 결정론 합성(상수 또는 rating 파생).
3. **연속 이동** — `playerXY`는 **존 중심 스냅샷**만, 점 사이 연속 이동 경로 없음 → 보간/트윈은 렌더러.
4. **모델 바인딩** — `player.id` → 3D 애셋(모델+스켈레톤) 매핑이 아직 없음 → 애셋 레지스트리 신설 필요.

### 2.4 준비도 판정

| 항목 | 상태 |
|---|---|
| 액션 분류·시전자·결과 | ✅ 있음(`RallyEvent`) |
| 2D 코트 좌표(미터) | ✅ 있음(`court.ts`/`spatial.ts`) |
| 결정론·승패 불변 | ✅ 있음(별도 srng, 테스트 검증) |
| 교체·타임아웃 트리거 | ✅ 있음(`SubEvent`/`TimeoutEvent`) |
| 보드↔이벤트 배선 | ⚠ opt-in만, 현재 도구 전용(연결 필요) |
| Z(높이)·타이밍·연속이동·모델바인딩 | ❌ 렌더러가 결정론 합성 |

**종합**: 엔진은 애니 대본으로서 **대부분 준비됨**. 3D 과제의 본질은 "새 시뮬"이 아니라 **(a) opt-in 텔레메트리 배선 + (b) 높이·타이밍·이동·애셋의 결정론 합성 + (c) 렌더러 구현**. 시뮬 로직은 손대지 않는다.

## 3. 모션 소스 정책 (저작권·초상권)

- **테스트 단계**: 유튜브의 1인 스파이크/드릴/슬로우모션 영상으로 **비상업 기술 테스트**만 허용(실현성 판단용). 앱 미배포·소스 교체 전제.
- **프로덕션 단계**: 모션 소스를 **직접 촬영** 또는 **구매/라이선스 애셋(모션팩)**으로 **교체**한다.
  - 근거: (1) 유튜브 영상의 저작권·실연자 초상권을 상업 배포물에 실을 수 없음. (2) 프로젝트의 **가상 리그 전략**(실명·실팀 회피)과 정합 — 실제 선수 모션을 그대로 쓰면 그 회피가 무의미해짐.
  - 무료 모캡 툴의 **무료 티어가 대개 "개인·비상업"**(예: DeepMotion 프리티어)이라, 상업화 시엔 유료/상업 라이선스 전환도 함께 필요.
- **원칙**: 테스트에서 확인하는 것은 "**툴이 배구 모션을 잘 뽑는가**"라는 **파이프라인 실현성**이지, 그 특정 클립의 사용권이 아니다. 좋으면 → 깨끗한 소스로 다시 뽑는다.

## 4. 결정론 격리 (안전성)

- 3D 렌더러는 **엔진 출력을 소비만** 한다(UI→data→engine 의존 방향 유지, 역방향 없음).
- `events` 텔레메트리는 별도 srng → **메인 결과 RNG 불간섭**(프리픽스 불변·바이트 동일성 보존).
- 무저장 리플레이 척추(base 스냅샷+currentDay+results)는 렌더러와 무관 — 3D가 붙어도 세이브 포맷 무변경.
- 높이·타이밍·이동 합성은 **시드 파생/독립 해시**로(Math.random·Date.now 금지) → 같은 경기 = 같은 3D 재생(리플레이·검증 가능).

## 5. 권고 파이프라인 (출시 후)

```
[1] 선수 베이스 모델 N종(5~10) 생성   → Meshy(원스톱) 또는 Tripo(대량). 포지션·체형 몇 종 + 유니폼/텍스처 변주
[2] 공통 스켈레톤 리깅                → Mixamo 규격으로 통일(리타게팅 열쇠). 필요 시 AccuRIG 2(무료)
[3] 배구 모션 M종(20~40) 확보
      · 일반(idle/run/jump/celebrate)  → Mixamo(무료)에서 바로
      · 배구 전용(스파이크·서브·리시브·디그·블로킹·토스) → 비디오 모캡(§MOCAP_TEST_GUIDE) → Blender 클린업
[4] 리타게팅(1 모션 → N 선수)         → Mixamo 자동 리타깃 또는 Blender Auto-Rig Pro/Rokoko
[5] 최적화 + GLB 익스포트             → Blender(+Blender MCP 반자동) 데시메이트·본 정리·텍스처 아틀라스
[6] 앱 번들 + 렌더                    → expo-gl + r3f + drei/native, useGLTF/useAnimations
[7] 엔진 이벤트 → 애니 트리거 매핑     → §2. RallyEvent(액션+좌표) → 클립 + 위치 트윈 + 높이/타이밍 합성
```

| [7] 이벤트 배선 | 무료 | 수 주 | 로직 아닌 배선+합성이나 높이/타이밍/이동 튜닝 공수 |

- **최대 리스크 3**: (1) Expo Go 이탈 → 빌드/배포 트랙 이원화, (2) 다중 캐릭터 모바일 성능, (3) 배구 전용 모션 확보 난이도.
- **완충 전략**: 풀 3D 이전에 **제한 카메라 3D 또는 2.5D(Spine/Rive)**로 "관전 임팩트"를 먼저 검증 → 성능·감성 확인 후 풀 3D 확장.
- **파일럿 감**: 선수 3종 + 핵심 모션 10종 + r3f 재생 PoC = 수 주. 프로덕션 품질(클린업·최적화·이벤트 매핑·성능) = 수 개월 별도 과제.
