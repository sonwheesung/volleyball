# DOC_DISCIPLINE — 문서 작업법

> 설계 문서·케이스 레지스트리·방법론 문서를 어떻게 쓰고 유지하느냐의 규율. 이걸 지키면 "코드와
> 문서가 따로 노는" drift와 "옛날 수치가 조용히 거짓이 되는" 사고를 막는다.
> 이미 실천 중인 것들(취소선 정정·`*_SYSTEM.md`·README 구현현황표)을 한곳에 명문화한다.

---

## 1. 결정은 코드보다 먼저 문서에

> **표준 작업 순서(정본)는 `CLAUDE.md` 11장 "표준 작업 순서"** — ① 플랜(상충 결정 확인+분해) →
> ② 문서 작성(코드보다 先) → ③ 개발 → ④ 오류 확인(N≥1만·A/B, 통과 후 커밋). 본 §1은 그중 ②의 규율.

- 새 설계 결정은 **코드보다 먼저** 문서에 반영한다 — 루트 `CLAUDE.md`(원칙·기둥) 또는 해당
  `docs/<SYS>_SYSTEM.md`(시스템 상세) + `docs/README.md` 색인. (CLAUDE.md 11장 "새 설계 결정은 코드보다 먼저".)
- "왜 이렇게 정했나"를 남긴다. 특히 **무엇을 안 하기로 했나(제외 결정)** 와 그 이유를 적는다 —
  나중에 같은 논의가 반복되는 걸 막는다.
  > 예: 트레이드 = "방치형과 결이 약하고 AI 거래 밸런스 난해"로 제외(7장). 타임아웃 건의 = "타임아웃은
  > 감독 고유 권한"이라 제거(3장). 이런 결정을 이유와 함께 문서에 박는다.

## 2. 시스템마다 `*_SYSTEM.md`

- 각 독립 시스템은 자기 설계 문서 `docs/<SYS>_SYSTEM.md`를 가진다(단일 책임 = 단일 문서).
- 문서 상단/README에 **"★ 구현 현황" 표**를 둬서 설계 vs 실제 코드 간극을 명시한다(✅/❌/🚫/보류).
- **일관성 검증은 이 구현 현황 표를 기준으로 한다**(README 상단 명시와 동일).

## 3. README 색인 (docs/README.md)

`docs/README.md`가 색인이며 다음을 유지한다:

| 항목 | 내용 |
|---|---|
| 문서 목록 표 | `문서 │ 범위 │ 핵심 엔진 파일` — 각 `*_SYSTEM.md` 링크 |
| 전체 구현 현황 요약 | 시스템별 ✅/❌ 한눈에 |
| 검증 루틴 | 타입체크·테스트·감사·퍼징 명령을 복붙 가능하게 |
| 아키텍처 원칙 | 의존 방향·결정론·계약 의존(CLAUDE.md 11장 요약) |

- 새 `*_SYSTEM.md`를 추가하면 **반드시** 이 색인 목록·구현현황 요약에도 추가한다(§부록 체크리스트).

## 4. 날짜·통계는 절대화(absolute)

- 결정·변경에는 날짜를 명시한다: `(2026-06-18 정정)` `(구현)` 등 — "최근/얼마 전" 상대 표현 금지.
- 통계 수치에는 `(N · 엔진 커밋 · 날짜)` 메타데이터 필수([STATS_PROTOCOL](./STATS_PROTOCOL.md)). 엔진 변경 시 갱신/무효.
- 정정 이력을 지우지 말고 **취소선/날짜 병기**로 남긴다 — "예전엔 이렇게 생각했다가 이래서 바꿨다"가
  보여야 같은 실수를 안 반복한다.
  > 예: CLAUDE.md의 `~~방치형~~ → 관전형(2026-06-11 정정)`, `~~세트 중 작전 토글~~ 제거(2026-06-18)`.

## 5. 케이스(무엇)와 방법(어떻게)을 분리

- **케이스 레지스트리**([EDGE_CASES](./EDGE_CASES.md)·[BOARD_RULES](./BOARD_RULES.md)): *어떤 버그가 있었나·무엇을 검사하나* — 증상→원인→수정→도구.
- **방법론**([TEST_METHODOLOGY](./TEST_METHODOLOGY.md)): *어떻게 새 버그를 찾나* — 발견 기법·후속 5단계·사각 분류·5렌즈.
- 둘을 섞지 않는다. 버그를 찾으면 케이스는 레지스트리에, 발견 기법은 방법론에 따로 기입한다([[bug-discovery-protocol]]).

## 6. 도메인 정의는 출처로 (추정 금지)

- 현실/배구·KOVO 용어·규칙·표준은 구현 전 **출처(공식/문헌/웹)로 확인**한다. 기억으로 단정 금지([[verify-domain-definitions]]).
- 확인한 정의는 코드 주석·docs에 **출처와 함께** 남긴다.
  > 예: 트리플 크라운(서브에이스·블로킹·후위공격 각 3+)을 확인 없이 잘못 구현했다가 교정 → KOVO 출처 명기.

## 7. 커밋 메시지 형식

- `YYMMDD :: <한국어 작업 요약>` ([[commit-message-format]]). 일관 유지.
- 멀티라인 메시지는 heredoc/임시파일 + `git commit -F`로 — 셸 따옴표/here-string 오염 회피.
- 문서 갱신을 코드 커밋과 **함께**(또는 먼저) 한다.

---

## 8. 큰 영역은 인덱스 + 세부 문서로 쪼갠다

한 영역(예: 테스트 전략)이 커지면 **한 문서에 다 넣지 않는다** — 비대해져 예시·절차가 누락된다.

- **인덱스 문서 하나**(`docs/README.md`): 종류 전수 목록 + 한 방 실행/검증 루틴 + "무엇을 만지면 무엇을 도나" 매핑.
- **종류별 세부 문서**(`*_SYSTEM.md`·`EDGE_CASES`·`TEST_METHODOLOGY`): 각 종류의 *예시·읽는 법·절차*.
- 인덱스는 **빠짐없는 목록**이 핵심(빠진 종류가 보이게), 세부는 깊이가 핵심.
> 다른 세션이 인덱스만 읽고도 전체를 돌릴 수 있어야 한다.

## 부록 — 새 시스템 추가 시 문서 체크리스트

```
[ ] docs/<SYS>_SYSTEM.md 작성 (설계 결정·구현 현황 표)
[ ] docs/README.md 색인 목록 + 구현현황 요약에 추가
[ ] 케이스 레지스트리(analyze-cases)로 정상/엣지 케이스 문서화
[ ] 검증 루틴(README)에 새 감사/측정 도구 명령 추가
[ ] 설계가 CLAUDE.md 기둥/원칙에 영향 주면 CLAUDE.md도 갱신
[ ] 통계 기입 시 (N · 엔진커밋 · 날짜) 메타데이터
[ ] 도메인 용어 쓰면 출처 확인 + 주석에 출처 명기
```
