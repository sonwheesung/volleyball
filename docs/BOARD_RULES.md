# BOARD_RULES — 관전 연출 검증 기준 (구단주의 눈)

> 사용자가 폰으로 경기를 **보다가** 직접 잡아낸 어색한 장면들을 검증 가능한 기준으로 정식화한 문서.
> 관전형 게임에서 경기 연출은 1순위 투자처(CLAUDE.md 기둥 1)이므로, 이 기준들은 전부
> `tools/auditBoard.ts`의 **상설 룰**로 자동 검사된다. 새 주의사항이 나오면:
> **① 이 문서에 행 추가 → ② 연출 수정 → ③ 감사 룰 추가(회귀 방지) → ④ 풀배터리 0건** 순서로 처리한다.
>
> 검증 실행: `/verify-board` 스킬 또는 `npx tsx tools/auditBoard.ts [경기수]`
>
> **수비/리시브 포지셔닝의 "왜"** (역할→위치 자동화 모델)는 [COURT_POSITIONING](./COURT_POSITIONING.md)에
> 별도 기준으로 둔다 — 본 문서가 *어색한 장면*(증상→수정)이라면, 그쪽은 *왜 그 자리인가*(역할 원리)다.
> 포지션 룰(32·43·45·46·49 등)이 오락가락한 근본 원인이 "magic 숫자"라 그 모델을 단일 기준으로 박는다.

## 대원칙

1. **보드는 엔진의 사실(`PointHow`)을 그대로 그린다** — 연출이 결과를 지어내지 않는다(분포 = 엔진 분포).
2. **모순 금지가 리얼리즘보다 우선** — "공에 닿았는데 못 잡음", "죽은 공에 대형 재배치" 같은
   장면은 한 컷만 나와도 관전 몰입이 깨진다.
3. 렌더(MatchCourt)와 감사기는 **같은 순수 모듈**(courtPath/courtDirector/courtLayout)을 쓴다 —
   화면에 보이는 위치 = 검사하는 위치.

### 설계 결정: 랠리 터치 스크립트(touches) — 보드가 엔진 랠리를 "재생"한다 (2026-06-23 채택)

> 사용자 결정: **엔진 기록 재설계**로 디그/세트(나아가 비종결 공격시도)까지 보드=박스 100%로 맞춘다.
> 근본 원인은 **보드가 자체 난수로 랠리를 즉흥 합성**하고(누가 디그/세트/공격) 박스는 엔진이 따로 산출 →
> 둘이 다른 선수·다른 hop 수. 1:1로 묶이지 않는 구조다(recv/set 종결만 단발 배선으로 맞췄음).

**해결 골격**: 엔진이 한 점의 **터치 순서**를 `PointLog.touches: TouchEvent[]`로 그대로 내보내고
(서브→리시브→[세트→공격→(디그→전환)]\*→종결), 보드 `courtPath`가 **그 순서·그 선수를 재생**한다
(좌표·연출은 보드가 합성하되 "누가·몇 번 터치했나"는 엔진이 정본). 그러면 디그/세트/공격시도가 **구성상**
박스와 일치 — 형제 칸 전부 닫힌다.

**왜 안전한가(단계 분리)**:
- **1단계(가산·중립) ✅ 완료(2026-06-23)**: 엔진이 `touches`를 **기록만** 한다(이미 정해진 sp/recvPlayer/setter/attacker/dg/blocker
  id를 순서대로 push — rng 추가 소비 0). `MatchOpts.touches`로 켜며, 끄면 `PointLog.touches`=undefined → playRally가 no-op.
  box·boxTimeline처럼 결과 바이트 불변(`_ev_box` neutrality PASS로 증명). 보드는 아직 안 씀. 코히런트 sanity(`_ev_touches`):
  첫 터치=serve·종결 atk==byId·ace serve==byId **100%**(cap은 byId 생략이라 제외). `TouchAct`/`TouchEvent` 타입은 `engine/rally.ts`.
- **2단계(보드 재생)**: `courtPath`가 즉흥 hop 루프 대신 `touches`를 소비해 디그/세트/공격수 슬롯을 배치.
  auditBoard 16룰(워프·수비홀·유령터치…) 전부 0 유지가 통과 기준. 디그 분포가 박스(87.7% 리베로 등)와
  일치하는지 `_ev_digmatch`(재작성, A/B 민감)로 측정.
- **3단계(밸런스 재검증)**: 디그/어시 분포가 KOVO 밴드를 깨지 않는지 N≥1만 재시뮬(STATS_PROTOCOL).

> 단계마다 커밋·검증. 1단계는 기존 동작 무변경이라 회귀 위험 최소. 2단계가 보드 핵심 변경이라 auditBoard가 관문.
>
> **에이스(네트인) 경계**: 보드가 클린 리시버를 안 그리는 유일한 정상 케이스 = 에이스(공이 닿을 수 없는 곳). 박스는
> 지정 리시버를 `recvErr`로 기장(bookkeeping)하되 코트엔 터치가 없다. idx=recvIdx로 억지 귀속하면 "공에서 먼 처리자"
> = auditBoard 유령터치(I) 443건이 떴다 → 되돌림. `_ev_recvmatch`는 클린/노클린을 갈라 노클린이 전부 how=ace인지(누수 0) 검증.
