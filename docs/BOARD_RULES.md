# BOARD_RULES — 관전 연출 검증 기준 (구단주의 눈)

> 사용자가 폰으로 경기를 **보다가** 직접 잡아낸 어색한 장면들을 검증 가능한 기준으로 정식화한 문서.
> 관전형 게임에서 경기 연출은 1순위 투자처(CLAUDE.md 기둥 1)이므로, 이 기준들은 전부
> `tools/auditBoard.ts`의 **상설 룰**로 자동 검사된다. 새 주의사항이 나오면:
> **① 이 문서에 행 추가 → ② 연출 수정 → ③ 감사 룰 추가(회귀 방지) → ④ 풀배터리 0건** 순서로 처리한다.
>
> 검증 실행: `/verify-board` 스킬 또는 `npx tsx tools/auditBoard.ts [경기수]`
>
> **수비/리시브 포지셔닝의 "왜"** (역할→위치 자동화 모델)는 [COURT_POSITIONING](./COURT_POSITIONING.md)에
> 별도 기준으로 둔다 — 본 문서가 *어색한 장면*(증상→수정)이라면, 그쪽은 *왜 그 자리인가*(역할 원리)다.
> 포지션 룰(32·43·45·46·49 등)이 오락가락한 근본 원인이 "magic 숫자"라 그 모델을 단일 기준으로 박는다.

## 대원칙

1. **보드는 엔진의 사실(`PointHow`)을 그대로 그린다** — 연출이 결과를 지어내지 않는다(분포 = 엔진 분포).
2. **모순 금지가 리얼리즘보다 우선** — "공에 닿았는데 못 잡음", "죽은 공에 대형 재배치" 같은
   장면은 한 컷만 나와도 관전 몰입이 깨진다.
3. 렌더(MatchCourt)와 감사기는 **같은 순수 모듈**(courtPath/courtDirector/courtLayout)을 쓴다 —
   화면에 보이는 위치 = 검사하는 위치.

#### 2단계 착수 전 보드 루프 조사(2026-06-24) — 핵심 긴장점 규명

> docs-first로 `components/courtPath.ts` `ballPath` 합성 루프를 정독한 결과. 추정 아님(코드 정독).

**현재 보드 합성 구조**: `ballPath`는 hop 루프(`for hop<6`)에서 **매 hop 자체 난수**로 결정한다 —
디거(`nearestDig`=근접+리베로콜 1.7배), 토서(`tosserIdx`=근접 비-firstTouch), 공격수(`atkIdx`=공격종류
난수 픽), **그리고 hop 구조 자체**(블록커버 살림 `BLOCK_COVER_RATE`·프리볼 전환 0.5·팁 디그세이브 0.18).
종결 `byId`/`setId`/`recvId`만 마지막에 슬롯을 덮어쓴다 → 종결 칸만 맞고 비종결이 어긋난다.

**긴장점 ①(터치 수 불일치)**: 보드는 엔진엔 **없는 연출 터치**를 추가로 그린다 — 블록커버(룰 25/36),
프리볼(룰 37), 팁 디그세이브(룰 9), 클린 디그 후 커버 무버. 엔진 `touches`는 박스 사건(serve/recv/set/atk/dig)만.
→ 보드가 `touches`만 그대로 재생하면 이 연출들이 사라진다(사용자가 명시 요청한 장면 — 1순위 연출 품질 훼손).
**해소 방향**: 박스 귀속 터치(엔진 dig/set/atk)는 `touches` id로 **정본 배치**하되, 보드 고유 연출 터치
(커버 다이브·프리볼 아치)는 박스 미귀속 **순수 비주얼 무버**로 유지(idx 있는 무버지만 box 사건 아님).

**긴장점 ②(디그 귀속이 리베로 집중)**: 엔진 dig 터치는 현재 `defenders(df).best-dig`(전역 R.dig 최고)
= **양팀 통틀어 5명·리베로 87.7%**(`_ev_digdist`). 보드가 이 id로 디그를 그리면 **리베로가 거의 모든 디그를
혼자** 처리하고 다른 후위가 안 움직인다 → 현 보드(근접 기반 분산)보다 **연출이 단조로워진다**. 즉 1:1 정합과
연출 다양성이 **충돌**한다(디그 칸이 "구조적 한계"인 진짜 이유). **해소는 엔진 dig 귀속을 per-touch 현실화**
(전역 best-dig가 아니라 그 코스의 기하/역할 가중 디거 — 리베로가 1위는 유지하되 후위 OH/OP/세터가 분담)
**해야** 박스도 현실적이고 보드도 다양하면서 둘이 일치한다. 팀 디그 총량(15.9/세트, KOVO 정렬)은 보존,
**개별 귀속만 재분배**(경기 승패 불변 — 디그는 이미 성공, 누구에게 기록되나만 바뀜 → 개인 스탯/디그왕·리베로
가치엔 영향). 이건 박스 분포를 바꾸므로 KOVO 밴드 재검증 필요(STATS_PROTOCOL).

**→ 단계 재정의(조사 반영)**: 디그 정합의 뿌리가 **엔진 귀속 모델**이라, 보드 재생(2단계)보다 **엔진 dig/assist
per-touch 현실화가 먼저** 와야 보드가 맞출 대상(touches)이 현실적이 된다. 순서를 **2a(엔진 귀속 재모델+KOVO
재검증) → 2b(보드 재생)** 로 한다. 어시(setId)도 같은 뿌리 — 세터가 1구 디그하면 어시는 실제 세트한 선수로
(현재는 무조건 지정 세터). **사용자 결정 사안**(연출 다양성 vs 박스 정합, 디그 재분배 강도) → 착수 전 확인.

**사용자 결정(2026-06-24): 현실적 분산 — 리베로 1위 유지.** 엔진 dig 귀속을 전역 best-dig(리베로 87.7%
독식)에서 **후위 수비수 가중 분산**(리베로가 최다지만 후위 OH/OP·가끔 세터가 분담)으로 바꾼다. 실제 배구처럼
박스 디그왕은 리베로지만 독식은 아니다.
- **승패·팀 디그 총량 불변(엄수)**: 디그 사건 수·체력 소모(drain)는 기존 best-dig(`dg0`) 그대로 둬 **메인 rng·결과
  바이트 불변**. **귀속(box digSucc·touches dig·telemetry)만** 현실 분산 선수(`dgCredit`)로. 즉 "누가 지치나"는
  그대로, "누구 기록인가"만 바뀐다 — 사용자 결정 문구 그대로.
- **전용 rng 스트림 `digRng`**(boxRng처럼 항상 생성, 메인·boxRng 불간섭)로 가중 추첨 → recvId(이미 커밋·검증)
  스트림 무교란, box on/off 바이트 중립 보존. 디그마다 항상 소비(box 유무 무관)해 스트림 lockstep.
- **모델**: `defenders(df)`(후위 3인, 후위 MB→리베로) 중 dig 레이팅 가중 추첨. 목표는 **리베로 단독 1위·나머지
  유의미 분담**(허위 KOVO 포지션 오라클 금지 — 정량 타깃 대신 정성: 리베로 최다·0인 수비수 없음). `_ev_digdist`로
  분포 측정·튜닝(N≥1만, STATS_PROTOCOL). 팀 디그/세트(15.9, KOVO 정렬)는 사건 수 불변이라 자동 보존.

#### 2b 설계(2026-06-24) — 보드 디그 마커를 엔진 디그 귀속(touches)에 그래프트

> 통계(box·통산)는 이미 단일 진실. 2b는 **코트 위 디그 애니메이션의 선수**를 박스 디그 귀속자와 맞춘다
> (rules 52/55/56과 같은 "화면=기록" 클래스의 디그 칸). recvId/setId/byId 그래프트 패턴의 연장.

**배선**: `data/matchBox.ts`가 sim에 `touches: true` 추가 → `reconstructRallies`가 `pt.touches`를 `RallyState`→
`RallyLike`로 전달 → `ballPath`가 소비. (관전 보드·경기 상세 둘 다 `buildMatchBox` 단일 경로라 자동 일치.)

**그래프트(점진·저위험, 루프 전면 재작성 회피)**: 엔진 디그 터치 순서 `digTouches = touches.filter(act==='dig')`를
큐로 들고, 보드가 **att를 뒤집는 실제 디그**(클린 디그·팁 디그세이브·블록 전환 디그 — `att=def`로 전환되는 것)를
그릴 때마다 큐에서 하나 꺼내 **그 선수 슬롯을 디거로** 쓴다. 블록커버(att 유지=재공격)·프리볼(엔진에 없는 보드
연출)은 큐를 **소비 안 함**(비박스 비주얼 — 보드 고유 선수로 그대로, "살림/찬스볼"로 식별).
- **워프 회피(핵심)**: 엔진이 고른 디거가 공에서 멀면 순간이동(룰 A/N)이 난다 → 디그 좌표를 **그 디거 위치 쪽으로**
  잡는다(스파이크 낙구를 디거 존으로 유도 — 공격은 원래 수비수 쪽으로 가므로 자연스러움). nearestDig가 고르던
  좌표 대신 "엔진 디거의 실제 위치"로 공을 보낸다.
- **hop 구조 불일치**: 보드가 프리볼/커버로 att-flip을 추가하면 보드 디그 수 > 엔진 디그 수일 수 있다 → 큐 소진 시
  잔여 보드 디그는 nearestDig 폴백(그 디그는 박스에 없는 보드 연출 디그라 무방). 큐가 남으면(보드가 더 일찍 종결)
  남은 엔진 디그는 안 그려짐(그 박스 디그는 여전히 집계됨 — 통계 불변). **per-event 100%는 구조 재생이 전제**라,
  2b 그래프트의 목표는 **분포 일치 + 높은 per-event 일치율**(setId 86%처럼) + auditBoard 0. 측정 후 잔여 판단.

**검증 관문**: ① `_ev_digmatch`(재작성·A/B 민감) — att-flip 보드 디그 == 엔진 디그 큐 순서 일치율 + 분포 대조.
② auditBoard 16룰 **0**(워프·수비홀·유령터치 — 디거 좌표 유도가 새 위반을 안 만드는지). ③ 기존 `_ev_*`·205 테스트 불변.
**측정 먼저**: 그래프트 일치율이 낮으면(보드 구조가 자주 어긋나면) 구조 재생(hop 수도 touches서)으로 escalate.

**2b ✅ 완료(2026-06-24) — 100.0%**: 그래프트 + 종결 동기 + 엔진 터치 보강의 3단으로 per-event 100% 달성.
- **보드(courtPath.ballPath)**: `digQ`(touches dig 큐)+`takeEngineDig`(슬롯 매핑·리베로→후위MB display-equiv)+
  `diggerSpot`(공을 엔진 디거 위치로 유도=워프0). 3개 att-flip 디그(클린·팁세이브·블록전환)+**프리볼 수신**도 그래프트.
  종결 타이밍 = `digQ` 소진 시(`HOP_MAX=digQ.length+6`) → 보드 랠리 길이 = 엔진 attack 수.
- **엔진(rally.ts) — 결정적 한 수**: att-flip 3종 중 **클린 디그만** dig 터치를 냈고 **팁-dug·소프트블록 전환은 누락** →
  보드가 그 전환도 디그로 그려 큐와 사이드 드리프트(측정서 h0부터 100% 사이드 어긋남으로 규명). → 팁-dug에 `digSucc`+dig
  터치(stats.digs에 이미 포함 → digSucc==stats.digs), 소프트블록에 보드정합용 dig 터치(digSucc 미가산, 보존식 유지).
- **배선**: `data/matchBox.ts` `touches:true` → `reconstructRallies`(pt.touches) → `RallyLike`. auditBoard도 `touches:true`로
  실제 렌더 감사. `ballPath`에 `digSink`(측정 전용·가산).
- **결과**: 보드 디거==엔진 귀속 **100.0%**(A/B chance 10%≪·드리프트 0), 분포 byte-동일. auditBoard **0**·205 테스트·
  byId 100%·recvId 100%·setId 100%·KOVO 밴드·생산 0분기·digdist 리베로 1위 전부 불변.
> 교훈 1(리베로 표시): 엔진은 리베로 id로 귀속하나 보드 six엔 후위 MB → `takeEngineDig`가 리베로를 후위 MB 슬롯에 매핑(룰 55).
>  안 하면 리베로 디그(29%)가 전부 폴백돼 보드서 L이 사라짐(측정서 발견 63.6%→88.5%).
> 교훈 2(사건 누락): 박스/터치가 같은 클래스 사건(디그)을 **일부만** 기록하면(클린만, 팁·소프트 누락) 보드 정합이 깨진다 —
>  엔진의 모든 att-flip 전환에 터치를 발행해야 보드가 전부 따라온다(측정으로 누락 규명 → 88.5%→100%).

> 단계마다 커밋·검증. 1단계는 기존 동작 무변경이라 회귀 위험 최소. 2a는 박스 분포가 바뀌니 KOVO 밴드가 관문,
> 2b는 보드 핵심 변경이라 auditBoard가 관문.
