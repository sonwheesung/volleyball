# SHUTDOWN_POLICY — 서비스 종료 고지·환불 런북 (#107)

> 정본. 서비스 전부 종료 시 집행 절차. 설계 근거는 `MONETIZATION_SYSTEM.md` §14.2, 법령 요건은
> `payment-security-compliance` 스킬(유저레벨). 이용자 대면 문구는 `data/legalText.ts`(약관 제5조·정책)와
> 공개 페이지 `server/app/terms/page.tsx`가 정본 — 이 문서는 **운영자 집행 절차**만 담는다.

## 0. 한 줄 요약

종료 예정일 **최소 30일 전** 앱 공지 + 이메일 병행 고지 → 유상 다이아 구매 중단(스토어 상품 비활성) →
D-day 이후 **최소 30일** 환불 접수 유지 → **미사용 유상 다이아**를 원장 재생으로 산정해 환불.

## 1. 법 근거 (한 줄씩)

- **공정위 모바일게임 표준약관** — 일방적 종료·무환불 조항은 불공정약관으로 무효. 30일 전 고지 + 미사용 유상 재화 환불 의무.
- **전기통신사업법**(부가통신사업자 휴·폐업 이용자 고지) — 종료 사실의 사전 고지 의무.
- **콘텐츠산업진흥법 / 콘텐츠이용자보호지침** — 미사용 유료 콘텐츠(재화)의 환불 준거.
- **게임산업법 §25 계열**(영업 폐지 시 이용자 보호) — 종료 시 이용자 재산(유상 재화) 보호.
- **전자상거래법 §6**(보존기간) — 결제·환불 원장 5년 보존이라 종료 후에도 환불 산정 근거가 남음(BACKEND_SYSTEM §13.9).

## 2. 절차 (타임라인)

### D-30 이전 — 고지
- **앱 내 공지**: 기존 공지 인프라 재사용(`announcements` 테이블, `pinned=true`). 관리자 `/ops-9f3a2c` → 공지 발행,
  `startsAt`=고지일 · `endsAt`=null(무기한, 종료까지 상시 노출). 무푸시 관전형이라 앱 진입 시 bootstrap이 활성분 반환(§13.11).
- **이메일 병행**: 표준약관상 앱 단독 고지는 부적합 — 이메일 고지 병행. (계정에 이메일 미저장이므로 스토어/플랫폼
  공지 채널·개발자 연락 채널·앱 배너로 보강. 개인정보처리방침상 이메일 미수집 정책과 충돌 없음 — 별도 수집 없이 공지.)
- 공지 문안: 종료 예정일 · 사유 · 유상 다이아 구매 중단일 · 환불 접수 기간·방법(문의하기 환불 카테고리).

### D-30 ~ D-day — 신규 결제 차단
- **스토어 상품 비활성으로 충분** — Google Play 콘솔에서 인앱 상품을 비활성(deactivate)하면 신규 구매가 원천 차단된다.
  RevenueCat offering도 내려 노출 자체 제거. **판단(기록)**: 서버 `confirm`/웹훅에 별도 "결제 차단 플래그"를 추가하지 **않는다** —
  구매 자체가 스토어에서 발생 불가라 서버에 도달할 트랜잭션이 없다(방어선 중복 불필요). 만약 종료를 되돌릴 가능성이 있으면
  상품 비활성만으로 가역적(코드 배포 불필요)이라는 점도 스토어 비활성 채택 근거.
- 이미 지급된 유상 다이아는 그대로 사용 가능(종료일까지 정상 플레이).

### D-day — 종료
- `server_setting.maintenance=true` + `maintenanceTitle/Body`에 종료 안내 → 앱 진입 차단(부팅 게이트, §13.11·§13.16).
- 환불 문의 창구(문의하기 #45)는 **유지**(종료 후에도 접수).

### D-day 이후 — 환불 접수·집행 (최소 30일)
1. **산정**: `server/tools/shutdownRefundReport.ts`(읽기 전용)로 유저별 **미사용 유상 다이아**를 원장 재생으로 계산.
   - 실행: `cd server && npx tsx tools/shutdownRefundReport.ts`(운영 겨냥 시 `DATABASE_URL` 오버라이드).
   - `--selftest`로 산정 로직 자체 검증(손계산 4시나리오 A/B). `--include-sandbox`는 테스터 지급 포함(기본 제외).
2. **환불 실행**: 리포트는 **지급 실행이 아니다**(읽기 전용). 실제 환불은 스토어 환불 절차 + 문의 접수 건별 수동 처리.
   환불 시 원장에 `reason='refund'`(음수) 기록 → 잔액 클로백(음수 잔액=부채, §13.17). 산정과 실행의 이중차감은
   문의 상태(`refunded`)로 방지.
3. **접수 기간**: D-day 이후 **최소 30일** 유지(표준약관 종료 후 환불 창구).

## 3. 환불 산정 규칙 (도구와 약관·정책 일치)

원장(`wallet_ledger`, append-only)을 유저별·시간순 재생:

| 원장 항목 | 처리 |
|---|---|
| `purchase`(+) | **유상** 적립(환불 대상 풀) |
| `ad`/`achievement`/`coupon`/`welcome`/`adjust`(+) | **무상** 적립(환불 비대상) |
| `camp`(−) 등 소비 | **무상 우선 소진**(약관 제2조③·정책과 일치) → 무상 소진 후 유상 차감 |
| `refund`(−) | **유상 풀에서 회수**(해당 구매 클로백) |
| 최종 잔액 < 0(부채) | 유상 잔여 **0** 처리 |
| `ref` 끝이 `:sandbox`인 `purchase`(+) | **집계 제외**(테스터 지급 — `--include-sandbox` 시만 포함) |

> "무상 우선 소진"은 이용자에게 유리(유상 잔여를 최대한 보존 → 환불액 최대화)하며, 약관 제2조③·제5조③·
> 운영정책 재화 조항과 일치. 유상/무상 구분은 원장 `reason`+`ref` 태그가 진실.
