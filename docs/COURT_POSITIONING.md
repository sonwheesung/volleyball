# COURT_POSITIONING — 수비/리시브 포지셔닝 모델 (보드 연출 단일 기준)

> **상태: ✅ 구현 반영(`receiveFormation`/`serveFormation`/`switchedSpots`에 좌표 박힘 — 룰 43·45·47·49 등).
> 단 "magic 숫자 제거"(역할유도 `positionOf` 자동화)는 미달 — 여전히 하드코딩 좌표.** 사용자 교습(2026-06-21) + 5-1 서브리시브 리서치 기반. 추측 금지 —
> 이 문서가 `receiveFormation`·`serveFormation`·`switchedSpots`·`courtDirector`의 **단일 기준**이다.
> 위치 변경은 코드보다 **이 문서를 먼저** 고친다([[verify-domain-definitions]]·DOC_DISCIPLINE).

---

## 0. 왜 이 문서가 필요한가 (자동화가 안 되던 이유)

지금까지 보드 포지션은 **자리마다 magic 숫자**(예: 전위 패서 y=0.645→0.695→0.75→0.665)를 박았다.
그래서 로테이션마다 어긋나고 같은 곳을 반복 수정했다(BOARD_RULES 룰 32→43→49 오락가락).

> **핵심 진단**: 위치는 magic 숫자가 아니라 **역할에서 유도(derive)** 되어야 자동화된다.
> "이 선수는 왜 여기 있나"가 역할로 설명되면, 어떤 로테이션이든 자리가 자동으로 나온다.
> 이 문서는 그 **역할 → 위치 규칙**을 정의한다.

---

## 1. 지배 원칙 (모든 자리의 상위 제약)

1. **오버랩(서브 컨택 *순간만*)**: 같은 행 좌<중<우(전위 4·3·2 / 후위 5·6·1), 같은 열 전위<후위
   (전위가 네트에 더 가까움: 4·5 / 3·6 / 2·1). 이 순서만 지키면 나머지 자리는 자유.
   - **인접 쌍만 따진다(FIVB 7.4, 출처 확인 2026-06-24)**: 전위 선수는 **같은 열의 후위 짝**보다만 네트
     쪽이면 된다 — **다른 열의 후위 선수보다 깊어도 합법**이다. 예) 전위 OH(z2)가 리베로(z6)·백OH(z5)보다
     깊어도 무방, **세터(z1, 같은 우열 짝)보다만** 네트 쪽이면 OK. ⇒ "전위 패서가 후위밴드(y≥0.72)에
     들어감"은 **오버랩 위반이 아니다**(자기 열 짝보다 깊을 때만 위반). `_dv_receive`의 "후위밴드 침범"은
     룰이 아니라 **전술 프록시**일 뿐(룰 49 재고 — 아래 변경이력 2026-06-24).
   - **서브 팀은 오버랩 제약 없음(FIVB 2025–2028)**: 서브 순간 **서브 팀 전원**이 아무 위치나 가능
     (받는 팀만 로테이션 순서 유지). ⇒ 코드 검사(`auditBoard`)가 **서버만** 면제하는 건 과한 제약 —
     서브 팀 전원 면제가 룰에 맞다(별도 정리 대상). 출처: FIVB Official Rules 2025–2028.
2. **역할이 자리를 정한다**: passer / setter / blocker / digger / attacker. 위치는 역할의 *일*에서 나온다.
3. **보드 좌표**: 네트 y=0.5, 우리 진영에서 **전위=네트 쪽(0.5 근처), 후위=베이스라인 쪽(1.0 근처)**.
   (홈=하단, 원정=상단 점대칭.)

## 2. 역할 정의

| 역할 | 일 | 위치 원리 |
|---|---|---|
| **세터 S** | 세트 전담. **받으면(서브/디그) 공격을 못 짠다** → 가능한 한 첫 터치를 피하고 네트-우 타깃으로 | 받기: 패스 레인에서 빠짐 / 수비: 후위 디그 |
| **패서** | 서브 리시브 = **리베로 + OH 2명**(3-패서, 5-1 표준) | 좌·중·우 레인, 3~5m |
| **블로커** | 전위 3인, 상대 히터 정면에서 블록 | 네트, x=상대 히터 |
| **디거** | 후위, 블록 뒤/옆 공간 디그 | 페리미터 베이스(깊은 코너+중앙) |
| **공격수** | 전위 비패서(MB 속공·OP 우측), 패스 후 어택 | 네트 앞 자기 공격 출발점 |
| **리베로 L** | 후위 MB 대체. 항상 패서·최전방 디거 | 전위·서브·공격·블록 **불가** |

## 3. 시나리오 A — 우리가 받을 때 (상대 서브)

목표: **서브를 잘 패스 + 세터를 네트로 보내 공격을 짠다.**

### A-1. 세터가 **후위**일 때 (핵심 — 사용자 교습)
| 선수 | 자리 | **왜** |
|---|---|---|
| **패서 3(L+OH 2)** | 좌·중·우 레인, 3~5m | 서브 코스를 3등분해 "마이볼". 리베로 중앙 깊게(최고 패서), OH 좌우 |
| **세터(후위)** | 자기 열 **전위 파트너 뒤**(더 깊게) | **받으면 세트 못 함** → 패스 레인에서 빠진다. 전위 파트너 뒤 = 오버랩 합법(세터>전위) + 서브 회피 |
| **세터 앞 전위 파트너** | 한 발 내려와 세터 앞 | 세터가 숨을 수 있게. 공 오면 **리시브**, 안 오면 **공격 전환**, 리시브 깨지면 **커버** |
| **전위 MB/OP(비패서)** | 네트 앞 공격 출발점 | 패스 안 함. 패스 후 바로 때리러 대기 |
| **서브 순간(전환)** | 세터만 **네트-우 타깃 침투** | 나머지는 제자리에서 받거나 전환. **전위가 세터 자리로 가는 게 아니다 — 세터가 뒤→앞** |

### A-2. 세터가 **전위**일 때
- 세터가 이미 네트 근처 → **침투 불필요, 문제 없음.** 패서는 L+OH 2(또는 로테이션 상황).

### A-1 구현 — 패서 레인 3등분(레인 재분배, 2026-07-16 ✅)
> A-1의 "좌·중·우 레인 · 서브 코스를 3등분" 을 `receiveFormation`이 실제로 만든다.
- **구현 전 버그**: 패서 x를 자기 존의 lateral 컬럼(`RF_FRONT_X`/`RF_BACK_X`)에 고정 → 세 패서가 우연히
  같은 쪽 존들이면 한쪽에 뭉쳐 반대편이 무패서. 사용자 전 로테이션 덤프 실측: rot0/3은 3등분 정상이나
  **rot1/4 중앙 공백**, **rot2/5 좌측 절반 통째 무패서**(세터가 z5 코너 0.13에 은신, 그 뒤가 비고 패서는
  0.52/0.56/0.88) — 최대 무패서 구간 1.04W.
- **수정**(패서 3인의 **x만** 재분배 — 깊이 y·비패서·세터·jit 시드는 무변경):
  1. 목표 레인 `[0.18, 0.5, 0.82]`(+jit ±0.015 결정론 흔들기)에 패서를 배정.
  2. 각 패서의 **합법 x 구간** = 자기 행 좌 이웃 존 점유자 x + ε ~ 우 이웃 존 점유자 x − ε(ε=0.03W,
     룰 Q strict>0.5px 여유). 이웃이 없는 끝 존(4·5=좌끝 / 2·1=우끝)·이웃이 패서 본인이면 코트 여백(0.08~0.92W).
  3. 패서 3인 × 레인 3개의 **순열 6가지**를 평가: 각 패서 x=clamp(레인 x, 합법 구간), 같은 행 패서 순서 위반이면
     불가, 점수=Σ|배정 x − 레인 x|. **최소 점수** 채택(동점=순열 인덱스↓, 결정론). 레인 배정은 **전단사**라
     세 레인이 항상 각 1인씩 커버.
- **오버랩(룰 Q) 보존**: 깊이(y)는 안 건드리므로 같은 열 전후 관계 무영향, 같은 행 순서는 합법 구간 클램프 + 순열
  순서 검증으로 유지. 원정(away)은 홈 프랙션에서 계산 후 기존 `mx()` 미러로 통과.
- **가드**: `tools/_dv_receive_lanes.ts`(전 로테이션·양 사이드·7팀 — 최대 무패서 구간 ≤0.55W + 오버랩 위반 0 +
  결정론 + 구 존컬럼 로직 A/B로 4/6 로테이션 붕괴 증명). `_dv_receive`(깊이 y 핀)·`_dv_overlap`(위반 0)은 무변경 통과.

## 4. 시나리오 B — 우리가 서브할 때 (서브 후 수비 전환)

목표: **상대 공격을 블록 + 빠진 공 디그.**

### B-1. 세터가 **후위**일 때 (핵심 — 사용자 교습)
| 선수 | 자리 | **왜** |
|---|---|---|
| **전위 3인** | **블록 라인**(네트) | **세터가 후위라 블록을 못 뛴다 → 전위 3명이 블록 라인.** x=상대 히터 정면(MB=상대 속공, 윙=핀) |
| **세터(후위)** | **우측 후위 디그** | 블록 못 뛰니 디거. 우측 후위 = 디그 후 바로 세트하러 네트-우 타깃에 가깝게 |
| **리베로+나머지 후위** | 페리미터 베이스(깊은 코너+중앙) | 리베로가 최전방 디펜더 |
| **좌/우 전위** | 기본 블록 준비, 코스 보이면 **팁(페인트) 커버** 풀오프 | 가운데 팁 무방비 방지(룰 48과 연결) |

> **핵심**: 세터 후위 = **전위 3명 블록, 세터가 받는다(디그)**. 상대 셋업 중엔 전위가 리시브 안 함(블록 전념).

### B-2. 세터가 **전위**일 때
- 세터도 전위라 **블록 가능**(3블록에 세터 포함될 수 있음). 후위는 리베로 포함 디그. **[디테일 확인 필요]**

## 4.5 OP 후위 백어택 발사 자리 (§7-OP — 리서치·발사계층 입증·구현 ⏸ 보류, 2026-06-27)

> **용어 교정(verify-domain-definitions)** ✅: 사용자가 "OP 파이프"라 했으나 리서치상 **OP 백어택 = 우측 후위(Zone 1)**,
> 3m 라인 뒤에서 친다. **파이프(Pipe)=중앙 후위(Zone 6)** 로 보통 **OH**가 친다 — OP와 다른 자리.
> 출처: [joinstriveon 5-1 로테이션](https://joinstriveon.com/blog/5-1-volleyball-rotation),
> [goldmedalsquared 5-1](https://www.goldmedalsquared.com/post/5-1-volleyball-rotation).

| 후위 공격수 | 발사 자리(목표) | **왜** |
|---|---|---|
| **OP** | **우측(Zone 1), 3m 라인 뒤** | OP 백어택은 우측 후위에서. 전위 우측(Zone 2) 공격과 같은 레인 |
| **OH** | **중앙(Zone 6 파이프), 3m 라인 뒤** | 파이프는 중앙 후위 OH 백어택 |

- **⏸ 1차 시도 회귀(2026-06-27, 폐기)**: `switchedSpots(offense=true)`에 후위 OP를 우측 3m 뒤로 고정하는 **순진한 정적
  오버라이드**를 넣었더니 **auditBoard 9건 회귀**(유령터치 등 — A/B로 확정: WITHOUT 0건 vs WITH 9건). 원인: switchedSpots는
  *공 무관 순수 레이아웃*이라 인플레이 전환에서 **OP가 그 공의 처리자(디그/패스)일 때도** 무조건 공격 자리로 끌어 공에서
  멀어지거나 `separateTargets`가 인접 마커(실제 처리자)를 밀어냄. → **"준비 자리"(수비 베이스)를 건드린 게 잘못.**
- **2차 시도 — "발사 자리"(올바른 계층, 발사 x는 입증·그러나 ⏸ 보류, 2026-06-27)**: 사용자 제안("공 위치를 알고
  그 백어택하는 선수를 옮긴다")대로 수비 베이스가 아니라 **실제 공격 발사 프레임**(`courtPath.ts` toss/spike, `atk==='back'`)에서만
  발사 x를 포지션별로 지정 — 기존 `sw[att].pos[atkIdx].x` → `zonePx(att, OP?1:6).x`(OP=Zone1 우측·OH=Zone6 중앙 파이프).
  - **발사 x 정확성은 입증됨**: `_dv_op_backrow.ts` 30경기 — OP 550/550 Zone1·OH 1157/1157 Zone6·결정론 0불일치·A/B 유효.
    "공격자 아닌 OP 고정"(1차 회귀 원인)은 실제로 사라졌다 — 발사 프레임은 진짜 공격자만 움직이므로.
  - **그러나 새 회귀 1건(리바운드홀, A/B 확정: WITHOUT 0 vs WITH 1)**: 경기28/랠리212에서 **OH 파이프(중앙) 백어택**의
    깊은 발사점 부근 커버가 1.7m(>1.6m 임계). 원인은 발사 계층이 아니라 **커버 완전성**: 발사 x를 *그 선수의 실제 위치에서
    멀리(중앙/코너)로 순간이동*시키면, 커버(`coverSpots`)는 새 타점(ahx) **최근접 3인**만 배치하므로 네트~깊은 타점 회랑을
    전부 덮지 못하는 랠리가 생긴다(룰 E는 6인 전원 최근접 ≤66px 요구). 1차 시도가 *공격자 아닌* OP를 끌어 9건을 낸 것과
    달리, 2차는 *공격자 본인*을 멀리 끌어 **커버 풀이 회랑을 못 메우는** 1건 — 클래스는 같다(억지 좌표 ↔ 정적 커버 불일치).
  - **올바른 선행조건(다음 시도 시)**: 발사 x 변경 전에 **백어택 커버 완전성**을 먼저 보장해야 한다 — `coverSpots(backAtk)`가
    네트~타점 회랑을 빈틈없이 덮도록(전위 1인 네트 커버 + 깊은 타점 측면 커버 보강), 그 자체를 auditBoard 룰 E로 검증한 뒤
    발사 x를 켠다. 커버 로직 수술은 파급이 커 단독 작업으로 분리. **머지 조건: `_dv_op_backrow`(발사 x) + auditBoard 0건 회귀 동시 통과.**

## 4.9 시나리오 C — 랠리 중 전환 (공이 넘어간 뒤) — 태스크 #131

목표: **우리 공격이 상대 코트로 넘어간 순간부터, 상대 공격을 막을 수비 대형으로 즉시 전환한다.**
시나리오 A/B가 "서브 컨택 국면"을 다뤘다면, C는 **인플레이 랠리 중**(상대가 리시브/디그 → 세트 → 스파이크)
우리(수비팀)가 어디에 서는가다. 정본 진단·회차는 `POSITIONING_TUNING_LOG.md` R1·R2.

### C-1. 우리 공격 직후 — 상대가 받는 `pass` 국면
| 선수 | 자리 | **왜** |
|---|---|---|
| **전위 3인** | **네트 블록 레디 라인**(번치, 중앙 3분의 1) | 상대 세트 전 네트에 붙어 3인 블록 준비. `serveFormation` `SV_FRONT_X`와 같은 결(중앙 모음 → 세터 읽고 핀으로 릴리즈). **구석/베이스로 후퇴 금지** |
| **후위 3인** | **페리미터**(라인딥·크로스딥·리베로 셰이드) | 코트를 넓게 덮는 딥 베이스. 윙 2인 = 라인/크로스 딥 코너, 리베로(중앙) = 블록 그림자 피한 **크로스 셰이드**로 최심 |
| **앵커** | 공 **현재 위치**(상대 패스 지점) / 중립 컵 | 아직 안 올라온 **토스 x(미래) 미사용** — 상대가 세트를 올리기 전엔 공격 코스를 모른다(앵커 인과성) |

> **핵심(B-1과 결 같음)**: 세터 후위 = 전위 3인이 블록 준비(네트), 후위 3인이 페리미터로 코트를 덮는다.
> A/B는 "서브 순간"의 정적 오버랩 베이스, C는 그 직후 **랠리가 도는 동안**의 동적 수비 자리다.

### C-2. 상대 세트~스파이크 — `toss`/`spike` 국면
- **toss부터**: 전위 3인 중 코스에 맞는 2~3인이 **블록 벽**(`blockerWall`, 공격 x 정면), 벽 미선택 전위는
  **팁 풀오프**(룰 48, 시임/팁 커버). 후위 3인은 **기존 `fanSlots`(공격 x 컵)로 수렴**.
- 즉 pass의 (중립/공 현재 위치) 페리미터 → toss/spike의 공격 x 페리미터로 **자연 이동**(구석 왕복 없음).

### C-3 — 이전 결함 (구현 전, R2·R1)
- **구 동작**: `pass` 국면 수비팀이 `switchedSpots` 베이스(전위 `XF`·후위 `XB` 구석 `[0.15,0.5,0.85]`/y0.80·0.905)로
  떨어지고 `toss`에서야 `blockerWall`+`fanSlots` 발동 → 매 hop마다 **후위 부채꼴↔구석 왕복 진동** +
  **블록 레디 국면 부재** + **벽 소환식**(quick 토스 230ms에 x0.2W→벽 이동 ~950ms로 물리적 도착 불가).
- **형제 결함 ⓐ**: 멀티볼 `return` 구간에서 서브팀이 `receiveFormation`으로 향하다 `walk`에서
  `serveFormation`으로 꺾이는 지그재그(`holdReceive`가 양 팀 적용) → 서브팀은 `return`부터 `serveFormation`.
- **형제 결함 ⓑ**: 디그 기하(`courtPath` `chasersTo`/`nearestDig`/`diggerSpot`)가 `swDef`(구 베이스) 기준 —
  목표 대형을 바꾸면 디그 선정/유도가 신 목표(전환 페리미터) 기준으로 정합해야 한다.

### C-4 — 통과 조건 (독립 리뷰 확정 — 위반 시 실패)
1. **WP kind·hop 수 불변** — 새 세그먼트 추가 금지. 기존 `pass`/`toss` 세그먼트에서 `segmentTargets`가 주는
   **목표 좌표만 재지정**. 터치 스크립트·`auditBoard` 구조 무접촉.
2. **앵커 인과성** — 상대 `pass` 중 우리 수비 대형의 앵커에 **미래의 토스 x 사용 금지**. 공 현재 위치/디그 지점
   또는 중립 컵만. `toss` 구간부터 기존 `fanSlots`(공격 x) 수렴.
3. **벽 정렬** — 블로커 선정·순서를 (구 `switchedSpots`가 아닌) 시나리오 C **실제 사전 위치(블록 레디 라인)**
   기준으로 갱신. `checkBlockerCross` 0% 재확인(룰 35 재발 방지).
4. **룰 48/31 우선순위** — 블록 레디는 전위 3인이되, `toss` 구간에서 벽 미선택 전위는 **룰 48 팁 풀오프가 우선**.
   방금 스파이크한 공격수의 착지 정지(룰 31)는 **렌더 전용**이므로, 신규 가드(`_dv_detour`)는 **director 목표 계층**을
   측정한다(렌더 `jumpHold`는 목표가 아니라 애니메이션 상태라 director가 주는 좌표엔 안 나타남).
5. **rng 미소비·프리픽스 불변** — 흔들기는 `jit`(시드 해시)만. `ballPath` rng 소비 순서 불변.
6. **파라미터 격리** — 신규 값만 `components/formationParams.ts`에(블록 레디 라인 y·번치 폭, 페리미터
   라인딥/크로스딥/셰이드 계수, defTransition 깊이), 값마다 근거 주석. **기존 검증 좌표
   (`receiveFormation`·`serveFormation`·`switchedSpots`·`fanSlots` 상수) 이동 금지.**

## 5. 자동화 방향 (구현 시 — magic 숫자 제거)

자리마다 숫자를 박는 대신 **역할 → 위치 함수**로:

```
positionOf(player, rotation, phase) =
  role(player, rotation, phase) 에 따라:
    passer            → 레인 x(자기 열) · 패스 깊이 y
    setter(후위·받기)  → 전위 파트너 뒤 → (서브 컨택) 네트-우 침투
    setter(후위·수비)  → 우측 후위 디그
    front 비패서       → 네트 앞 공격/블록 출발점
    blocker(수비)      → 네트, x=상대 히터 정면
    digger             → 페리미터 베이스
```

코드 매핑:
- `receiveFormation` ← 시나리오 A
- `serveFormation`·`servingDefBase`·`switchedSpots` ← 시나리오 B
- `courtDirector` 서브 순간 전환 ← A의 "세터만 침투" 로직
