# TEST_METHODOLOGY — 오류를 "새로" 찾는 방법론

> 이 문서는 **테스트 케이스(무엇을 검사하나)** 가 아니라 **오류 발견 방법(어떻게 새 버그를 찾나)** 을
> 적는다. 케이스 레지스트리는 [`EDGE_CASES`](./EDGE_CASES.md)(영입·오프시즌)·[`BOARD_RULES`](./BOARD_RULES.md)(보드).
> 본 문서는 그 케이스들을 **만들어낸 발견 기법**과, 버그를 찾았을 때의 **후속 프로토콜**을 표준화한다.
>
> **핵심 원칙**: 케이스 검사는 "이미 아는 버그의 재발"만 잡는다. 새 버그는 **다른 방법**으로 찾는다.
> 그 방법을 여기 적어, 다음에도 같은 방식으로 사냥한다.

---

## 1. 발견 기법 (새 버그 사냥법)

### A. 악질/원숭이 퍼징 (adversarial · monkey)
**무엇**: 시드 난수로 **실제 사용자 진입점(zustand store 액션)** 을 수천~수만 시퀀스 난사하고,
매 스텝 불변식을 검사. "보통 사람이 안 하는 말도 안 되는 진행"을 자동 생성.
- **악질**: 경계 악용(하한까지 방출 후 또 방출, 정원 꽉인데 영입), 순서 꼬기(드래프트 전 endSeason),
  비정상 값(타팀 id·음수·NaN·빈 문자열).
- **원숭이**: 전 액션 무작위 혼합 + 장기 진행(수십 시즌)으로 **교차 국면 상호작용**을 노출.
- **도구**: `tools/_gt_monkey.ts`(store 구동)·`_gt_adversarial.ts`(데이터층)·`_gt_seqbreak.ts`(순서)·
  `_gt_determinism.ts`. **반드시 `import './_gt_mock'`**(AsyncStorage 모킹) 선행.
- **왜 강한가**: 케이스 테스트가 보통 ① **데이터/리플레이층만**(store 게이트 우회) ② **단일 국면만**
  ③ **valid 입력만** 보는 사각을 전부 친다.

### B. 독립 문서기반 검증 (independent, doc-derived)
**무엇**: 별도 세션(에이전트)을 띄워 **문서만 보고**(코드 의도 미참조) 불변식을 도출해 측정 도구를
새로 짜고 엔진과 대조. 기존 감사 로직을 **재사용하지 않는다**.
- **왜 강한가**: 코드를 보고 만든 오라클은 코드와 **같이 틀린다**(허위 오라클). 문서·현실 규칙에서
  독립적으로 도출하면 **문서-코드 drift**와 **오라클 사각**을 잡는다.
- **도구/방법**: `Agent`로 `independent-verifier` 역할 스폰(아래 §3) → `tools/_dv_*.ts`.

### C. A/B 자가검증 (anti-false-oracle) — 모든 체크에 필수
**무엇**: 새로 만든 측정/검사는 **일부러 깨진 상태를 주입했을 때 반드시 실패**하는지 증명한다.
항상 통과만 하는 체크는 신뢰하지 않는다(그게 허위 오라클).
- 예: Q2(서브대기 오버랩) 추가 → walk 릴리즈를 임시 주입 → **990건 검출** 확인 후 원복.
- 예: `_gt_repro_release` → 클린 상태 0위반 + 익스플로잇 상태 검출, 둘 다 참이어야 신뢰.

### D. 육안 관전 + 실험실 (board 전용)
보드 연출은 `app/board-lab.tsx`(개발용)로 사람이 프레임씩 보고 어색한 장면을 좌표로 내보내 제보 →
측정으로 재현. 룰에 없는 새 연출 결함의 1차 발견 경로.

### E. 화면 UI 이벤트 검증 (RN 렌더 없이 정적)
RN 컴포넌트 렌더 테스트 인프라가 없으므로 **정적**으로 화면 이벤트를 검증한다:
- **죽은 네비게이션 링크**(자동): `tools/_ev_routes.ts` — expo-router 파일트리에서 유효 라우트를 도출해
  모든 `router.push/navigate/replace` 목적지를 대조(동적 `[id]`/`${}`·`(tabs)`·쿼리 정규화, A/B 자가검증).
- **핸들러 정적 감사**(화면별): 각 `app/**/*.tsx`의 이벤트(onPress·토글)를 ① 동적 라우트 잘못된 id 크래시
  (`getX`=Map.get→undefined를 화면이 `if(!x)`로 가드하나) ② store 액션 인자 유효성 ③ disabled↔store 가드
  일치 ④ 연타 멱등성 ⑤ 빈/결손 상태로 추적. 폭이 넓으면 에이전트로 화면별 분담.
- 한계: "엣지 데이터로 화면이 throw하는가"는 정적 추적(데이터층 함수가 null에 graceful한지)으로 근사.
  진짜 렌더 크래시는 RNTL 도입 시 보강(현재 미도입).
- **값→표현 매핑 완전성(범위 확장 회귀)**: 값을 색/라벨/별점으로 *구간 매핑*하는 UI는, 상한·구간이 늘면
  **새 구간 전부**가 구별·정확히 매핑되는지 본다. enum-키 `Record`는 컴파일러가 누락을 잡지만,
  **숫자 구간 비교·switch default·배열 인덱스(`floor(v/10)`)** 로 표현을 고르는 매핑엔 컴파일 가드가 없다.
  특히 *닫힌 버킷*(고정 길이 배열)은 상한을 올리면 새 구간이 옛 마지막 색으로 뭉개진다(기능 0, 시각 일관성만 깨짐).
  > 현황(2026-06-21 grep): 우리 스탯/OVR→색·별점 매핑(`Screen.tsx:109·149`·`draft.tsx:16`·`staff.tsx:68`)은
  > 전부 **open-ended `>=` 사다리**(최상위가 그 이상을 흡수)라 이 버그에 면역. **닫힌 버킷 도입 시 반드시
  > "각 구간 대표값+확장 경계 → 서로 다른 표현" 단위 테스트**를 붙인다.

### F. 결과 배선 정적 강제 (reported-but-unwired) — 메타
**무엇**: 엔진이 *보고*하는 결과 필드를, 호출측이 *실제로 적용*하는지 정적으로 강제한다. 엔진이 값을
리턴해도 어느 호출측도 안 쓰면 그 효과는 통째 증발한다(크래시·범위 위반 없이 조용히).
- **방법**: 결과 타입의 모든 필드를 manifest로 전수 분류 — `info`(표시용)/`mustApply`(호출측 적용 필수)/
  `optional`(보고만·미구현 추적). 새 필드가 미분류면 테스트 **FAIL**. mustApply는 호출측 소스에서
  소비되는지 + 회귀 테스트가 그 배선을 검증하는지 정적 확인.
- **왜 강한가**: 단위 시뮬은 엔진을 *단독* 호출해 "결과가 맞나"만 보고 "호출측이 그 결과를 쓰나"는
  안 본다. 이 사각(§1.5 ④ 의미정합)을 기계가 닫는다.
> 예(볼리볼): `computeSeasonAwards`가 `titles` 6종·`best7`·`roundMvps`를 전부 리턴하는데 `data/news.ts`는
> **`titles.scoring` 하나만** 소비 → 베스트7·기록왕 5종·라운드MVP 기사가 조용히 증발(2026-06-21 발견,
> NEWS 재설계로 배선). 같은 패턴 감시 대상: `SimResult.subUse`/`SubEvent`, `OwnerFx`, `SeasonFinance.net`.

### G. 변이 테스트 (mutation testing) — 메타
**무엇**: 소스에 *작은 버그(mutant)*를 일부러 심고 테스트(`npm test`·`tools/sim*`)를 돌려 **잡는지**(실패)
본다. 잡으면 KILLED(테스트가 일함), 못 잡으면 SURVIVED = **테스트 공백**. "스위트가 완벽한가"를 정량화.
- **방법**: 변이 매니페스트{파일·유니크 substring·치환·잡을 시뮬}를 **임시** 적용→실행→**즉시 원본 복원
  (+git 백스톱)**. 살해율(KILLED/총) 보고, SURVIVED는 그 불변식을 검증하는 케이스를 추가해 닫는다.
- ⚠️ **안전(기계가 막는다 — 규율 아님)**: `tools/_mut.ts`는 ① 대상이 **HEAD-클린**일 때만 실행 ② 실행 중
  `.mutation.lock` 생성 → **`.git/hooks/pre-commit`이 락 있으면 커밋 차단** ③ finally 복원 + 바이트검증(불일치 시
  `git checkout`) ④ SIGINT/SIGTERM/uncaught 에서도 복원+락제거 ⑤ 종료 시 git checkout 백스톱. 종료 후
  `git status` 클린 재확인 전까지 후속 커밋 보류 ([[mutation-testing-no-commit]]).
- **왜 강한가**: A/B 자가검증(§1.C)이 "이 한 체크가 사나"라면, 변이는 "스위트 전체가 *어떤 버그*를 놓치나"를 친다.

### H. 결함 주입 (fault injection · BUGGIFY)
**무엇**: 시드 결정론으로 *드물게* "유효하지만 극단적인" 스트레서(자원 결핍·지연·경계값)를 주입점에
일으켜, 평범한 진행으로는 잘 안 닿는 엣지를 강제·재현(FoundationDB BUGGIFY 패턴).
- **방법**: `if (buggify()) injectStressor()` 한 줄. 무장(arm) 시에만 시드 확률로 발동, 평소 no-op(프로덕션 무영향).
  스트레서는 *엔진이 견뎌야 할* 유효 극단이어야 한다 — 못 견뎌 불변식이 깨지면 그게 버그.
- **왜 강한가**: 원숭이 퍼징(§1.A)이 "밟는 경로"만 본다면, 결함주입은 그 경로에 *희소 스트레서*를 얹어 깊은 곳을 친다.

### I. 속성 기반 테스트 (property-based)
**무엇**: "입력이 무엇이든 이 속성은 참"을 자동 반례 탐색으로 검증(fast-check 등). 실패 시 **최소 반례로 shrink**.
- **방법**: `engine/` 순수 함수(rng·overall·production·aging 등)에 단조성·범위·포화·보존식 속성을 건다.
- **왜 강한가**: 고른 케이스(레지스트리)·밟는 경로(퍼징)가 못 보는 *입력공간의 빈 구석*을 친다.
  ⚠️ arbitrary 범위가 좁으면 경계를 못 친다 → **변이 테스트(§G)로 입력 범위 충분성을 검증**.

### J. 교차 계층 귀속 일치 (cross-layer attribution agreement) — 2026-06-23 신설
**무엇**: 같은 개념의 데이터(예: "이 득점/리시브/디그를 *누가* 했나")를 **두 계층이 각자 만들면**, 각 계층의
내부 유효성만 검사하고 **두 계층이 같은 선수를 가리키는지**는 안 본다 — 그 사각에서 "화면 ≠ 기록"이 숨는다.
- **사례(볼리볼)**: 경기 보드(courtPath)가 스파이커/리시버/디거/세터를 **자체 난수 proximity로 따로 골랐고**,
  스코어박스(엔진 박스 싱크)는 **엔진 실제 귀속**으로 집계 → "득점했는데 박스선 공격 0%"(2026-06-23). auditBoard는
  보드 *기하·HOW 분포*만 검사해 못 잡음(귀속 비교 룰 부재).
- **방법**: 두 계층의 산출에서 **같은 이벤트의 선수 id를 추출해 일치율 측정**(보드 종결 토스 idx ↔ 엔진 byId →
  `_ev_scorer`; 보드 서브리시버 분포 ↔ 박스 recvAtt 분포 → `_ev_recvmatch`; 스터프 중계 ↔ byId → `_ev_blockcomment`).
  A/B 자가검증(shuffle 대조로 무작위 수준 확인)으로 도구가 진짜 정렬을 잰다는 걸 증명.
- **왜 강한가**: 단일 계층 검사(보드 룰·박스 오라클)가 각자 PASS여도, **계층 간 불일치**는 통째로 비어 있다.
  엔진이 종결 선수 id(byId)를 내주고 보드가 그걸 쓰게 해야 일치.
- **근본 해결 진척**: 종결(득점/블록/서브)=byId 100%. **리시브 완료(2026-06-23)** — 엔진 `recvId`를 PointLog로
  내주고 courtPath recvIdx가 소비 → 클린 리시브 100%(`_ev_recvmatch`, A/B 15.2%). 핵심 함정: recvId가 box 유무로
  갈리면 sim.points 바이트중립이 깨진다 → boxRng를 **항상** 생성(메인 rng 불간섭)해 결정론·중립 보존.
  또 에이스(네트인)는 보드에 클린 리시버가 없는 게 사실 → idx 억지귀속은 유령터치를 만든다(클린/노클린 분리 검증).
  **남은 비종결**: 디그·세트(어시)·공격시도(비종결) — 같은 recvId 패턴으로 후속.

### K. 실기기 E2E 초기상태 전 화면 육안 점검 (emulator E2E) — 2026-06-30 신설
**무엇**: 헤드리스 도구는 **순수 함수/데이터 계층을 자기가 고른 입력으로** 부른다 — ① 앱의 *호출부 배선*(화면이
함수에 무엇을 넘기나)을 안 타고 ② 보통 *편한 큰 값*(풀시즌 MAX·중간 날짜)으로만 보고 ③ *한 화면씩 고립*해서
본다. 그래서 "**배선이 함수에 틀린 값을 먹임**"·"**경계 상태(첫 경기 전·시즌 막날)**"·"**두 화면이 같은 상태를
다르게 그림**"은 통째로 사각이다. 실기기(에뮬레이터)에서 **진짜 앱을 진짜 초기/경계 상태로 띄워, 사람이 전
화면을 차례로 보며 대조**하면 이 셋을 한 번에 친다. 대본·좌표는 [`EMULATOR_E2E.md`](./EMULATOR_E2E.md).
- **사례(EC-NEWS-01, 첫 경기 전)**: `simNews`·`_ev_debutgate`·`_ev_bodylen`은 전부 `buildNewsFeed`를 **MAX**로
  직접 불러 무결성·게이트·본문만 봤다(앱 호출부 미경유·풀시즌만). 첫 경기 전(0경기) 에뮬에서 `경기 결과`="치른
  경기 없음" 옆에 `리그 뉴스`="데뷔전 15점·미래 부상·후반 사건"이 **나란히 모순**으로 보여 처음 드러남. 원인은
  화면 3곳이 raw `currentDay`(앱 표준 `leagueDisplayDay` 미사용)를 넘긴 *배선* 버그.
- **방법**: 신규 세이브로 **경계 상태**(① 첫 경기 전 0경기 ② 시즌 막날 ③ 오프시즌 진입)에서 전 탭을 훑고,
  같은 상태를 그리는 **두 화면을 대조**한다(결과↔뉴스, 순위↔성적, 박스↔관전). 발견 시 ⓐ 헤드리스 가드로 환원
  (`_dv_newsday0`: 경계 컷오프에서 측정 + A/B) ⓑ **컷오프/배선 형제 사냥**(아래).
- **환원 짝 — 컷오프 차원 가드 + 배선 형제 사냥**: "치른 경기까지만" 같은 **공통 컷오프 헬퍼**(`leagueDisplayDay`/
  `playedThroughDay`)를 쓰는 화면이 늘면, **새 파생 표면이 같은 헬퍼를 쓰는지** grep으로 전수 확인한다(raw
  `currentDay`를 day-컷오프 셀렉터에 넘기는 곳 0건). 헤드리스 테스트는 **MAX만이 아니라 경계 컷오프(−1·막날)**
  에서도 한 번 더 돌려, "진행 차원"을 검사축에 넣는다. (교훈: 같은 버그를 한 곳에서 고쳤으면 — standings의
  playedThroughDay — 그 원칙을 **자동 형제 점검**으로 전파해야 한 칸이 안 샌다. 수동 규율은 샌다.)
- **왜 강한가**: 슬라이스 도구가 각자 PASS여도 *배선·경계·화면 간 모순*은 비어 있다. 사람의 눈+진짜 앱이 그 셋을
  동시에 본다. 단 육안은 재현이 약하므로 **반드시 헤드리스 가드로 환원**해 회귀를 막는다(육안=발견, 가드=상설).

### L. 문서 조항 대조 + 서버 5렌즈 (backend doc-contrast) — 2026-07-06 신설
**무엇**: 서버(백엔드) 표면을 정본 문서(`BACKEND_SYSTEM.md §들`)와 **조항 단위로 대조**하고, 엔진 지향
기법이 못 보는 **서버 보안/격리 불변식**을 아래 5렌즈로 겨눈다. §B(독립 문서기반)와 짝이지만 축이 다르다 —
**B는 엔진 수치**(분포·확률·범위) 지향, **L은 서버 라우트/보안 불변식**(인증·격리·타임존·머니패스) 지향.
- **서버 5렌즈**: ⑴ **인증 귀속** — `resolveUserId`(익명 폴백) vs `requireUserId`(귀속 강제)를 라우트별
  대조, "특정 유저 귀속" 라우트가 폴백이면 버그 ⑵ **proj 스코프 4메서드 전수** — POST/GET/PATCH/DELETE
  하나라도 스코프 누락이면 격리 뚫림 ⑶ **상한의 단위** — 문서 캡 문구(호출당/일일/평생)와 구현 단위 일치
  + 보상 테이블 총합↔캡 정합 ⑷ **date-only 타임존** — `'YYYY-MM-DD'` 입력 라우트는 `normalizeEndsAt`
  (`server/lib/dates.ts`) 필수, 형제 사냥은 "같은 입력 형태(date-only)를 받는 전 라우트" ⑸ **관찰 채널의
  머니패스 오염** — 알림·로그·flush는 `afterSafe`(머니패스 catch 밖·throw-none), 돈 이동 후 응답을 못 바꾸게.
- **실적(2026-07-06 1일 14건)**: afterSafe 1(결제 관찰 채널 500) · 공지 6(DELETE proj 미스코프+허위 ok·
  KST 9시간·재열람 미기록 등) · 쿠폰 4(익명 폴백 C1·KST 형제 C2 등) · 업적 3(평생합 백스톱 0·카탈로그
  총합>캡·비대칭). 전부 기존 장치를 통과한 채 잠복 — 서버 표면을 보는 렌즈가 없었다.
- **왜 강한가**: A~K는 전부 엔진·보드·앱 지향이라 서버 라우트의 인증 귀속·멀티게임 격리·타임존·머니패스
  응답 오염은 통째 사각이었다. L은 그 5축을 조항 대조로 친다.
- **도구/방법**: **`backend-verify` 스킬**(`.claude/skills/backend-verify/`) — 파이프라인(조항 추출→표면
  전수→5렌즈→심각도표→Agent 위임 수정→가드 재실행→§4 등재) + 상설 라이브 가드 배터리. 가드는
  `server/tools/_dv_*.ts`(라이브 dev DB·프리픽스 정리·A/B) — "검증 후 삭제" 금지, README 배터리 등록이 완료 조건.

### M. 명세 대조 발견 7기법 (spec-contrast discovery) — 2026-07-06 신설
**무엇**: **회귀 배터리(§0)가 못 보는 새 버그를 캐는 발견 프로토콜.** 기대치를 코드가 아니라 **정본
문서에서 매번 새로 도출**해 구현·라이브에 들이댄다(코드→코드 자기일관이 아니라 문서→코드). 2026-07-06
하루 6라운드에서 기존 배터리가 0건일 때 20건을 발견한 방식을 7기법으로 추출(발견·추출=Fable 5). 각
기법은 그날의 **실제 발견을 앵커**로 든다(창작 금지 — 앵커가 없는 기법은 M이 아니다):
- **① 조항 추출→실측**: 문서의 동작 주장을 명세 목록으로 뽑아 코드·라이브에 대조한다. — "Bearer→userId
  귀속"(§13.17)을 쿠폰 redeem에 들이대 **익명 폴백 C1** 발견.
- **② 주장 수치 재측정**: 문서의 숫자·%는 사실이 아니라 **가설** — 반드시 다시 잰다. — "평생합 캡 5000"
  vs 카탈로그 실측 **16,220**(업적 A2) / 룰55 "100%" vs 실측 **94.0%**(리시버 회귀).
- **③ 대칭 격자 전수**: 같은 클래스 표면(라우트 4메서드·earn reason·가드 파일)을 격자로 놓고 한 속성을
  전수 대조 — **하나만 빠진 게** 버그다. — DELETE만 proj 미스코프(공지 F1) / achievement만 누적 백스톱
  없음(A1) / `_ev_` 9개만 exit 미배선(R1).
- **④ 단위·의미 정독**: 문구의 단위·시점·조건을 **문자 그대로** 요구한다("평생합"≠호출당, "매 부팅"≠
  활성-있을-때만, "검증 후 삭제" 정책 자체를 의심). — 업적 A1 / 공지 prune 오프라인 오삭(F3) / 임시 E2E 삭제 정책.
- **⑤ 중간 상태 낀 경계 재현**: 멱등 재시도를 *사이에 다른 거래를 끼고*, 캡을 *경계 직전까지 채우고*,
  화면 재진입을 *상태 변화 후에* 때린다. — 지갑 stale 잔액(welcome→camp→재진입) / 평생합 19,900→+100 경계.
- **⑥ 가드 감사(verify the verifier)**: 배터리의 exit만 믿지 말고 가드의 **출력 텍스트·수치**를 읽어
  문서 주장과 대조하고 ✅/❌ ↔ exit 일치를 확인한다. — ❌ 출력인데 exit 0(`_ev_recvmatch`, 12일 잠복) 발견.
- **⑦ 작성자≠검증자**: 구현 컨텍스트 없이 **정본 문서만 들고** 대조(공유 가정 차단), 수정·문서는 다른
  모델(Agent opus)에 위임 후 메인이 재검증. — 6라운드 전부([[verification-model-attribution]]).

**기법 L과의 관계**: L(서버 5렌즈)은 **M의 서버 인스턴스**다 — M이 상위 발견 방법론이고, L·`backend-verify`
스킬은 그 M을 서버 라우트(인증 귀속·proj 격리·타임존·머니패스)에 특화한 것. 서버를 벗어난 표면(보드·엔진·
클라·가드 자신)엔 M을 직접 적용한다.
**D·B와의 차이**: D(육안 관전)는 사람 눈이 1차 발견(재현 약함 → 반드시 가드로 환원)인 반면, M은 문서
조항을 축으로 한 체계적 발견이다. B(독립 문서기반)는 **엔진 수치**(분포·확률·범위) 도출 지향이고, M은
그보다 넓다 — 조항 추출·대칭 격자·가드 감사·단위 정독까지 포함하는 **발견 프로토콜**(B는 M의 한 갈래).
**실행 스킬**: `spec-audit`(`.claude/skills/spec-audit/`) — 임의 시스템을 발견 모드로 감사하는 M의 실행 절차.
**왜 강한가**: 회귀 배터리는 인코딩된 불변식만 재확인해 *불변식에 없는* 버그를 구조적으로 못 본다. M은
기대치를 매번 문서에서 새로 길어 올려 그 사각을 친다 — **"배터리 올그린인데 불안"할 때 돌리는 축**이다.

## 1.5 어디에 버그가 모이나 — 게임-상태 렌즈 (공격)

§1이 *어떻게* 찾나(기법), §4가 *왜 놓쳤나*(테스트 사각, 수비)라면, 이건 *어디를 먼저 뒤지나*(버그가
모이는 자리, 공격)다. 상태가 시간에 따라 변하는 이 게임에서 버그는 아래 5렌즈에 쏠린다 — 퍼저·검사를
이 순서로 겨눈다(앞쪽일수록 광맥이 굵다).

| 렌즈 | 한 줄 | 볼리볼 실제 사례 |
|---|---|---|
| **① 시간차** | 행동 *시작*과 *결과 확정* 사이에 상태가 변함(스냅샷 stale) | 면담→오프시즌 OwnerFx, 벤치 fromDay↔시즌말 분노 정산 |

## 2. 버그 발견 시 후속 프로토콜 (필수 5단계)

새 버그를 찾으면 **고치고 끝내지 않는다.** 다음을 전부 한다:

1. **재현 고정**: 시드+입력 시퀀스로 재현하는 최소 하네스를 남긴다(`tools/_gt_repro_*` 등).
2. **왜 이전 테스트가 못 잡았나 (사각 분석)**: 기존 도구 코드를 읽어 **어느 사각**이었는지 규명.
   사각 분류 → §4. 이걸 케이스 레지스트리(EDGE_CASES/BOARD_RULES)에 **"왜 못 잡았나"로 기입**.
3. **케이스 등록 + 가드**: 레지스트리 §3에 `증상→원인→수정→잡는 도구` 행 추가 + 그 클래스를 잡는
   상설 가드를 감사/도구에 심는다(A/B로 검증). **수정 후엔 좁은 repro만이 아니라 넓은 퍼저(여러 시드)를
   다시 돌린다** — 좁은 테스트는 내가 *상상한* 공격만 보지만, 무작위 퍼저는 불완전 수정의 변종을 잡는다
   (EC-TX-04: 음수 연봉만 막은 1차 수정이 좁은 repro는 통과했으나, 확장 몽키가 거대 양수 연봉으로 캡을
   뚫어 재수정 필요를 드러냄).
4. **동종 버그 사냥 (sibling hunt)**: 그 버그를 찾은 **렌즈를 코드 전체에 적용**한다. 예) "store 액션이
   id를 받아 멤버십 검증 없이 변이" → 전 액션을 훑어 같은 클래스를 색출(release가 유일 예외였고,
   signInSeason/replaceForeign/replaceAsian/toggleProtect는 모두 가드 보유 → 잔존 0 확인).
5. **방법 기록**: 어떤 **발견 기법**으로 찾았는지 본 문서(§1)에 기법이 없으면 추가. 케이스(무엇)와
   방법(어떻게)을 **따로** 남긴다.

## 4. 테스트 사각 분류 (왜 놓치나 — 진단 체크리스트)

새 버그의 "왜 못 잡았나"를 적을 때 아래 중 무엇인지 분류한다:

| 사각 | 설명 | 닫는 법 |
|---|---|---|
| **계층 우회** | 데이터/리플레이층만 검사, **사용자 진입점(store 액션) 미검사** | store 구동 퍼징(A) |
| **단일 국면** | 각 국면을 따로 검증, **교차 상호작용**(시즌중 churn × 오프시즌 충원) 미검사 | 장기 원숭이(A) |
| **valid 편향** | 정상 입력만, **적대 입력**(타팀 id·음수·NaN) 미주입 | 악질 퍼징(A) |
| **허위 오라클** | 검사가 코드 의도를 베껴 **같이 틀림**(세터 면제 등) | A/B 자가검증(C) + 독립 검증(B) |
| **재계산 좌표** | 그려지는 실제 값이 아니라 **재계산한 값**으로 검사 | 실제 출력으로 검사 + A/B(C) |
| **재구현 오라클** | 테스트가 검사 대상(partialize·rehydrate 등)을 **직접 베껴** 검사 → 진짜 설정이 바뀌어 필드 누락돼도 못 잡음 | **실제 코드 경로** 호출(`useGameStore.persist.getOptions()`) + A/B(필드 누락 주입 검출) |
| **귀속 사각(cross-layer)** | 두 계층(보드/박스)이 같은 이벤트의 *선수*를 각자 골라 — 각 계층 내부는 PASS인데 **둘이 다른 선수**(화면≠기록). 단일 계층 검사로는 통째로 안 보임 | **교차 계층 귀속 일치 검증(§1.J)** — 두 계층의 같은 이벤트 선수 id 추출·일치율 측정 + A/B(shuffle) |
| **정적 시드 데이터 교차 불변식 미검사** | 손으로 적은 시드 데이터(구단 정체성 recentRanks·strengthBias 등)가 **항목별로는 그럴듯하나 교차 제약**(한 시즌 순위=1..N 유일 순열·bias 합=0)을 깬다. 단일 항목만 보면 안 보임 — 아키타입별 따로 작성돼 열로 모으면 중복(이전 시즌 순위 1×2·7×3, 2026-06-24 사용자 보고) | **데이터를 매트릭스로 모아 교차 검사**(`checkClubRanks` — 시즌 열별 순열·bias 합) + A/B(현 데이터 검출→수정 후 PASS). 표시 전용 flavor라도 내부 정합(현실 리그처럼) 가드 |
| **이중 계산(parallel attribution)** | 같은 per-player 통계를 **두 모듈이 독립 난수로 따로** 산출(스코어박스 `rally.ts` box ↔ 통산/연봉 `production.ts`) → 각자 KOVO 팀합엔 맞아 단일 검사는 통과하나 **개인은 22~62% 분기**(N=2,000, 2026-06-24). "본 기록≠쌓는 기록" | **두 소스 교차 측정**(`_ev_statsource` — 같은 시드로 box·prod 둘 다 뽑아 선수별 \|차이\|) + A/B(box vs box=0). 팀합만 보면 못 잡음 → **개인 단위** 대조 필수. 해소 = 단일 진실(box 집계, 선수별 0 분기) |
| **부분 기록(partial event emission)** | 한 클래스 사건(디그=att-flip 전환)을 엔진이 **일부만** 기록 — 클린 디그만 box/터치, 팁-dug·소프트블록 전환은 누락 → 보드는 전 전환을 그려 **사이드 드리프트**(2b 디그 88.5%서 막힘, 첫 드리프트 h0부터). "터치 일부 누락"은 분포는 비슷해도 per-event를 깬다 | **순서 정렬 + 드리프트 분해**(`_ev_digmatch` — 사이드어긋남/six없음/다른선수 분리, 첫 드리프트 hop) → 누락 사건 위치 규명. 해소 = 엔진이 **모든 동종 전환에 터치 발행**(팁·소프트도) → 100% |
| **루틴 누락 가드(orphan guard)** | exit 0/1로 **멀쩡히 작동하는 가드**(simFinance 등)가 `docs/README.md` "검증 루틴"에 **안 묶여 어떤 "전체 테스트"에서도 안 돈다** → 회귀가 쌓여도 아무도 안 봄. 재정 회귀(net −2.87억·보전 50%)가 06-14~06-27 내내 누적됐는데 simFinance가 루틴 밖이라 미발견(2026-06-27, engine-verify 문서_drift→형제사냥). 가드 작성≠회귀 묶음 등록 | **형제 사냥**: `grep -rl process.exit tools/*.ts` ↔ README 대조로 orphan 전수 → 진짜 회귀 가드(건강·무결성)는 루틴 등록(2026-06-27 12종 추가). 새 가드는 만든 즉시 README 검증 루틴에 한 줄(run-all-tests 스킬 §3-B) |
| **밸런스 드리프트(invariant blind)** | 불변식 가드(한사람=한팀·NaN0·결정론)는 **구조는 멀쩡한데 밸런스만 나빠진** 회귀를 못 본다 — 게임은 정상 작동, 재정만 만성적자/스탯만 무효. 또 타 시스템 변경(구단 정체성→payroll↑)이 **다른 시스템(재정) 기준선을 조용히 무효화**(STATS_PROTOCOL 재측정 트리거가 시스템 경계 못 넘음) | **분포/건강 임계 가드**(simFinance·simStatEffect 등 exit 0/1 self-judge)를 루틴에 상설. 엔진 변경 시 *영향권 밖* 시스템도 재측정(특히 경제·생산). 도구도 의심 — simStatEffect 대조군 id-편향처럼 가드 자신의 baseline drift를 A/B(대조군 ~50%)로 점검 |
| **귀속 추상화(attribution abstraction)** | 박스(엔진) 귀속 자체가 **통계 추상**이라 코트 현실과 1:1 대응 안 됨 — (구)디그=best-dig(≈리베로)에 전부, 어시=지정 세터에 전부(세터가 디그해도). 보드(현실 안무)와 **세는 사건이 다름**. "배선"으로 못 맞춤 — 강제하면 수비홀·워프(2026-06-23 규명). **디그는 엔진 귀속 재모델로 해소(2a, 2026-06-24)** — best-dig 독식→가중 분산(리베로 1위·12명), 보드는 2b에서 재생. **어시(setId 잔여 14%)는 미해소**(같은 뿌리) | **분포 진단**(`_ev_digdist`)으로 추상화 노출 → 배선 가능(recv/set 종결)과 모델재설계 필요(dig 완료·set 잔여)를 **구분**. 해소는 엔진 귀속 모델 변경(승패 불변·KOVO 재검증) — 배선 시도 전에 분포부터 본다 |
| **상수 그룹 손복제(constant re-listing)** | 한 코드 상수(`engine/aging.ts DECAY_STATS`)가 정의한 **그룹 멤버십**을 여러 문서 표가 **각자 손으로 다시 나열** — 한 표(`TRAINING §1.6`)는 `agility`를 "노쇠 없음"으로 오기, 형제 표(`STAFF §1`)는 코치 노쇠둔화 대상에서 `agility` 누락. 같은 문서 내 다른 표(§1.3·§0)·`CLAUDE.md 5.1`·`aging.test.ts`는 정확 → **단일 표만 보면 모순 안 보임**. 단위 테스트(`aging.test`)는 *동작*(민첩 하락)은 검증했으나 *문서 표의 분류*와는 대조 안 함(2026-06-25) | **문서 enum을 코드 상수에서 직접 읽어 대조**(`_dv_drift2_agility` — 노쇠 그룹을 `DECAY_STATS`로 실측, A/B로 비노쇠 대조군 반응·위치선정 Δ0 확인). 손으로 적은 그룹 나열은 **그 그룹을 정의한 상수와 1:1 대조하는 가드**를 건다 |
| **병렬 재구성 드리프트(parallel reconstruction)** | 같은 논리 집합을 **두 함수가 같은 원천(txLog)에서 따로 재구성** — authoritative forward-pass(`applyTx` faAvail)는 `!isForeign` 가드가 있는데 on-demand 셀렉터(`availableFAsOnDay`)는 누락 → 방출 외인이 FA 풀로 샘(2026-06-25, 코드세션). 각 함수 내부는 자기완결이라 단독 검사는 통과. 정상플레이로 도달(`replaceForeign`이 release tx를 남김), 소비처 `signInSeason`은 멤버십만 보고 재영입 허용(익스플로잇) | **두 재구성을 같은 입력에 돌려 출력 교차 대조** + **소비처(signInSeason)까지 구동**해 실제 행동 검증(`_dv_foreign_fa_leak` — release 후 외인 미포함·재영입 거부, A/B로 구 전부-add 로직 검출). 동일 집합을 두 곳서 만들면 **한쪽을 정본으로 고정하거나 둘을 대조하는 가드** 필수 |

| **처치 효과를 비쌍대조로 측정(non-paired treatment comparison → confounded)** | 방침/처치 효과(감독 trainingFocus 등)를 **비쌍대조**(집단 간 *상이한 스탯*끼리 비교 — "감독 primary 주스탯 Δ vs 전스탯 평균 Δ")로 재면 스탯별 **자연 성장률 교란**(posRelevance·나이배수·XP 양자화)이 처치 신호를 덮어 **위음성·위양성** 둘 다 낸다(훈련 방침 1차 오판, 2026-07-07). 형제 함정: **완성 선수 표본만** 보면 기술 방침이 죽은 것처럼 보임(천장=포텐−GAP 도달, WAI — TRAINING §1.9) | **처치 효과는 동일 개체 쌍대조로**(같은 선수·같은 rng를 두 방침으로 각각 키워 *같은 스탯*의 Δ를 서열 비교 — `_dv_focus` (b)(c)) + 헤드룸 살아있는 표본(신인)을 반드시 포함. 자연 성장률이 다른 스탯끼리 비교 금지 |
