## 기록 형식

```
### R{n} — {날짜} {국면 요약}
- 관찰(사용자): 무엇이 어색했나 (실험실 스텝·좌표 포함)
- 원인: 코드의 어떤 로직/상수 때문인가 (파일:라인)
- 수정: 무엇을 어떻게 바꿨나
- 형제 적용: 같은 클래스 원리로 함께 고친 다른 국면/역할/좌표 목록 (없으면 "없음")
- 재검수: ✅통과 / ❌반려(사용자 재지적)
- 오답 분석(반려 시): 왜 잘못 고쳤나 — 어떤 가정이 틀렸나
```

## 형제 사냥 규칙 (2026-07-18 사용자 확정)

1. **자동 실행**: 회차 수정 때마다 같은 클래스의 다른 국면·역할·좌표를 능동 수색해 함께 고친다
   (TEST_METHODOLOGY §2 5단계). 단 **"형제 적용" 필드에 전파 대상을 반드시 남긴다** — 이게 이력의 핵심.
2. **형제 적용분이 반려될 때**: 사용자가 지적한 위치가 과거 회차의 형제 적용분이면,
   ① 원회차(R{n})의 판정 근거를 회고하고("그때 왜 이 원리를 여기에도 적용했나")
   ② 어떤 가정이 이 국면에선 안 맞았는지 **오답 분석을 원회차와 신회차 양쪽에 기록**한 뒤
   ③ 재수정한다. — 원리 자체가 틀렸으면 형제 전체 재검토, 이 국면만 예외면 예외 사유를 남긴다.
3. 목적: 수정→전파→반려→회고 사이클로 "어떤 원리가 어디까지 일반화되는가"의 데이터를 쌓는 것.
