# FA·세대교체 시스템 설계 (FA_SYSTEM)

> CLAUDE.md 7장(단장 의사결정)·6장(누적 서사)을 확장. SALARY_SYSTEM과 맞물린다.
> 목표: KOVO 여자부 FA의 "그 느낌" — **등급제 + 보상선수/보호명단 + 원소속 우선협상(프랜차이즈)
> + 샐러리캡**. 모든 수치는 placeholder(튜닝 대상).

**실제 상수(문서 예시값과 차이):**
- 샐러리캡 `LEAGUE_CAP=350000`(**35억**, 문서 예시 30억), 개인상한 `MAX_SALARY=80000`(8억),
  프랜차이즈 `FRANCHISE_SEASONS=6`·`FRANCHISE_MAX=110000`(11억).
- 자격: `FIRST_FA_SEASONS=6` — `career.seasons≥6 AND contract.remaining≤1`. (문서 "이후 3시즌마다"는
  현재 계약 만료 기준으로 단순화.)
- 등급: 연봉 순위 **상위 35% A / 다음 35% B / 나머지 C**.
- 요구연봉 프리미엄 `PREMIUM = A 1.25 / B 1.15 / C 1.1`(2.2표의 보상 배수와는 별개).
- 보상금 `compensationMoney = A 2.0× / B 1.0× / C 0`, 보상선수는 **A·B만** 필요.
- `offerScore = 0.35·연봉비 + 0.30·팀전력 + 0.20·출전기회 + 0.15·충성도 + 0.05·rand`.
- AI 잔류 판단 `aiKeepsFA`: 나이<32 AND OVR≥70. 로스터 이상치 `ROSTER_IDEAL{S3,OH5,OP2,MB4,L2}=16`.

> 6장 데이터 모델(`clubTenure`/`career.seasons` 등)은 반영됨. 7장 모듈 매핑의 `leagueState.ts`는
> 실제로는 `data/league.ts`(가변 로스터·레지스트리·`baseVersion` 캐시 무효화)로 구현.

## 1. 단계 ① — 선수 생애 토대 (선행)

FA가 돌아가려면 선수가 빠지고(은퇴) 채워져야(유망주) 한다.

### 1.1 경력 누적
- `career.seasons` 를 롤오버마다 +1. 시드 생성 시 베테랑은 `career.seasons ≈ age - 19`(데뷔 추정).
- FA 자격 타이머의 기준.

### 1.2 은퇴 (retirement)
- 롤오버 시점 확률적 은퇴. 나이·능력(OVR)·출전 기회에 의존.
  ```
  retireChance(age, ovr):
    age < 30 : ~0
    30~32    : 낮음 (능력 낮을수록↑)
    33~35    : 중간
    36+      : 높음 → 38~40 사실상 은퇴
    저능력(벤치) 노장은 더 빨리
  ```
- 은퇴 선수는 리그에서 제거 → **로스터 공백** 발생(통산 기록은 보존 — 6장).

### 1.3 유망주 풀 (rookie pool)
- 매 시즌 신규 유망주 N명 생성(결정론 시드 + 시즌번호). 나이 18~20, 포텐 높고 현재 낮음.
- ③ 드래프트로 팀에 배정(그 전까지는 자동 충원으로 로스터 유지).

### 1.4 로스터는 이제 가변 (영속)
- 은퇴/영입/드래프트로 팀 구성이 바뀐다 → **팀 로스터를 세이브에 영속**.
- 선수도 신규 생성되므로 **전역 선수 레지스트리**(시드 + 생성분)로 통합. id로 항상 조회 가능.

## 2. 단계 ② — FA 코어

### 2.1 자격 공시
- **최초 FA: 6시즌, 이후 3시즌마다.** 시즌 종료 시 자격 선수 공시 → FA 시장 등장.
- 잔여 계약이 남았으면 비자격(계약 만료/자격 충족 시만).

### 2.2 등급 A/B/C (직전 연봉 순위 → 보상 크기)

| 등급 | 기준(연봉) | 타 구단 영입 시 보상 |
|---|---|---|
| **A** | 상위 | 직전연봉 **300%** (보상선수 X) **또는** **200% + 보상선수 1명** |
| **B** | 중위 | **200%** (보상선수 X) **또는** **100% + 보상선수 1명** |
| **C** | 하위 | **보상 없음** |

> 영입 구단이 두 옵션 중 선택(보상선수 줄지/돈 더 낼지).

### 2.3 보상선수 + 보호선수 명단 ★
- 보상선수 옵션 선택 시: **영입 구단이 보호선수 명단(예: 6명) 제출**.
- **원소속 구단이 비보호 선수 중 1명 지명** → 데려감.
- 단장 결정: "누구를 보호하나"(영입 가치 vs 잃을 위험). AI는 비보호 중 최고 가치 선수를 집어감.

### 2.4 원소속 우선협상 / 프랜차이즈 ★
- **우선협상 기간**: 시장 오픈 전, 원팀만 먼저 재계약 시도.
- **프랜차이즈 스타**(한 팀 장기근속, 예: 그 팀에서만 ≥6시즌):
  - 재계약 시 **샐러리캡 일부 예외**(충성 프리미엄),
  - **충성도 보너스**로 잔류 확률↑,
  - 타 구단 영입 시 **보상 가중**(한 등급↑ 취급 등 — 데려가기 더 비쌈).

### 2.5 협상·수락 로직 (선수가 고른다)
```
asking = marketValue × FA프리미엄(1.1~1.3)
offerScore = wSal·연봉/asking + wYrs·계약연수 + wWin·팀전력
           + wPlay·출전기회(포지션 뎁스 역수) + wLoyal·충성도(원소속·프랜차이즈) + wAge·나이적합
선수는 최고 offerScore 수락. 동점·근소차는 원소속 우선.
```

### 2.6 FA 센터 UI (실제 하는 느낌)
시즌 종료 → **FA 센터**:
1. **내 FA 자격 선수**: 우선협상·재계약 / 떠날 위험 경고
2. **FA 시장 보드**: 등급별 목록 · asking · 경쟁 구단 · 예상 보상
3. **영입**: 오퍼(연봉·연수) → (보상선수 옵션 시) **보호명단 제출** → 선수 수락/거절 → **보상 처리**
4. **결과 요약**: 이적/잔류, 보상선수 이동, 캡 현황

## 3. 단계 ③ — 신인 드래프트
- 유망주 풀에서 **역순 지명**(하위 팀 우선) 또는 추첨(로터리).
- **스카우팅**: 포텐/일부 스탯만 공개(7장). 픽 가치 판단.
- 인간 단장 지명 + AI 지명. 미지명 유망주는 일부 잔류(다음 시즌 풀) 또는 소멸.

---

## 4. 단계 ④ — AI 구단
- AI가 자동으로: 자격 FA 재계약 / 타팀 FA 경쟁 입찰 / 보호명단 제출 / 보상선수 지명 / 드래프트 픽 / 은퇴 공백 충원.
- AI 성향(공격/수비/밸런스·예산)에 따라 영입 우선순위 차이 → 리그가 살아 움직임.

---

## 5. 샐러리캡 (단계 ② 함께)
- **팀 캡**: 총연봉 ≤ CAP(예: 30억). 초과 영입/재계약 불가.
- **개인 최고연봉 상한**(예: 8억). 프랜차이즈는 예외 한도(+α).
- (선택) 최저연봉. 외국인은 캡 포함 or 별도 — 우선 캡 포함 단순화.
- 기존 느슨한 `Team.budget` → `Team.cap` 으로 정식화.

## 6. 데이터 모델 추가 (예정)
```ts
interface Player {
  // career.seasons 누적 사용
  teamId: string | null;        // 소속(없으면 FA/유망주)

interface LeagueState {          // 세이브 영속(시드 위 오버레이)

  registry: Record<string, Player>;       // 신규 생성 선수 포함
  faPool: string[];                       // 공시된 FA
  rookiePool: string[];                   // 드래프트 대기 유망주
}
```

## 7. 엔진 모듈 매핑 (순수·SOLID)
```
/engine
  retire.ts     은퇴 판정(나이·능력)
  rookies.ts    유망주 생성(시즌 시드)
  faMarket.ts   자격·등급·asking·offerScore·수락 판정
  compensation.ts 보상금·등급·보상선수 후보(보호명단 외)
  draft.ts      지명 순서·AI 픽
  aiGM.ts       AI 구단 의사결정(재계약/입찰/보호/지명)
/data
  leagueState.ts 가변 로스터·레지스트리·풀 셀렉터(저장 오버레이)
```
> 각 모듈 단일 책임. UI→data(셀렉터)→engine(순수). FA를 바꿔도 경기/훈련/생산 불변.

## 8. 밸런싱 체크리스트
- [ ] 6시즌/3시즌 자격이 맞게 공시되는가
- [ ] A/B/C 등급·보상이 영입 결정을 실제로 무겁게 하는가
- [ ] 보호명단 선택이 "알짜를 뺏길 위험"으로 긴장을 주는가
- [ ] 프랜차이즈 스타가 잘 안 떠나고, 떠나면 충격적인가
- [ ] 캡이 영입을 제약하고, 프랜차이즈 예외가 의미있는가
- [ ] 은퇴·드래프트로 로스터가 무한정 늙지 않고 세대교체되는가
- [ ] AI 구단이 합리적으로 재계약·입찰·지명하는가
- [ ] 결정론: 같은 시드·같은 단장 선택 = 같은 결과
