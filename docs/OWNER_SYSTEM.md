## 0. 왜 넣는가 (서사적 목표)

- 지금 "우승 야망 선수가 하위권 팀을 떠난다"는 FA 수식 안에서 **조용히** 일어난다.
  플레이어는 시즌말 명단에서 빠진 이름을 보고서야 안다. 드라마가 수식 속에 숨어 있는 것.
- 면담은 그 수식을 **장면**으로 꺼낸다: "구단주님, 저는 우승이 하고 싶습니다." — 여기서
  무엇을 약속할지, 약속이 통할지, 실패하면 마음이 더 떠날지. 의사결정의 무게(기둥 4).
- 감독 건의·팬심은 그 결정의 **파장**이다. 선수 하나를 벤치에 앉히는 결정이
  감독과의 신경전 → 팬 분노 기사 → 관중 감소 → 예산 타격 → 선수 이탈로 번진다.
  관전형 게임에서 "보는 맛"은 경기만이 아니라 이런 연쇄에서 나온다.

## 1. 선수 면담 (Interview)

### 1.1 불만은 이미 존재한다 — 새로 만들지 않는다
선수마다 FA 성향(`faPref`: money/winnow/loyal/minutes/hometown)이 이미 있다.
면담 시스템은 성향과 현실의 **불일치**를 불만으로 표면화한다:

| 성향 | 불만 조건 (파생, 저장 없음) | 면담에서 하는 말 |
|---|---|---|
| winnow(우승) | 팀 최근 2시즌 평균 순위 5위 이하 | "우승권 팀에 가고 싶습니다" |
| minutes(출전) | **벤치 사유 × 성격 × 주전 기대치**(아래 1.1b) | "뛰고 싶습니다" |
| money(연봉) | 연봉 < 시장가치 × 0.75 | "제 가치를 인정받고 싶습니다" |
| hometown(연고) | 선호팀이 따로 있음 | "○○에서 뛰는 게 오랜 꿈이었습니다"(선호팀 이름 표기 — 2026-06-30) |
| loyal(충성) | 거의 불만 없음 | (면담 요청 시 "괜찮습니다, 구단주님") |

**선수 상세**에 **불만 뱃지**(😟)를 표시 — 면담할 이유를 플레이어가 발견하게. (스쿼드 목록엔 비노출 —
명단은 이름·기본 상태만, 감정은 상세에서. 2026-06-30 사용자 요청.)

### 1.1b 출전 불만은 "사유 + 성격 + 기대치"로 (2026-06-22 — ROTATION_MORALE B+C 구현)
출전율만 낮으면 무조건 불만이던 것을, **왜 못 나오는지**와 **선수가 어떤 사람인지**로 분기했다.
- **벤치 사유**(`benchCauseOf`): 부상·징계 결장 → 불만 없음(구단 탓 아님, 묵묵히 복귀/자숙) · 구단주 벤치 →
  불만(부당) · 실력 밀림 → 절반(투지/수긍) · 휴식(#3, 추후) → 양해.
- **성격**: 출전 갈망(`prefWeights.play`)이 높을수록 같은 벤치에도 강하게 반응. 무던한 선수는 감내.
- **주전 기대치**(`expectsPlayOf`): 동포지션 최약 주전과의 OVR 격차+경력. **OVR 낮고 경력 짧은 후보는 못 나와도
  불만 없음**("아직 부족하지"), 에이스가 벤치면 부글부글. → 출전 불만 = `w.play × 사유 스케일 × 기대치`.
- **기분 3종**(`moodOf`): 불만(😟)/무감정(😐)/긍정(😊, 주전+상위권). 선수 상세 "지금 마음" 한 줄로 노출.
- 엔진 `discontentOf`(`engine/owner.ts`)·셀렉터 `discontentNow`(`data/owner.ts`). 파생·무저장. 검증 `tools/simMood.ts`.
- **★ 성능(2026-07-07)**: 프리게임(`day≤0`, 구단 선택·온보딩 `currentDay=0`) `discontentNow`는 경기가 0이라 벤치 상황
  자체가 없다 → 시즌 시뮬(`computeStandings`/`leagueProduction`)을 **부르지 않고 중립 상수**(`cause:'starter'`, topic null,
  mood neutral)로 즉시 반환한다. 이전엔 조기반환이 `benchCauseOf`를 호출했는데, 그게 `restedOnDay→computeStandings(0)`을
  전이적으로 타 **온보딩 첫 선수 상세가 콜드 시즌 시뮬(폰 ~15s)로 freeze**했다. 이 콜드 비용은 워밍 후 측정하면 캐시에
  가려 안 보인다(0.28ms 허위 PASS) → **콜드 측정**(`resetLeagueBase()` 직후 워밍 없이 첫 호출)으로만 드러났고, 수정 후
  콜드 첫 호출 0.13ms. 근거: TEST_METHODOLOGY §사각분석 "성능 fix 검증이 워밍 후 측정으로 false PASS".
- **누적→FA 이탈(C.4)**: 시즌 내내 부당하게 벤치에 묵힌 선수(낮은 출전율 × 출전 불만)는 정 떨어져 **재계약 거부↑ → FA**.
  `sustainedBenchRefuse`를 `buildOwnerFx` 거부 확률에 가산(출전율=묵힌 기간). 부상 결장은 무관(불만 아님).
- **재계약 거부는 명시 재계약(인시즌 override)도 대상(2026-07-10 봉인)**: "단장이 잡아도 선수가 떠날 수 있다"는 오프시즌 만료 재계약뿐 아니라
  **인시즌 재계약(계약 관리 화면 '재계약')에도 적용**된다. 이전엔 override가 `refuses()`를 우회해 "재계약 확정=100% 잔류"였던 실버그를 봉인
  (override 보유 만료자도 거부 롤). 불만 선수는 명시 재계약도 뿌리치고 FA로 나갈 수 있다. + '후하게'는 **연봉 불만**만 실제로 푼다(우승·출전의 마음은 면담·기용). 정본 FA_SYSTEM §2.5c-보완.
- 상세 설계(+ #3 휴식): [ROTATION_MORALE](./ROTATION_MORALE_SYSTEM.md).

## 2. 감독 건의 — "이 선수, 주전에서 빼주시죠"

### 2.1 왜 감독을 거치는가
구단주가 라인업을 직접 짜면 감독 시스템이 장식이 된다. **건의**로 만들면
감독의 성향·자존심이 게임 요소가 되고, 거절당하는 경험이 감독 교체 동기가 된다.

> **주전/벤치 표시(2026-06-18)**: 선수 상세 헤더에 `주전`(녹색)/`후보`(회색) 뱃지 —
> `buildLineup(availableTeamPlayers)`(경기 엔진과 동일 출전 라인업) 기준. 결장 사유가 있으면
> `부상 결장`·`출장 정지`·`벤치(감독 지시)` 우선 표시. 구단주가 선발/벤치 **건의 여부를 판단**할
> 핵심 정보(사용자 보고: 현재 기용 상태를 알아야 건의 결정).

### 2.2 판정 — 감독은 합리와 소신 사이에서 답한다
**벤치 건의** 수락 확률(`benchAccept`, clamp 0.05~0.95):
```
base 0.5
+ 0.3 × ovrGapT           // 대체자 합리성 — 벤치대상 vs 동포지션 최고 대체자 OVR 격차가 작을수록 1(수긍↑). 대체자 없으면 gap=10→0
− aceGuard                // 에이스 보호 — 팀 OVR 1위 −0.4 · 2위 −0.2 · 그 외 0 (2026-07-03 문서 정렬: 종전 "1~2위 −0.4"는 코드와 불일치였음)
− 0.2 × (charisma−50)/50  // 카리스마 높은 감독 = 소신
+ reasonT                 // 사유 타당성 — noResign(내년에 함께 안 함) +0.2 · form(폼 하락) +0.1 · prospect(유망주 기회) +0.05
```
**선발 기용 건의** 수락 확률(`startSuggestAccept`, clamp 0.05~0.95, 2026-07-03 문서화 — 종전 축약):
```
0.35 + 0.5 × gapT − 0.3 × (charisma−50)/50
   // gapT = clamp(1 − (overall(현주전 최약) − overall(건의선수))/10, 0,1). 건의선수가 주전보다 좋거나 같으면 1(만점)
```
시드: `bench:{id}:{season}:{day}` / `start:{id}:{season}:{day}`.

**★ Form 반영 — 비대칭(2026-07-03 결정, 독립리뷰+GPT+사용자 합의)**: 당초 "20% blanket 반영"(GPT C안)은 **철회**.
근거: Form(`applyForm`)은 **안 뛰는 선수의 sk*만 최대 7% 깎는 한 방향 감쇠기** → 두 건의 모두 항상 비교의 *약자*를 깎아
**건의 성공률을 낮추는 방향으로만** 작동한다(실제 최종 p 변동 ±0.02로 미미). blanket은 UX 개선보다 플레이어 도구 약화 리스크가 큼.
→ **비대칭 적용**(현실 감독 사고에 정합):
- **선발 기용 건의**: 건의 선수의 Form을 **보지 않는다**(순수 OVR = 원래 능력 평가). 플레이어는 *뛰게 해서 폼을 회복*시키려 건의하는 것 —
  자기 러스트로 자기 승격을 막는 자충수(악순환의 진원) 제거. → `suggestStart`는 **현행 순수 OVR 유지**(변경 없음).
- **벤치 건의**: 대체자(alt=백업)의 Form을 **고려한다**(지금 투입 가능한지). 폼 나쁜 백업 위해 주전을 빼는 건 무리 → 감독 거절이 정당.
  → `suggestBench`에서 `gap = overall(target) − overall(applyForm(alt, formFactorOnDay(팀,alt,day)))`. target(현 주전)은 폼≈1.0이라 순수 OVR.
- `overall()` **재정의 금지** — 콜사이트(store)에서만 blend/apply. 건의는 플레이어 전용(AI 시뮬 밖, 코드 확인)이라 **밸런스 회귀·세이브 마이그레이션 없음**.

**★ 거절 사유 반환 — 실제 감점량 랭킹(2026-07-03 결정)**: `benchAccept`/`startSuggestAccept`는 boolean 유지, **store가 거절 시 사유 파생**해 `{ok, reason}` 반환.
고정 우선순위 금지(가짜 인과) → **각 항의 실제 감점량을 계산해 최대 항을 사유로**:
- 에이스 보호 = `aceGuard`(0/0.2/0.4) · 실력차 = `0.3×(1−ovrGapT)`(벤치)·`0.5×(1−gapT)`(선발) · 감독 소신 = `max(0, 0.2×(charisma−50)/50)`.
- **p 게이팅(필수)**: 최종 p가 높았는데(≥0.55) 시드가 불리해 거절됐으면 구조적 원인 없음 → **"감독의 판단"**. p 낮을 때만 최대 감점 항 표기.
- **"운" 워딩 금지**: 시드(`bench/start:{id}:{season}:{day}`)는 결정론 — 그날 재시도해도 동일. UI는 "이날 감독은 마음을 바꾸지 않았습니다"류(재도전하면 바뀔 것처럼 호도 금지).
- reason은 **UI 반환용 ephemeral**(저장 안 함 — 재계산 가능). 반환 타입 변경 시 `app/player/[id].tsx` 2개 콜사이트 동시 수정(누락 시 객체 truthy→항상 수락 버그).

### 2.3 수락 시 — 벤치 지시
- `benchDirectives: { playerId, fromDay, toDay? }[]` 저장. **부상과 같은 패턴**으로 시즌 계층이
  "그날 출전 후보"에서 제외(로스터엔 남음, 페이롤 유지). 언제든 철회 가능(다음 미관전 경기부터 복귀).
- **건의 반영 시점 — 관전 중인 경기엔 미적용, 다음 경기부터(2026-06-28 결정, 옵션 A)**:
  ~~평소 `fromDay = currentDay`~~ · ~~이어보기 대기 중이면 `fromDay = currentDay + 1`~~
  → **정정(A2, 2026-07-08 — 소급 방어)**: `fromDay = max(currentDay + (이어보기 대기? 1 : 0), playedThroughDay(results) + 1)`.
  옛 식(`currentDay` 또는 `+1`)만으로는 **오늘 경기를 관전·기록한 직후(watchProgress 비움·setDay 전)** currentDay가 아직
  그 기록된 경기일이라 `fromDay=currentDay`가 **이미 관전·기록한 경기를 리플레이에서 소급 변경**했다(감사 5b: 리플레이 첫
  변경일 == fromDay == 기록된 경기일). `playedThroughDay+1` 바닥을 대 **기록된 경기일엔 절대 지시가 걸리지 않게**(forward-only,
  본 역사 보존). 관전 중 방어(이어보기 +1)만 있던 사각을 닫음. 플레이어 화면은 "관전 중인 경기엔 적용되지 않고 다음 경기부터" 알림.
  - **왜**: 경기는 매 진입 시 `availableTeamPlayers(dayIndex)`로 재시뮬(EC-UI-01). 관전 중(이어보기 대기)
    경기의 라인업을 바꾸면 ① 저장된 이어보기 지점이 *바뀐 경기*로 이어져 어긋나고(시간차/stale-resume —
    TEST_METHODOLOGY §4 ①) ② "질 것 같으면 나가서 선발 바꿔 리롤"이 가능해진다(관전형 = 결과는 정해진다).
    `fromDay`를 다음날로 빼면 그 경기는 시작 라인업 그대로(코히런트) + 변경은 다음 경기부터 — 현실(이미 시작한
    경기는 못 바꿈)과도 맞다. `watchProgress`는 다음 미관전 경기에만 생기고 건너뛰기가 없어(schedule
    `setDay(nextFixture)`) 스테일 항목이 없으므로 "비어있지 않음 == 현재 경기 관전 중" 단순 판정이 성립.
  - 가드: `tools/_ev_suggest_defer.ts`(이어보기 대기 시 fromDay=currentDay+1·미대기 시 currentDay·A/B — 기록 0일 때 baseline).
    A2 소급 방어(기록 직후 fromDay>기록일)·A3 종결일은 `tools/_dv_batch_a123.ts`(임시 프루프)로 실측.
- **철회 = 삭제가 아니라 종결일(A3, 2026-07-08 — 소급 삭제 방지)**: `unbench`는 지시를 배열에서 **지우지 않고**
  `toDay = playedThroughDay(results)`(마지막 치른 경기일)를 박는다. 옛 삭제 방식은 지시가 **관전·기록된 구간에서
  통째로 사라져** 리플레이가 그 경기들의 라인업을 다시 썼다(서사 위반 — 감사: 리플레이 첫 변경일 == 옛 fromDay).
  종결일을 박으면 **[fromDay, toDay]=치른 구간은 벤치 유지(본 역사 보존)**, 그 이후 **미관전 미래 경기는 복귀**한다.
  `playedThroughDay < fromDay`(아직 한 경기도 안 치름)면 `toDay < fromDay`라 구간이 비어 **완전 취소와 동치**(부작용 0).
  - 리플레이 필터(`data/dynamics.benchedOn`): `fromDay ≤ day ≤ (toDay ?? Infinity)`.
  - **슬롯(BENCH_MAX=2)·중복·후보제외·팬분노 뉴스·표시는 활성(`toDay==null`)만** 카운트 — 종결된 지시는 슬롯을 비우고
    재건의를 허용한다. 소비처(활성 필터 적용): `store.suggestBench`/`suggestStart`(슬롯·중복·후보제외)·`data/news.ts`(팬 분노)·
    `store.endSeason`(벤치 분노는 [fromDay,toDay] 구간만). **UI 표시**(`app/(tabs)/squad.tsx`·`app/player/[id].tsx`의
    `benchDirectives.some(...)`)도 `b.toDay==null` 필터 필요 — **동시 작업 UI 에이전트 인계**(store는 원배열 노출).
  - 가드: `tools/_gt_owner.ts`(활성 카운트로 BENCH_MAX·중복 검사)·`tools/_dv_bench.ts` I7b(종결일 박힘+data층 복귀).
- 연쇄 효과(이 시스템의 본체):

## 5. 엣지 (미리 박아두는 가드)
- 벤치 지시로 출전 가능 인원이 부족해지면(부상과 겹침) **부상이 우선, 지시는 그 경기 한정 무시**(경기는 성립해야 한다).
- 면담 약속의 이행 판정은 시즌 경계 1회 — 파기 시 다음 시즌 첫 면담 거절 확률에 반영.
- 전원 벤치 지시 같은 악용은 지시 동시 상한(2명)으로 차단.
- 내 팀 전용 기능(AI 구단은 면담·건의 없음 — AI는 이미 수식으로 최적 행동).

### 기존 시스템과의 연계·중복 구분 (흩어짐 방지)

- **팬심(fanScore) 연계**: 반응의 **양·톤을 fanScore·팬덤(§4.5 `teamFanbaseNow`)이 조절** — 팬심 높으면 응원 위주,
  바닥이면 냉랭·비판 위주(§3.2 "인기 스타 무단 벤치→분노"의 자연 확장). SNS 반응은 **fanScore의 표면화**이지 새 감정 원천이 아니다.
- **NEWS_SYSTEM과의 역할 구분(중복 방지)**: 뉴스(`kind='owner'`)는 이미 "팬들 술렁"(간판 벤치)·"홈 관중 감소"를 낸다(§0 구현).
  SNS 피드는 **더 짧고 잦은 "팬 목소리" 레이어**(신문 기사 ≠ 트윗). **구현 시 결정**: (a) NEWS의 `owner`/`fans` kind를
  확장할지, (b) 별도 SNS 피드 화면을 둘지 — 어느 쪽이든 **같은 사건을 두 번 세지 않게** 게이팅(뉴스 기사 ↔ SNS 반응 dedupe).
- **§3.2 팬심 증감과의 관계**: SNS는 팬심을 *보여주는* 것이지 팬심을 *바꾸는* 새 레버가 아니다(팬심 증감식 불변 — 표시 레이어).

- **가짜 드라마 금지**: 실 경기·이적·순위 결과만. 없는 사건·인신공격성 악플 생성 금지(**건전한 비판** 톤 고정 — 사용자 명시).
