# independent-verify — 독립 검증 세션 스폰

> **왜**: "내가 구현한 걸 내가 만든 도구로 테스트"하면 같은 가정을 공유해 **같이 틀린다**(허위 오라클·확증편향).
> 구현을 **모르는** 신선한 세션이 **문서에서 기대를 도출**해 **처음부터** 측정 도구를 짜고 실제 출력과 대조하면,
> 내 사각·문서-코드 drift·오라클 결함이 드러난다. 방금 "100%"라고 말한 걸 진짜로 만드는 마지막 관문.
>
> 이 스킬은 **오케스트레이션 래퍼**다. 검증자 역할·규칙·보고형식의 정본은:
> - `.claude/.agents/independent-verifier.md` (역할 정의 + Spawn prompt 골격)
> - `docs/TEST_METHODOLOGY.md` §1.B(독립 문서기반 검증)·§3(스폰 템플릿)·§4(사각 분류)
> 이 스킬에서 다시 쓰지 말고 그 파일을 인용/전달한다.

## 실행 순서

### 1. 검증할 **주장(claim)** 을 사양 용어로 적는다
- "무엇이 보장되어야 하나"를 **구현이 아니라 사양/사용자 경험**으로. (예: "관전 중 실시간 점수판에서 어떤 행동을 한
  선수가 그 기록에 귀속된 선수와 같다", "FA 영입 후 단일 소유·캡·정원 불변".) 구현 방식은 적지 않는다(편향 방지).

### 2. `Agent`(general-purpose) 스폰 — **필요한 정보만**
검증자가 자력으로 검증하도록 **자기완결적** 프롬프트를 준다. 반드시 포함:
- **프로젝트 한 줄 + 작업 디렉터리**, 검증할 **주장**(1단계, 사양 용어).
- **읽을 진입점**: 관련 소스 파일 경로 + **읽을 문서**(`docs/TEST_METHODOLOGY.md`·`docs/STATS_PROTOCOL.md` +
  해당 `*_SYSTEM.md`/`BOARD_RULES.md`/`EDGE_CASES.md`). "무엇을 보장한다고 주장하는지 문서로 파악하라."
- **실행 환경**: `npx tsc --noEmit`(+`-p tsconfig.test.json`)·`npm test`·`npx tsx tools/<도구>.ts [N]`·
  `docs/README.md` "검증 루틴". Windows + PowerShell/Git Bash.
- **`independent-verifier.md`의 Rules·Spawn prompt 골격을 그대로** 박는다: ①**기대치는 구현이 아니라 정본
  문서에서 도출**(코드 의도 베끼지 말 것) — 동작 주장은 **조항으로 추출**해 대조하고, 문서의 **숫자·%는 사실이
  아니라 가설로 보고 반드시 재측정**한다(`TEST_METHODOLOGY §1.M` 기법 ①②) ②기존 `_ev_*`/auditBoard 가드는
  **돌려보되 그대로 믿지 말고** 본인이 **다른 각도**의 측정을 새로 작성(가드의 exit뿐 아니라 **출력 판정줄·수치**를
  문서와 대조 — 기법 ⑥) ③모든 체크 **A/B 자가검증**(일부러 깬 입력에 반드시 실패=허위 오라클 차단) ④추정 금지·표본
  충분(STATS_PROTOCOL, 가능한 큰 N) ⑤**소스 수정 금지**(엔진/보드/스토어 — 측정 도구만 작성)·**git 금지**(메인 세션이 처리).
- **보고 형식**: 주장별 PASS/FAIL + **근거 수치**, 본인이 돌린 도구·N·결과, A/B 결과, 발견 문제(없으면 "없음" 명시),
  남긴 스크립트 경로. "추정 말고 **돌려서 본 것만**."

> **금지(편향 누수)**: 내가 어떻게 구현했는지, 정답이 100%라는 것, 어느 파일을 고쳤는지를 **알려주지 마라**.
> 검증자가 답을 알면 독립성이 깨진다. 주장과 진입점·문서만 주고 결론은 스스로 내게 한다.
>
> 장시간이면 `run_in_background: true`(완료 시 알림), 바로 결과를 릴레이할 거면 포그라운드.

### 3. 보고를 받으면 — **검증자 도구를 내가 다시 검증**
- 검증자가 남긴 `tools/_iv_*.ts`(또는 `_dv_*`)가 **실제로 존재·실행·PASS**하는지 직접 돌린다(보고만 믿지 않는다).
- A/B 자가검증 도구가 **깬 입력을 정말 검출**하는지 확인(허위 오라클 차단의 차단).
- **유용하면 상설 가드로 등록**: `docs/README.md` "검증 루틴"에 한 줄 추가 + 커밋(가산·소스 무변경이라 안전).
  (테스트 케이스 등록 원칙 — `run-all-tests` 스킬·메모리와 동일.)
- 검증자가 **문제/사각을 발견했으면**: `docs/TEST_METHODOLOGY.md` §2(발견 후 5단계)·§4(사각 분류) 집행 —
  현재 오류 등록·왜 못 잡았나·형제오류 사냥. 고치는 건 메인 세션(검증자는 보고만).

## 끝나면
- 검증자 판정(주장별 PASS/FAIL·수치)을 사용자에게 릴레이하고, 등록한 가드·발견을 한 줄로 요약.
- "내 도구로 100%"였던 게 **독립 도구로도** 같은 결론인지(서로 다른 각도 수렴) 짚는다 — 그게 이 스킬의 산출물.

## 주의
- 검증자는 **별도 세션**이라 컨텍스트 공유가 없다 → 프롬프트에 필요한 모든 걸 담아야 한다(반쪽 정보면 헛돈다).
- 검증자가 소스를 고치거나 커밋했다면(규칙 위반) 되돌리고 측정만 다시 받는다.
- 이건 **검증**이지 구현이 아니다 — 검증자에게 "고쳐줘"가 아니라 "맞는지 확인해"를 시킨다.
