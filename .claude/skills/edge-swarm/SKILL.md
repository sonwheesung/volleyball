# edge-swarm — 100+ 독립 세션 엣지 케이스 발굴 + 자기보강

> **왜**: [`analyze-cases`](../analyze-cases/SKILL.md)는 **1개** 세션(나)이 코드를 읽고 엣지를 도출한다 — **내 한
> 사각에 갇힌다**(내가 안 떠올린 경계·동시·고갈은 통째로 안 적힌다). 100+개를 **area×lens×seed로 분산 + 의도적 중복**
> 으로 띄우면:
> 1. **넓은 표면 동시 채굴** — 엔진 44모듈·데이터 36셀렉터에서 불변식 파괴 지점을 한 번에.
> 2. **합의도 + 생성 사각** — 여러 세션이 같은 엣지를 떠올리면 **명백**(우선 등록), *한* 세션만 떠올렸으면
>    **희귀·창의적**(가치 큼). "**A는 떠올렸는데 B는 왜 못 떠올렸나**"(B의 area/lens 갭) = 생성 사각.
> 3. **자기보강** — 엣지를 잘 캐는 렌즈/area를 [`engine-verify/lenses.md`](../engine-verify/lenses.md)에 환류 → 다음 run이 커버.
>
> **engine-verify(검증)의 생성 짝.** edge-swarm은 *발굴*(아직 문서화 안 된 엣지를 만든다), engine-verify는 *검증*
> (이미 만든 게 맞나 A/B로 확인). 발굴된 엣지의 도구·검증은 [`verify-cases`](../verify-cases/SKILL.md)로 인계.
> 케이스 작성 규칙의 정본은 [`analyze-cases`](../analyze-cases/SKILL.md)·[`docs/EDGE_CASES.md`](../../../docs/EDGE_CASES.md).

## 사전조건 (반드시 확인)

- **오케스트레이션 = `Workflow` 도구.** 이 스킬 호출 자체가 다중 에이전트 opt-in(스킬 지시로 Workflow 허용).
  100+ 에이전트는 **비용 크다** — 사용자 명시 호출 시에만. 규모는 `budget`로 스케일(없으면 기본 N=100).
- **검증자는 소스 수정·git·도구 작성 금지** — 엣지 *후보를 도출해 보고*만. 도구·검증은 verify-cases.

## 실행 순서

### 1. 범위·렌즈·기존 케이스 로드
- [`engine-verify/lenses.md`](../engine-verify/lenses.md) 읽기 — **area 목록**(엔진/데이터 표면)·**lens 목록**(불변식·분포·
  결정론·문서↔코드 drift·경계/극단·동시성·장기누적·단위정합·교차계층 귀속). engine-verify와 **같은 카탈로그 공유**.
- `docs/EDGE_CASES.md` 현황(§0 불변식·§1 도구표·§3 레지스트리) — **이미 등록된 케이스 = 중복 회피 기준**(검증자에게 전달).
- **추정 금지** — area·불변식은 코드/문서에서 가져오지 지어내지 않는다.

### 2. 분산 설계 (중복 = 합의도/사각 측정의 핵심)
- **N ≥ 100 = areas(M) × lenses(L) × seeds**. **각 area를 서로 다른 렌즈/seed 세션 ≥3개가 겹쳐** 본다 —
  겹침이 없으면 "어느 엣지가 자주/드물게 나오나"(합의도)와 "A 떠올림·B 못 떠올림"(생성 사각)을 **측정할 수 없다**.
- 각 세션에 `{area, lens, seed, 읽을 파일·문서}`를 배정표로 고정(결정론·재현). 배정표 보관.

### 3. Workflow 팬아웃 (100+ 독립 발굴자)
- `Workflow`로 N개 `agent()` 스폰. 각 프롬프트 = **analyze-cases 규칙**(코드 정독: 입력/출력 타입·분기·가드
  `if…continue/return`·상수, 특히 **돈·계약·소속을 바꾸는 지점**과 **루프 제외 조건**=불변식의 자리. **WAI 구분**:
  정상 동작은 엣지 아님) + 배정(area·lens·seed) + 진입점(파일·문서) + 이미 등록된 케이스(중복 회피).
  **구조화 스키마로 반환**(파싱 불요):
  ```
  { area, lens, seed,
    edges: [{ invariant(깨질 불변식), trigger(어떤 입력/시드/경계/동시/고갈), expectedNormal, suspectedBreak,
              codeBasis(파일:심볼 — 그 분기/가드 근거), alreadyKnown(EDGE_CASES에 있나 추정), severity, isWAI }],
    surveyed: [읽고 정상 확인한 area·불변식] }   // surveyed = 이 세션이 무엇을 봤나 = 생성 사각의 분모
  ```
- 동시성은 Workflow가 캡(≈cores−2), 100+는 큐 소화. 길면 `run_in_background`. 편향 누수 금지(정답/내 구현 비공개).

### 4. 2차 검수 (나 = 오케스트레이터 — 후보를 거른다)
- 각 엣지 후보를 검사: (a) **이미 EDGE_CASES에 있나** → 중복 제거 (b) **codeBasis가 실재**하나(그 분기/가드가 진짜
  코드에 있나 — 헛것·환각 제거, 직접 Read로 확인) (c) **WAI인가**(문서가 정상이라 명시 → 엣지 아님, 기록만).
- 살아남은 것만 "신규 엣지 후보". **추정 금지** — codeBasis를 코드에서 직접 확인한 것만 채택.

### 5. 디둡 + 합의도 매트릭스 (생성 사각 측정)
- 후보를 **시그니처**(area+불변식+trigger)로 묶어 디둡. **엣지 × 세션** 매트릭스: 어느 세션이 떠올렸나.
- **합의도**: 여러 세션이 떠올린 엣지 = 명백(우선 등록·아마 고위험). 한 세션만 = 희귀(창의적 — 검토 후 등록).
- **"A 떠올림·B 못 떠올림" 사유**: B의 배정(area/lens/seed)·surveyed를 비교해 **무엇이 B를 막았나** 한 줄
  (예: "B는 불변식 렌즈만 — 장기누적 고갈은 장기누적 렌즈에서만 나옴"). = 생성 사각(TEST_METHODOLOGY §4의 생성판).

### 6. 등록 + 자기보강
- **신규 엣지 등록**: `docs/EDGE_CASES.md §3`(또는 해당 레지스트리)에 **"감시 대상"** 으로 `불변식·trigger·예상정상·
  의심파괴·codeBasis`를 적는다(아직 도구 없음). **도구·검증은 `verify-cases`로 인계**(이 스킬은 발굴까지).
- **생성 사각 → `lenses.md` 환류**: 엣지를 못 캐게 만든 누락 렌즈/under-covered area를 `engine-verify/lenses.md`
  "run 환류"에 append(engine-verify와 공유 — 두 스웜이 같이 똑똑해짐). 엣지를 많이 캔 렌즈는 표준 승격.

## 끝맺음
- 보고: 돌린 **세션 수·area×lens 배정**, **신규 엣지 후보**(불변식·trigger·codeBasis·합의도·심각도), **합의도 매트릭스**,
  **생성 사각 분석**("A 떠올림·B 못 떠올림" 사유), **lenses.md 환류분**, EDGE_CASES 등록분.
- "이제 `verify-cases`로 이 엣지들의 도구를 만들어 검증하면 됩니다"로 인계.

## 원칙
- **코드가 진실·추정 금지** — codeBasis를 코드에서 확인한 엣지만. 환각 trigger 기각.
- **중복은 자산** — 겹침이 없으면 합의도·사각을 못 잰다. 분산은 "넓게+겹치게".
- **발굴 ≠ 검증** — 여긴 *후보를 만든다*. 진짜 깨지나는 verify-cases가 A/B로 확인. WAI는 고치지 말고 기록.
- **비용 인지·자기보강이 목적** — 엣지를 찾는 것만큼 *"왜 어떤 세션은 못 떠올렸나"를 lenses.md에 남기는 것*이 산출물.
